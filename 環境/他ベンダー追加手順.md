# 他ベンダー機器を Containerlab Docker 化する手順（汎用プレイブック）

> IOL以外の機種・他ベンダーを `~/vrnetlab/<vendor>/<platform>/` でビルドし、Containerlabで使う手順。
> 前提と全体像は `環境説明.md`、経緯は `log/作業ログ.md` §5 を参照。最終更新: 2026-06-02。

---

## 0. 大前提（arm64の現実）

- この環境は **arm64 + /dev/kvm 無し**。x86のフルVM(qcow2/vmdk)は **QEMU TCG完全エミュ**で動く＝**遅い**。
- 目安: **RAM ≤2GB / 1-2 vCPU = 現実的**。4GB以上や dual-VM や「ビルドにKVM必須」は **非現実的**。
- `common/vrnetlab.py` 改造済み（`-cpu host`→`qemu64`）により x86 VMが起動可能。新規機種でも自動で効く。

## 1. 共通4ステップ

```bash
ssh clab@orb
cd ~/vrnetlab/<vendor>/<platform>          # 例: mikrotik/routeros, extreme/exos, cisco/vios
cat README.md                               # ★ファイル名規則を毎回確認（重要）

# (a) イメージを配置。形式により：
#   .qcow2      → そのまま規則名でコピー
#   .vmdk       → そのまま置ける builder もあれば、qemu-img convert が要る builder もある
#   .vmdk→qcow2 → qemu-img convert -f vmdk -O qcow2 in.vmdk <規則名>.qcow2
#   .tgz/.zip   → tar xzf / unzip して中の qcow2/vmdk/img を取り出す
#   .img(gz)    → gunzip

# (b) ビルド（生成物は vrnetlab/<kind>:<ver>）
make docker-image                           # ※同dirに余分な *.qcow2 を残さない（globで拾う）
docker images | grep -i <platform>          # 出来たタグ確認

# (c) clab.yml に書く（下記テンプレ）
# (d) sudo containerlab deploy -t <lab>.clab.yml
```

> ⚠ ツール: `qemu-utils`(qemu-img) / `unzip` / `gdown` / `sshpass` は導入済み。
> 無ければ `sudo apt-get install -y qemu-utils unzip` 等。

## 2. ダウンロード手段

| 元 | 方法 |
|---|---|
| Google Drive (hegdepavankumarリポ) | `python3 -m gdown "https://drive.google.com/uc?id=<ID>" -O out.tgz`。**quotaで失敗する事あり**→別バージョン/ミラー、またはユーザーが手動DL |
| ベンダー直(無料) | `curl -fsSL -o file URL`（MikroTik/OpenWrt/Extremeは無料・無認証） |
| ベンダー(要ログイン) | ユーザーがブラウザでDL → Mac `~/Downloads/` に置く（OrbStackで `clab` VM内も同じパスで見える）→ こちらでビルド |

## 3. 実証済みレシピ（コピペ可）

### MikroTik CHR（最軽量・無料・✅healthy実証済）
```bash
cd ~/vrnetlab/routeros       # ⚠ 実パスは routeros 直下（mikrotik/routeros ではない。2026-07-17確認）
curl -fsSL -o chr.zip https://download.mikrotik.com/routeros/7.21.4/chr-7.21.4.vmdk.zip
unzip -o chr.zip && rm chr.zip
DOCKER_DEFAULT_PLATFORM=linux/amd64 make docker-image   # → vrnetlab/mikrotik_routeros:7.21.4
```

> ⚠ **ビルドは必ず `DOCKER_DEFAULT_PLATFORM=linux/amd64` を付ける**（2026-07-17実測）。
> 素の `make docker-image` はarm64ネイティブイメージになり、`qemu-system-x86_64` が入らず
> `command not found` → QemuBroken で起動不能。
>
> ⚠ **VM型NOSの起動には `QEMU_CPU=max` 環境変数が必須**（2026-07-17実測）。KVMの無い本環境では
> 既定の `-cpu host` でqemuが即死する。clabトポは `env: { QEMU_CPU: max }`、docker直起動は `-e QEMU_CPU=max`。
> 詳細: [イメージ可用性検証_2026-07-17.md](イメージ可用性検証_2026-07-17.md)

### OpenWrt（最軽量128MB・無料）
```bash
cd ~/vrnetlab/openwrt
curl -fsSL -o owrt.img.gz https://downloads.openwrt.org/releases/25.12.4/targets/x86/64/openwrt-25.12.4-x86-64-generic-ext4-combined.img.gz
gunzip -f owrt.img.gz && mv owrt.img openwrt-25.12.4-x86-64-generic-ext4-combined.img
make docker-image            # → vrnetlab/openwrt_openwrt:25.12.4
# ※ `make build` は全バージョンを落とすので使わない（手動DLが安全）
```

### Extreme EXOS（無料）
```bash
cd ~/vrnetlab/extreme/exos
curl -fsSL -o EXOS-VM_32.7.2.19.qcow2 \
  https://akamai-ep.extremenetworks.com/Extreme_P/github-en/Virtual_EXOS/EXOS-VM_32.7.2.19.qcow2
make docker-image            # → vrnetlab/extreme_exos:32.7.2.19
```

### Cisco vIOS / vIOSL2（repoのDriveから）
```bash
cd ~/vrnetlab/cisco/vios
python3 -m gdown "https://drive.google.com/uc?id=<ID>" -O dl.tgz
tar xzf dl.tgz                                 # 中に virtioa.qcow2
cp <dir>/virtioa.qcow2 cisco_vios-<ver>.qcow2  # L2は cisco_viosl2-<YYYYMMDD>.qcow2
make docker-image            # → vrnetlab/cisco_vios:<ver>  (L2は :L2-<date>)
```

### FortiGate（要ログインDL・✅healthy実証済）
```bash
cd ~/vrnetlab/fortinet/fortigate
rm -rf /tmp/fgt && mkdir -p /tmp/fgt
tar xzf "/Users/shuya/Downloads/fortinet-FGT-vX.Y.Z-buildNNNN.tgz" -C /tmp/fgt   # 中に virtioa.qcow2
cp /tmp/fgt/*/virtioa.qcow2 ./fortios-vX.Y.Z.qcow2        # ★この命名が必須(fortios-v…)
# 初回のみ: docker/Dockerfile の apt一覧に `qemu-system-x86` を追加（arm64でx86 qemu必須）
make                         # ※ "make docker-image" ではなく "make"（→ vrnetlab/vr-fortios:X.Y.Z）
```

## 4. clab.yml テンプレ

```yaml
name: demo
topology:
  nodes:
    r1: { kind: <KIND>, image: <IMAGE> }
    r2: { kind: <KIND>, image: <IMAGE> }
  links:
    - endpoints: ["r1:eth1", "r2:eth1"]
```

| 機種 | KIND | IMAGE 例 | データIF |
|---|---|---|---|
| Cisco IOSv | `cisco_vios` | `vrnetlab/cisco_vios:159-3.M3` | eth1=Gi0/1 |
| MikroTik CHR | `mikrotik_ros` | `vrnetlab/mikrotik_routeros:7.21.4-amd64` | eth1=ether2 |
| Extreme EXOS | `extreme_exos` | `vrnetlab/extreme_exos:32.7.2.19` | eth1=port1 |
| OpenWrt | `openwrt` | `vrnetlab/openwrt_openwrt:25.12.4` | eth1 |
| FortiGate | `fortinet_fortigate` | `vrnetlab/vr-fortios:7.2.0` | eth1=port2 |

> kind名はclab公式（containerlab.dev/manual/kinds）準拠。起動が遅いノードは
> deployがhealth待ちで時間がかかる（vIOSで約2分/ノード）。`docker logs <node>` で進捗確認。

## 5. ベンダー別 実現性早見表

| 区分 | ベンダー/機種 |
|---|---|
| ✅ 無料DL・自動化可（推奨） | MikroTik CHR, OpenWrt, Extreme EXOS, Cisco IOL/vIOS |
| ✅ 要ログインDLだが実証済 | **FortiGate 7.2.x(2GB)**（ユーザー提供qcow2→`make`でビルド・healthy確認） |
| △ 要ログインDL（未実証だが可能） | Arista vEOS(2GB), Juniper vSRX(4GB・遅), FortiFirewall(同FortiOS) |
| ✖ arm64非現実的／Containerlab非対応 | Palo Alto(4-6GB), **FortiManager(clab kind無)**, Nokia SR OS, Juniper vMX/vJunos/vQFX, Aruba(8GB), Huawei, Dell FTOSv, SONiC |
| ⚡ ネイティブarm64の速い代替（VMより推奨） | Nokia **SR Linux**, Arista **cEOS**, Juniper **cRPD**, FRR |

> 「重い機種が欲しい」場合は、まず⚡のネイティブarm64コンテナを検討する方が快適。
