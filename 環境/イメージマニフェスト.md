## 再取得方法の注記

- **DockerHub / ECR Public公式イメージ**（debian, freeradius, osquery, bind9, network-multitool）: `docker pull <repository>:<tag>` で即座に再取得可能。ネットワーク疎通のみで完結する。
- **vrnetlabビルドイメージ（`vrnetlab/*`）の復元は docker load が第一選択**（2026-07-17改訂）:
  Google Drive `マイドライブ/Images/Dockerimege/` に **ビルド済みイメージのdocker saveバックアップ（tar.gz+sha256）** が存在する。
  `docker load < vrnetlab_<name>_<tag>.tar.gz` 一発で復元でき、再ビルドは不要。
  バックアップに無いイメージのみ、vrnetlabビルド（素材は同Drive `Images/` 直下、手順は[rebuild_runbook.md](rebuild_runbook.md)と[他ベンダー追加手順.md](他ベンダー追加手順.md)）。
  **⚠ ビルド時は必ず `DOCKER_DEFAULT_PLATFORM=linux/amd64` を付けること**（arm64ネイティブビルドはqemu-system-x86_64が入らず起動不能。2026-07-17実測）。
- **重複疑いは解消済み**（2026-07-17実測）: `vr-fortios`↔`fortinet_fortios`、`vr-ubuntu`↔`canonical_ubuntu:jammy`、
  `mikrotik_routeros:7.5-amd64`↔`:7.5` は全てイメージID同一の別名タグだった。`vr-fortios` / `vr-ubuntu` / `:7.5-amd64` の3タグは削除済み（実体は残存）。
- **osquery/osquery:latest**: arm64非対応（amd64実体）のため、Apple Silicon上ではQEMUエミュレーション経由で動作している可能性が高い。性能・安定性に問題が出る場合は代替手段の検討が必要（詳細: [軽量検証結果_2026-07-04.md](../ZERO_zero_trust/03_詳細設計/軽量検証結果_2026-07-04.md)）。

## 動作検証（2026-07-17実測）

全イメージの「そもそも使えるか」を実機検証した。**結果の詳細・機種別の判定・LLM運用プロトコル・OOM事故の教訓は
[イメージ可用性検証_2026-07-17.md](イメージ可用性検証_2026-07-17.md) を参照**。要点:

- Linux系は全て動作（新発見: `freeradius/freeradius-server`は2タグとも**amd64実体**。arm64代替は自作`nac-freeradius:local`）
- **VM型NOSは環境変数 `QEMU_CPU=max` が必須**（無いとqemu即死）。Tier3（x86フルVM）は単独スロット運用が絶対条件
- FULL確定: RouterOS 7.5 / 7.21.4（REST API込み）、NX-OS titanium。BOOT_ONLY: IOL 2タグ・vIOS 2タグ・FortiOS。NG: vEOS（代替=cEOS arm64）
- 機種別の実測クレデンシャル・癖は [イメージ検証/devices.yaml](イメージ検証/devices.yaml) に機械可読で保存
