## 1. 全体像（3層構造）

```
┌─ macOS (Apple M4 / arm64, macOS 15.6.1) ──────────────────────────┐
│  ・OrbStack 2.1.3 / Docker 29.4.0                                  │
│  ・VSCode（Remote-SSH で下のLinuxマシンに接続して操作）            │
│                                                                    │
│   ┌─ OrbStack Linux マシン "clab" (Ubuntu noble / arm64) ───────┐  │
│   │  接続: ターミナルで  ssh clab@orb                            │  │
│   │  ・Docker 29.5.0                                            │  │
│   │  ・Containerlab 0.75.0                                      │  │
│   │  ・Rosetta + qemu-x86_64（x86イメージのエミュレーション）   │  │
│   │                                                            │  │
│   │   ┌─ Docker コンテナ（＝ネットワーク機器）──────────────┐  │  │
│   │   │  clab-ospf3-r1..r4  (Cisco IOS / cisco_iol)         │  │  │
│   │   │  clab-cisco-lab-r1,r2 (FRR)                         │  │  │
│   │   └────────────────────────────────────────────────────┘  │  │
│   └────────────────────────────────────────────────────────────┘  │
└────────────────────────────────────────────────────────────────────┘
```

**重要な性質**
- OrbStack は Mac のファイルを Linux マシン内に **同じパスでマウント**している。
  例: Mac の `/Users/shuya/Downloads/x.bin` は `clab` マシン内でも `/Users/shuya/Downloads/x.bin` で見える。
- Containerlab は Linux 専用ツール。Mac では直接動かないので、この `clab` マシンの中で動かす。

## 2. CPUアーキテクチャの大前提（最重要）

- この Mac は **arm64 (Apple Silicon)**。一方ネットワーク機器イメージはほぼ全て **x86_64 / i386**。
- そのため Cisco などの x86 イメージは **Rosetta / QEMU でエミュレーション**して動かしている。
  - 結果: **動くが起動が遅い**（IOLで1ノード約2分）。これは arm64 の宿命で、回避不可。
- arm64 ネイティブで快適に動くもの: **FRR / Nokia SR Linux / Arista cEOS(arm) / Juniper cRPD**。

### 速度ティア（2026-07-17 全イメージ実測に基づく選定基準）

新規ラボのイメージ選定はこの表に従う（実測の詳細: [イメージ可用性検証_2026-07-17.md](イメージ可用性検証_2026-07-17.md)）:

| Tier | 分類 | ブート実測 | 使いどころ |
|---|---|---|---|
| **1** | arm64ネイティブ（SR Linux / FRR / OpenWrt / Linux系） | **秒**（SR Linux 12秒） | 日常ラボの既定。API-first学習（JSON-RPC/gNMI/REST） |
| **2** | x86ユーザーモード（**IOL** / RouterOS CHR） | 1〜3分 | Cisco CLI学習の主力（従来どおり）。MikroTikはREST APIも使える |
| **3** | x86フルVM（CSR/C8000v/NX-OS/ASAv/vSRX/FortiOS/C9800CL） | **15〜90分** | 実OSの挙動確認のみ。**必ず単独スロット**（並走するとOOMで他ラボを巻き添えにする。2026-07-17に実事故あり）。並列管理は `イメージ検証/run_queue.sh` |

- **Tier3の必須設定**: 環境変数 `QEMU_CPU=max`（clabトポで `env: { QEMU_CPU: max }`）。無いとqemuが即死する
- vrnetlabビルドは `DOCKER_DEFAULT_PLATFORM=linux/amd64` 必須（[他ベンダー追加手順.md](他ベンダー追加手順.md)参照）
- 機種別の実測クレデンシャル・癖: [イメージ検証/devices.yaml](イメージ検証/devices.yaml)
- Tier3を高速化する唯一の方法はKVMが使えるx86環境: [x86実行環境_選定.md](x86実行環境_選定.md)
- VM RAMは **20GiB**（2026-07-17に15→20へ増量。`orb config set memory_mib 20480`）

## 3. アクセス方法まとめ

| やりたいこと | 方法 |
|---|---|
| Linuxマシンに入る | Macのターミナルで `ssh clab@orb` |
| VSCodeで操作 | Remote-SSH で `clab@orb` に接続 → 拡張「Containerlab」(srl-labs) を使用 |
| 動いてる機器一覧 | `sudo docker ps`（NAMES列が機器名） |
| ラボ一覧 | `sudo containerlab inspect --all` |
| 機器のCLIに入る | `sudo docker attach clab-ospf3-r1` → Enter → `r1#`。**抜ける: Ctrl-P → Ctrl-Q** |

> ⚠ `ssh admin@clab-ospf3-r1` は **失敗する**（No route to host）。
> 理由: この旧IOLイメージは管理インターフェース/SSHが未設定のため。
> → 機器へは上記の **コンソール接続（docker attach / VSCodeの右クリック Attach）** で入る。

Google Drive `マイドライブ/Images/IOURC.zip` のライセンスファイル（hostname `gns3-iouvm` 用）を
entrypoint.sh に埋め込み済み。entrypoint が自動で hostname を `gns3-iouvm` に設定するため、

## 6. 新しいイメージを追加する手順

```bash
ssh clab@orb
cd ~/vrnetlab/cisco/iol
rm -f cisco_iol*.bin                              # 前のbinを掃除
cp /Users/shuya/Downloads/<新イメージ>.bin ./cisco_iol-<タグ名>.bin   # L2なら cisco_iol-L2-<名>.bin
DOCKER_BUILDKIT=1 make docker-image
docker images | grep cisco_iol                    # 出来たタグを確認
```

## 7. ラボの起動・停止

```bash
ssh clab@orb
cd ~/network-lab/ospf3
sudo containerlab deploy  -t ospf3.clab.yml       # 起動（各ノード約2分）
sudo containerlab destroy -t ospf3.clab.yml       # 停止・削除
```

VSCode なら: Containerlabパネルでラボを右クリック → Deploy / Destroy / Redeploy。
