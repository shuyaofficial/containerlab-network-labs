## 検証方法

  （mgmt: 172.20.60.0/24、稼働中ラボと分離）を `run_one.sh <name>` で deploy→ブート待ち→SSH/API検証→destroy。
  containerlab 0.77.0 に kind が無い3機種（NX-OS titanium / C8000v controller / C9800-CL）は `docker run --privileged` 直起動。
- **コマンド出典**: `イメージ検証/commands/*.cmds`。各コマンドはベンダー公式リファレンスPDF（pdftotext化）に
  実在することをgrepで確認済み。出典PDF・該当ページはcmdsファイル内に記録。

- **判定**: ✅FULL（boot+アクセス+全コマンド成功）/ 🟡PARTIAL（アクセス可・一部不成立）/ 🟠BOOT_ONLY /

- 環境: OrbStack VM（Ubuntu 24.04 arm64、10CPU/15GiB、KVMなし=x86イメージはQEMU TCG完全エミュレーション）。
  検証中も稼働中2ラボ（qos-acl-policy-lab 11ノード / nwzt-lan 8ノード）は無停止で維持。

## ⚠ 重要発見 — VM型NOSは既定設定ではqemu即死する（QEMU_CPU=max で解決）

vIOS検証中に判明。**VM上の全vrnetlab VM型イメージは、qemu起動オプションに `-cpu host` を使う**が、
このフラグはKVM必須のため、KVMの無い本環境（Apple Silicon / OrbStack）では
`qemu-system-x86_64: CPU model 'host' requires KVM or HVF` で**qemuが即死**し、ブートが永遠に始まらない。

- 症状: `docker logs` に `Unable to connect to qemu monitor (port 4000)` の無限リトライ → `QemuBroken`
- **解決**: コンテナ環境変数 `QEMU_CPU=max` を渡す（全イメージのvrnetlab.pyが対応済みを確認）。
  clabトポでは `env: { QEMU_CPU: max }`、docker直起動では `-e QEMU_CPU=max`。リビルド不要。
- 本ハーネスの全qemu系トポ・dockerモードには適用済み。**既存/今後のラボでVM型NOSを使う場合も必須**
  （IOLはqemu非経由のため不要。既存IOLラボには影響なし）。

## OOM事故と対策（2026-07-17 15:38発生）

`deploy-all.sh destroy`→`deploy`の順が必要）。対策として①VM RAMを20GiBへ増量（実施済み）
②free実測でなく宣言RAM予約制の `イメージ検証/run_queue.sh` で並列を管理（新設）。
教訓: **qemuはRAMを遅延確保するため、起動直後の空きRAM実測は当てにならない**。

## 高速化PoCの結果 — arm64ネイティブqemuは効果なし(撤収)

仮説「amd64版qemu(Rosetta変換)→arm64ネイティブqemuにすれば二重変換が一重になり速くなる」を実測検証した。

- 方法: vrnetlabのDockerfileで `qemu-kvm` → `qemu-system-x86 qemu-utils` に差し替えarm64ネイティブビルド
  (CHR 7.21.4で同一vrnetlabコード・qemuバイナリarchだけが異なる公平なA/B)
- 結果: **amd64版 61/65秒 vs arm64ネイティブ版 60/61秒 — 差なし**(healthy到達、各2回)
- 解釈: OrbStackのRosettaはqemuバイナリ自体をほぼネイティブ速度で実行しており、ボトルネックは
  **TCGによるゲストx86コード変換そのもの**。qemuの実行arch変更では解決しない
- 副産物: vIOSはIOS起動自体が20〜25秒と高速で、遅さの正体はvrnetlab bootstrap層(scrapli)と確定
- 結論: **amd64ビルド標準を維持**(`DOCKER_DEFAULT_PLATFORM=linux/amd64`)。x86 VM型NOSの高速化は
  エミュレーション最適化では不可能で、KVMが使えるx86実行環境([x86実行環境_選定.md](x86実行環境_選定.md))が唯一の根本解
