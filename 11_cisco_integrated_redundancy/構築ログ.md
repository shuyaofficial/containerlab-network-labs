# テーマ11: 統合冗長（Track × HSRP × OSPF）構築ログ

## 構築で発生した問題

### 問題1: R1（ルーター）にL2スイッチ用のイメージ・設定を使用してしまった

#### 発生日時
2026-06-03 14:37 頃

#### 事象
- 初回のトポロジファイル作成時、`type: l2` を `kinds` 配下にグローバル設定してしまった。
- さらに、R1（純粋なルーターとして使用する機器）にも L2スイッチ用のイメージ `L2-advipservices-2017` を割り当ててしまった。
- R1はインターネット側のコアルーターであり、OSPF + Loopback0 (8.8.8.8) のみを動かす純粋なルーターのため、L2スイッチモードで起動させるのは明確に間違い。

#### 誤っていたトポロジファイル（修正前）
```yaml
topology:
  kinds:
    cisco_iol:
      type: l2     # ← 全ノードにL2を適用してしまっている（NG）

  nodes:
    r1:
      kind: cisco_iol
      image: vrnetlab/cisco_iol:L2-advipservices-2017  # ← ルーターにL2イメージ（NG）
```

#### 修正後のトポロジファイル
```yaml
topology:
  nodes:
    r1:
      kind: cisco_iol
      image: vrnetlab/cisco_iol:15.7.3M2  # ルーター用イメージ（L2-なし）
      # type 指定なし = ルーターモードで起動（正しい）
    sw1:
      kind: cisco_iol
      image: vrnetlab/cisco_iol:L2-advipservices-2017
      type: l2     # L3スイッチとしてL2モードで起動（正しい）
    sw2:
      kind: cisco_iol
      image: vrnetlab/cisco_iol:L2-advipservices-2017
      type: l2
```

#### 教訓
- **機器の「役割」を先に明確にしてからイメージを選定する**こと。
  - ルーター → `L2-` プレフィックスなしのイメージ、`type` 指定なし
  - L3スイッチ → `L2-advipservices-2017`、`type: l2`
  - L2スイッチ → `L2-15.2`、`type: l2`
- **`type: l2` をグローバル設定にしない**こと。ルーターとスイッチが混在する場合、各ノードに個別に設定する。
- 詳細なルールは [containerlab_iol_quirks.md](../切り分け/containerlab_iol_quirks.md) の「§6」に記載済み。
