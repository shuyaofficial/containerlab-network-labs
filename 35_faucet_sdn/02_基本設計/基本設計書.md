# 基本設計書: Faucet SDN

## 1. 設計方針

本テーマは「制御面（コントロールプレーン）とデータ面（フォワーディングプレーン）を分離する」というSDNの根本思想を、Faucet（OpenFlowコントローラ）とOVS（OpenFlowスイッチ）の最小構成で体験することを設計の軸に据える。ネットワークポリシー（VLAN・ACL）はすべてFaucet側の宣言的な設定ファイル（YAML）に集約し、OVS側にはそのポリシーを実現するための「プラミング（配線）」のみを用意する。

## 2. 全体構成

| ノード名 | 役割 | ゾーン |
|---|---|---|
| faucet | OpenFlowコントローラ（制御面）。VLAN/ACLの宣言的設定を保持し、OVSへフローエントリを配信する | Control |
| ovs1 | OpenFlowスイッチ（データ面）。userspaceデータパスでbr0を構成し、pc-a/b/cを収容する | Data |
| pc-a | エンドポイント。データ面 10.0.100.11/24（VLAN100想定） | Data |
| pc-b | エンドポイント。データ面 10.0.100.12/24（VLAN100想定） | Data |
| pc-c | エンドポイント。データ面 10.0.200.13/24（VLAN200想定） | Data |

制御チャネル（OpenFlow、TCP 6653）は containerlab の mgmt ネットワーク経由で `faucet` ↔ `ovs1` 間を流れる。データチャネル（実際のユーザートラフィック）は `ovs1` の `eth1`〜`eth3` に接続された `pc-a`/`pc-b`/`pc-c` 間を流れる。制御チャネルとデータチャネルが物理的に異なる経路を通る点は、SDNの制御/データ分離を可視化する上で重要な構成要素である。

## 3. アドレス設計方針

- mgmtサブネットは `172.35.35.0/24` を使用し、[clab運用規約.md](../../規約/clab運用規約.md) §4のmgmt-ipv4静的固定規約に従い、全ノードへ固定IPを割り当てる。
- データ面IPは `pc-a`/`pc-b` を `10.0.100.0/24`（VLAN100想定）、`pc-c` を `10.0.200.0/24`（VLAN200想定）とし、サブネットの違いがそのままVLAN設計上の狙い（同一VLAN=同一サブネット、別VLAN=別サブネット）と対応するようにする。詳細値は [IPアドレス管理表.md](IPアドレス管理表.md) を参照。
- どのポートがどのVLANに実際に割り当てられるかは `faucet.yaml`（学習者が作成）で決まるため、本書ではアドレス設計上の「狙い」までを示し、確定値は記載しない。

## 4. 冗長性・拡張性

Phase A（本テーマの範囲）は単一OVS・単一Faucetの最小構成であり、冗長性は考慮しない。将来的な拡張として、以下が考えられる。

- **Phase B（将来拡張・本テーマの範囲外）**: `ovs1` のミラーポート（SPAN）またはOpenFlowのサンプリング機能を用いて east-west トラフィックを複製し、`gauge`（Faucetのポート統計エクスポータ）や `goflow2` 等のフローコレクタへ転送することで、NDR基盤（[42_ndr_flow](../../42_ndr_flow/README_Lab_Challenge.md)）と連携する構成が考えられる。この場合、可視化されたトラフィック統計をもとに異常検知（east-westのスキャン検知等）へつなげることができる。本テーマでは `04_構築/faucet.clab.yml` に `gauge` ノードをコメントアウトで用意するのみに留め、実装は行わない。
- 複数OVSへのスケールアウト（マルチスイッチ・マルチコントローラ構成、Faucetの `stack` 機能によるスイッチ間トポロジ制御）。

## 5. セキュリティ方針

- OVSブリッジ `br0` の `fail-mode` は `secure` を採用する。これはコントローラとの接続が切断された場合に、既存フローのみで動作し新規フローを学習しない（デフォルトのL2スイッチのようにフラッディングしない）挙動であり、コントローラ不在時に意図しない通信が発生するリスクを抑える設計判断である。
- VLAN分離・ACLによるプロトコル制御は、Trustゾーン内であってもセグメント間のトラフィックを既定で遮断し、明示的に許可した通信のみを通す「ゼロトラスト的な」最小権限の考え方に基づく。

## 6. 設計判断の記録（考えどころ）

- **なぜ userspace データパス（netdev）を選んだか**: userspaceデータパス（netdev）はOVSのカーネルデータパス（`openvswitch` カーネルモジュール）に依存せず、TUN/TAPとAF_PACKET経由の netdev-linux provider で veth を収容できるため、ホストのカーネル構成に左右されず再現できる（OrbStackのバージョン差やカーネル差し替え状況に依存しない）。トレードオフとして性能コストがあるが、本ラボの学習目的（SDN制御モデルの理解）には影響しない。なお本テーマの検証VMでは実測上 `openvswitch` モジュールが読み込まれており `datapath_type=system`（カーネルデータパス）も選択可能だが、可搬性を優先して netdev を既定とする（2026-07-07 実機検証で netdev 構成の疎通を確認済み）。
- **なぜ制御面とデータ面を分けるか**: OpenFlowの本質的な価値は、個々のスイッチが自律的に学習・判断するのではなく、集中管理されたコントローラの指示に従う点にある。本テーマで制御チャネル（mgmtネット経由）とデータチャネル（vethリンク経由）を意図的に分離した構成にしているのは、この「制御とデータの分離」を物理的にも体感できるようにするため。
- **なぜFaucetを選んだか**: OpenFlowコントローラにはONOS・OpenDaylight（ODL）等の選択肢もあるが、いずれもJVMベースで重量級であり、コンテナ環境・arm64環境でのリソース消費が大きい。Faucetは Ryu ベースの軽量実装であり、Python製で設定がYAML1本に集約されるため学習コストが低く、arm64環境での動作実績も良好である。本ラボの「宣言的設定によるポリシー制御」という学習目的に対して最も適したコントローラと判断した。
