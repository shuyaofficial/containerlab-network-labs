# テーマ35 機器別ログインコマンド

## 1. VM接続

以下はすべてOrbStack Linux VM `clab` で実行します。

```bash
ssh clab@orb
cd /path/to/Mac仮想環境構築/35_faucet_sdn
```
<!-- 上記パスはVM内での相対的な作業ディレクトリを示す例。本文に実際の絶対パスは書かない。 -->

## 2. 稼働確認

```bash
cd 04_構築
./deploy.sh inspect
```

すべてのノード（`faucet` / `ovs1` / `pc-a` / `pc-b` / `pc-c`）が `running` であることを確認します。

## 3. Faucetコントローラの確認

### faucet
```bash
sudo docker logs -f clab-faucet-sdn-faucet
```

`faucet.yaml` のロードに成功しているか、エラーが出ていないかをログで確認します。設定変更後は同じコマンドでリロードが反映されたかを追えます。

コンソールに入る場合:
```bash
sudo docker attach --sig-proxy=false clab-faucet-sdn-faucet
```

### コンテナを止めずに離脱
`Ctrl-p` を押し、続けて `Ctrl-q` を押します。`Ctrl-c` は離脱には使いません。

## 4. OVS（ovs1）の確認

```bash
sudo docker exec clab-faucet-sdn-ovs1 ovs-vsctl show
```

`Controller` セクションの `is_connected: true` を確認します（`false` のままの場合はFaucet側の待受状態やmgmt経路を疑う）。

ブリッジの詳細（datapath種別・ポートのofport番号）:
```bash
sudo docker exec clab-faucet-sdn-ovs1 ovs-vsctl list bridge br0
sudo docker exec clab-faucet-sdn-ovs1 ovs-ofctl show br0
```

## 5. エンドポイント接続（docker exec）

### pc-a / pc-b / pc-c
```bash
docker exec -it clab-faucet-sdn-pc-a sh
docker exec -it clab-faucet-sdn-pc-b sh
docker exec -it clab-faucet-sdn-pc-c sh
```

Linuxシェルからは `exit` で抜けます。疎通確認は `ping` を使用します（例: `ping -c 3 10.0.100.12`）。

## 6. パケットキャプチャ（Wireshark）

mentor_guidelines.md §3 準拠。新しいテーマに入る際は必ずキャプチャ手段を確認しておく。`ovs1` のデータ面ポート（例: `eth1`）を対象にする例:

```bash
ssh clab@orb "sudo nsenter -t \$(docker inspect -f '{{.State.Pid}}' clab-faucet-sdn-ovs1) -n tcpdump -U -nni eth1 -w -" | /Applications/Wireshark.app/Contents/MacOS/Wireshark -k -i -
```

OpenFlow制御チャネル（mgmtネット・ポート6653）を見たい場合は対象インターフェースをmgmt側に変更する。

## 7. ログインできない場合

### 7.1 コンテナ確認
```bash
sudo docker ps --format 'table {{.Names}}\t{{.Status}}' | grep faucet-sdn
```

### 7.2 起動ログ確認
```bash
sudo docker logs --tail 50 clab-faucet-sdn-ovs1
sudo docker logs --tail 50 clab-faucet-sdn-faucet
```

### 7.3 ラボを作り直す場合
> 保存していない `faucet.yaml` の変更はコンテナ再生成で消えないが（bindマウントのためホスト側ファイルに残る）、OVSのブリッジ状態はリセットされる。
```bash
cd 04_構築
./deploy.sh destroy
./deploy.sh deploy
```
