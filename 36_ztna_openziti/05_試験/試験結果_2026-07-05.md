# テーマ36 SDP型ZTNA（OpenZiti）試験結果（2026-07-05）

NW-ZT トラック N2。SDP の本質「**アプリは内向きポートを一切開けないのに、認可されたクライアントだけがオーバーレイ経由で到達できる（ダークサービス）**」を実機で実証。

## 試験環境（docker）

| コンテナ | ネットワーク | 役割 |
|---|---|---|
| ziti | zn-ziti | controller + router 一体（quickstart, arm64） |
| darkweb | **zn-app のみ** | nginx。ホスト公開ポートなし。client の居る zn-ziti には現れない |
| apptun | zn-ziti + zn-app | host tunneler。webapp を bind し darkweb:80 を dial |
| clienttun | **zn-ziti のみ** | proxy tunneler。webapp を localhost:8080 に射影 |

## 結果表

| ID | 試験項目 | 期待 | 実測 | 判定 |
|---|---|---|---|---|
| T-N2-1 | OpenZiti 制御プレーン起動 | controller+router 稼働 | quickstart 起動、router ONLINE=true | ✅ |
| T-N2-2 | サービス/ポリシー定義 | webapp を Bind/Dial で認可 | config/service/2 identity/2 policy 作成 | ✅ |
| T-N2-3 | identity enrollment | tunneler が enroll 成功 | apphost/webclient とも `enrolled successfully` | ✅ |
| T-N2-4 | サービス hosting/intercept | 両 tunneler が webapp を扱う | host: `hosting webapp`、proxy: `listening 0.0.0.0:8080` | ✅ |
| T-N2-5 | **オーバーレイ経由到達** | client→overlay→darkweb 成功 | `curl localhost:8080` = **HTTP 200** | ✅ |
| T-N2-6 | **直接到達不可（ダーク）** | client→darkweb 直達は失敗 | `curl darkweb:80` = **HTTP 000（unreachable）** | ✅ |
| T-N2-7 | ネットワーク分離の裏付け | client は app 網を持たない | clienttun=zn-ziti のみ / darkweb=zn-app のみ・公開ポートなし | ✅ |

## 実測エビデンス（要点）

```text
# host tunneler（apptun）
msg="Hosting newly available service" serviceName="webapp"
msg="hosting service, waiting for connections" service="webapp"

# proxy tunneler（clienttun）
msg="service is listening" addr="0.0.0.0:8080" service="webapp"

# 実証
[1] オーバーレイ経由 client→localhost:8080→darkweb : HTTP 200
[2] 直接 client→darkweb:80（到達不能が正常）      : HTTP 000 / unreachable
```

## 総括

**SDP 型 ZTNA の核心を OSS（OpenZiti, arm64）で実証完了。** Zscaler ZPA / Cisco Secure Access が「内向きを開けず、外向きの張り出しでのみ到達させる」仕組みを、controller(=broker) / router / tunneler で 1:1 に再現できた。IAP 型（ZERO Phase2 の Pomerium＝アプリ前のリバースプロキシ関所）との違いは、**アプリ側が着信ポートを持たない＝攻撃面が消える**点にある。詳細な詰まりどころは [構築ログ](../04_構築/構築ログ_2026-07-05.md)。
