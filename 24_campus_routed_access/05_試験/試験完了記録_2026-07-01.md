## 5. 気になった点と判断

### 5.1 `docker attach`でIOSへ入力できなかった

- 原因: IOLをバックグラウンド起動し、PID 1のbashが`wait`していたため、attachした入力がIOLへ届かなかった。
- 判断: Theme 23方式の`exec /iol/iol.bin`を採用し、IOL自身をPID 1にした。
- 教訓: containerが`running`でもログイン可能とは限らない。新規ラボではEnterがIOSへ届くところまで確認する。
- 詳細: [ログイン切り分け記録](../00_ログイン/ログイン切り分け_2026-06-30.md)

### 5.2 PCのデフォルトルートが既に存在した

- `eth0`の`default via 172.24.24.1`はcontainerlab管理ネットワーク用であり、削除すると管理通信へ影響する。
- `ip route add default via 10.24.x.1`は既存デフォルトルートと競合して`RTNETLINK answers: File exists`になった。
- 判断: デフォルトルートは維持し、相手LANだけをデータプレーン側へ向けた。

```bash
# pc-a
ip route add 10.24.20.0/24 via 10.24.10.1 dev eth1

# pc-b
ip route add 10.24.10.0/24 via 10.24.20.1 dev eth1
```

### 5.3 Linux端末ではIOSの`show`コマンドを使えない

- PC-A/PC-BはLinuxコンテナなので、経路確認は`show ip route`ではなく`ip route`を使用する。
- `show ip route`、`show ip ospf neighbor`、`show ip cef`はCisco IOL側で実行する。

### 5.4 Routed Port試験の表現

- access1/access2はCatalyst L3スイッチではなく、Cisco IOLのルーターモードによる模擬である。
- そのため`no switchport`の投入可否を試験するのではなく、インフラIFの`/30`、`up/up`、OSPF P2P隣接、L2 Trunk/STP非使用を確認する。

### 5.5 ECMPはpingやtracerouteだけでは証明できない

- 同一送信元・宛先の通信はハッシュ結果により、通常どちらか1経路へ固定される。
- ECMPの存在は`show ip route <prefix>`で2 next-hop、転送表は`show ip cef <destination>`で確認する。
- pingは疎通確認、片系断試験は各next-hopが実際に転送可能であることの確認として使う。
- tracerouteで片方しか表示されなくても、それだけでECMP失敗とは判定しない。

## 6. 再開・再確認時の順序

1. [機器別ログインコマンド](../00_ログイン/ログインコマンド.md)で4台へ入れることを確認する。
2. `show ip interface brief`で使用IFが`up/up`か確認する。
3. `show ip ospf neighbor`で必要な隣接が`FULL`か確認する。
4. access1/access2で対向LANに2 next-hopがあるか確認する。
5. PC-A/PC-Bで`ip route`を確認し、双方向pingを行う。
6. コンテナを再作成した場合、Linux端末のIPと個別静的経路が消えていないか確認する。
7. IOS設定を再起動後も残す必要がある場合は、学習者が保存済みか確認する。

## 7. AAR

- 予想: L3 AccessまでOSPFを延伸すれば、STPに依存せずECMPと高速な片系収束を観測できる。
- 実際: 正常時ECMP、単一リンク障害、core1全停止、復旧後のOSPF/ECMP復元まで完走した。構築以前にIOLのPID 1と端末管理ルートが重要な切り分け点になった。
- 次回: Theme 25では、最初に管理経路とデータプレーン経路を分け、BGP/OSPF境界の経路表を正常時・障害時の両方で保存する。
