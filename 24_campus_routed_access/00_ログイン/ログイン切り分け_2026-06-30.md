# ログイン切り分け記録 — Theme 24

## 1. 事象

次のコマンドを実行してもIOSプロンプトが表示されず、入力しても反応しない。

```bash
sudo docker attach clab-campus-routed-access-core1
```

## 2. 確認結果

| 確認 | 結果 |
|---|---|
| Theme 24コンテナ | 6台running |
| Cisco IOLプロセス | 4台とも`qemu-i386 /iol/iol.bin`で稼働 |
| SSH 22番 | 4台ともclosed |
| Theme 24 L3 entrypoint | IOLをバックグラウンド起動し、bashがPID1のまま`wait` |
| Theme 23 L2 entrypoint | `exec /iol/iol.bin`でIOLがPID1 |
| IOLログ | `System Configuration Dialog`で入力待ち |

## 3. 根本原因

### 原因A: 旧IOLでSSH bootstrapが完了しない

containerlabが生成した管理用設定に空のIPv6行とRSA鍵生成が含まれた。旧IOSはRSA鍵生成をstartup-config内で実行できず、SSH初期化を完了できないまま`System Configuration Dialog`へ移行した。

### 原因B: docker attachの入力先がIOLではない

Theme 24のL3イメージはentrypoint内でIOLをバックグラウンド起動し、bashがPID1のまま`wait`していた。`docker attach`はPID1のbashへ接続されるため、改行や回答がIOLへ届かなかった。

Theme 23のL2イメージは`exec /iol/iol.bin`を使用してIOL自身がPID1だったため、同じ`docker attach`で操作できていた。

## 4. 対応

- 学習対象のデータプレーン設定は投入しない。
- Theme 23方式を踏襲した`entrypoint_attach.sh`を4台へbindする。
- `exec /iol/iol.bin`でIOLをPID1にし、attach入力をIOSへ直結する。
- deploy後に各IOLコンテナで`iouyap 513`を起動する。
- SSH bootstrapには依存せず、初期設定はattachしたIOSコンソールから行う。

## 5. 公式仕様との関係

containerlab公式ではCisco IOLの通常管理方法としてSSHが案内され、デフォルトstartup-configが管理IP・管理VRF・SSHを設定するとされている。

- https://containerlab.dev/manual/kinds/cisco_iol/

今回の環境は、古いIOL/vrnetlabイメージのSSH bootstrapとL3 entrypoint構造が現在のcontainerlab運用に合わなかった例外ケースである。

## 6. 再発防止

- 新しいラボでは`00_ログイン/ログインコマンド.md`を必ず作る。
- READMEへログイン方法を推測で書かず、実際にEnterがIOSへ届くことまで確認する。
- `running`表示だけで完成とせず、全機器のログインプロンプトを確認する。
- 管理・起動bootstrapと学習対象のデータプレーン設定を分離する。
