# 基本設計書 — Theme 24: Routed Access Campus LAN

> 対応要件: [要件定義書](../01_要件定義/要件定義書.md)

## 1. アーキテクチャ概要

Routed Accessでは、Core/DistributionとAccess間をL2 TrunkではなくL3 Point-to-Pointリンクにする。各Accessは両CoreとOSPF隣接を形成し、対向Access配下のネットワークへ2つの等コスト経路を持つ。

```text
                     10.24.0.16/30
                [core1]----------[core2]
                  |  \            /  |
                  |   \          /   |
                  |    \        /    |
                  |     \      /     |
              [access1]          [access2]
                  |                  |
               [pc-a]             [pc-b]

       インフラ間: Routed Port /30 + OSPF area 0
       端末収容: access1/2が各利用者セグメントのGW
```

詳細図は[ネットワーク物理構成図](ネットワーク物理構成図.mermaid)を参照する。

## 2. 各層の責務

| 層 | ノード | 責務 | この構成で使用しないもの |
|---|---|---|---|
| Core/Distribution | core1、core2 | Access間のL3中継、経路冗長、OSPF | 利用者端末の直接収容 |
| L3 Access | access1、access2 | 利用者GW、OSPF広告、両CoreへのECMP | 上位向けL2 Trunk、STPによる冗長化 |
| Endpoint | pc-a、pc-b | E2E疎通と障害試験の送受信 | 動的ルーティング |

## 3. 物理・リンク設計

| リンク種別 | 本数 | アドレス長 | 用途 |
|---|---:|---:|---|
| Core—Access | 4 | /30 | OSPF隣接、ECMP転送 |
| Core—Core | 1 | /30 | Core間到達性、迂回経路 |
| Access—Endpoint | 2 | /24 | 利用者セグメント |

インフラリンクはすべてL3として使用する。VLAN ID、Trunk許可VLAN、STP Root、HSRP VIPは設計対象外とする。

## 4. IPアドレス設計

| 用途 | アドレスブロック |
|---|---|
| インフラPoint-to-Point | `10.24.0.0/27`から5つの`/30`を使用 |
| access1利用者 | `10.24.10.0/24` |
| access2利用者 | `10.24.20.0/24` |
| Router ID / Loopback | `10.24.255.0/24`から`/32`を使用 |
| containerlab管理 | `172.24.24.0/24` |

個別割当は[IPアドレス管理表](IPアドレス管理表.md)を正本とする。

## 5. OSPF設計

| 項目 | 設計値 |
|---|---|
| OSPFエリア | area 0 |
| Router ID | 各ノードのLoopback0 |
| インフラリンク種別 | Point-to-Point |
| Core—Accessコスト | すべて同一 |
| 利用者セグメント | OSPF広告対象、隣接形成対象外 |
| ECMP | core1/core2の2ネクストホップ |

access1からaccess2配下、access2からaccess1配下への最短コストを同一にする。Core—Coreリンクは通常のAccess間最短経路ではなく、障害時やCore間到達性のために使用する。

## 6. 冗長化・障害設計

| 障害 | 正常時 | 障害時の期待動作 |
|---|---|---|
| Access—Core単一リンク断 | 2つの等コスト経路 | 残存Core経由の1経路へ収束 |
| core1停止 | core1/core2を併用 | core2経由へ収束 |
| core1復旧 | — | OSPF再隣接後、2経路へ復元 |
| pc側リンク断 | 単一リンク | 冗長性なし。端末収容リンク障害として停止 |

## 7. 管理設計

- containerlab管理ネットワークは`campus-routed-access-mgmt`とする。
- 管理IPは`172.24.24.0/24`から固定する。
- 管理ネットワークはOSPFへ広告しない。
- Theme 23と同じく、IOLをPID1にするentrypointを使用して`docker attach`可能にする。
- attachはIOSコンソールへ直接接続し、初期設定は学習者が行う。
- deploy後、仮想リンクの橋渡しに必要な`iouyap 513`だけを環境側で起動する。
- 学習対象のEthernet0/1以降、Loopback、OSPF、端末データプレーン設定は投入しない。

## 8. 構築境界

### 環境側で準備するもの

- ノード6台
- 仮想リンク7本
- ノード名
- attach可能なIOL entrypointと`iouyap`起動
- Cisco IOLとLinux端末の起動

### 学習者が設定するもの

- インフラリンクIP
- Loopback0
- OSPF
- 端末側データプレーンIP
- 端末のデフォルト経路または対向セグメントへの経路
- 障害試験後の復旧
