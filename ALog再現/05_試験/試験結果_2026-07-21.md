# 試験結果: ALog再現（実機検証 2026-07-21）

商用SIEM「ALog」の機能ギャップ（多様なログソース収集・翻訳変換・3観点検知・アラート通知・S3長期保管）をOSSで再現したラボを、OrbStack VM `clab`（arm64/aarch64、10コア/19GiB）で実機デプロイし、[試験計画書](試験計画書.md)のT-A1〜T-A6を検証した。**全項目合格**。

## 環境
- ホスト: OrbStack VM `clab`（aarch64）、docker 29.1.3
- スタック: 9コンテナ（logsrc / scanner / vector / loki / promtail / grafana / minio / suricata / webhook）、ネットワーク `alog-lan`(172.41.0.0/24, bridge `alog-br0`)
- ポート: Grafana 3001 / Loki 3101 / MinIO 9001-9002 / Vector syslog 5514 / Webhook 9099
- 使用イメージ（全arm64ネイティブ実測）: grafana/loki, grafana/promtail, grafana/grafana:13.1.0, jasonish/suricata, timberio/vector:0.57.0-alpine, minio/minio, wbitt/network-multitool

## 詳細エビデンス

### T-A1 多様なログソース集約
```
Lokiのjobラベル: ['alog', 'suricata']
job=alog  (Vector/syslog):  auth_fail / priv_exec / ad_logon_fail / nw_if_down
job=suricata (Promtail/eve.json): alert 536件 / flow 267件 / stats 89件
```
syslog（AD/サーバ/NW機器の代表）とNDR検知ログ（Suricata）の2系統が単一のLokiに集約されることを確認。

### T-A2 翻訳変換（正規化）
Vector VRLが生syslogを構造化し、人間可読な日本語 `message_ja` を付与（ALogの特許「翻訳変換」の思想の再現）:
```
[auth_fail]     test が 198.51.100.20 からのSSHログインに失敗した
[priv_exec]     alice が管理者権限(sudo)で「/usr/bin/cat /etc/shadow」を実行した
[ad_logon_fail] bob のドメインログオンが失敗した（送信元 203.0.113.12）
[nw_if_down]    GigabitEthernet0/1 のインターフェース状態が down に変化した
```

### T-A3 件数変化検知
```
sum(count_over_time({job="alog",event="auth_fail"}[5m])) = 22件 （閾値10超）
→ Grafana Alerting「件数の変化: 認証失敗の急増」= active(firing)
```

### T-A4 新規出現検知（真の初回出現の識別）
LogQL集合差 `sum by(src_ip)(...[5m]) unless sum by(src_ip)(...[1h] offset 5m)` により実装。
ベースライン注入から5分経過後に新IP 203.0.113.201 を注入し、クエリを実行:
```
新規出現IP: 203.0.113.201 = 1件
（既知IP 203.0.113.10 / 203.0.113.11 / 198.51.100.20 / 203.0.113.200 は正しく除外）
```
「everything is new」ではなく、既知IPを除外して真に新しいIPのみを検出する識別能力を確認。

### T-A5 アラート通知（Webhook）
Grafana Alerting の Webhook contact point が socat 受信器へHTTP POST。webhook.log の集計:
```
"件数の変化: 認証失敗の急増"          3件
"値の変化: イベント件数の急増"        3件
"新規出現: 未知の送信元IPからの認証失敗" 11件
（POST / HTTP/1.1 ヘッダ・JSONボディを確認）
```
計画の最低要件（件数変化＋新規出現）を超え、3観点すべての発報がWebhookに到達。

### T-A6 MinIO長期保管（S3オフロード）
Lokiのobject storeをMinIO(S3互換)に設定。`loki_cluster_seed.json`・index に加え、`POST /flush` 後にchunk本体が出現:
```
fake/523818318b74994c/19f84c88656:...   ← ログchunk本体
fake/624f652105daf96a/...
（計7 chunkオブジェクト） + index/delete_requests/ + loki_cluster_seed.json
```
LokiがローカルFSではなくMinIOへログchunkをオフロードすることを確認。

## デプロイ中に判明・修正した事項（詳細は[構築ログ](../04_構築/構築ログ_2026-07-21.md)）
1. `wbitt/network-multitool` に `ncat` 非同梱（nc/nmap/curl/socat はあり）→ syslog送信を `nc -w1`、webhook受信を `socat` に変更
2. 新しいVector(0.57)は静的プレフィックス無しの動的ラベル `{{ event }}` を既定拒否 → Lokiシンクに `dangerously_allow_unconfined_template_resolution: true` を明示（学習用途、ラベル基数は少数値に限定）
3. VRLの文字列連結が `string|null` 型で E103/E651 が交互発生 → キャプチャを `to_string()` で包み解決
4. RFC3164 syslogのTZ無しタイムスタンプをVectorがUTC解釈 → JSTのままだとLokiが "timestamp too new" 拒否 → `date -u` でUTC出力に統一

## 再現度の位置づけ（過大評価しない）
- 3観点検知は**AIの教師なし学習ではなく、固定閾値・固定時間窓のルールベース近似**（[OSS対応表](../02_基本設計/網屋ALog_OSS対応表.md)の通り）。「3観点という検知の考え方」の再現であり、ALogのAI適応能力は再現していない。
- リスクスコアリング・監視テンプレート・マルチノード・MDR運用代行・課金モデルは対象外（❌）。
- 本ラボはALogの**中核データパイプライン（収集→正規化→検知→通知→保管）**をOSSで通しで再現し実機実証したもの。
