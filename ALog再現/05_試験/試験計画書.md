# 試験計画書: ALog再現

> **2026-07-21 実施済み。T-A1〜T-A6 全項目合格。** 実測値・エビデンスは [試験結果_2026-07-21.md](試験結果_2026-07-21.md) を参照（本計画書の各行「判定」列は結果ドキュメントに集約）。

## 試験方針

ALogの機能ギャップ（多様なログソース収集・翻訳変換・3観点検知・アラート通知・S3長期保管・ダッシュボード）が、[基本設計書](../02_基本設計/基本設計書.md)の設計通りに実際のデプロイで動作することを、T-A1〜T-A6の6項目で確認する。実デプロイ・判定はメインセッションが別途行う（本タスクではdocker/ssh/デプロイを実施しない）。判定結果は`05_試験/試験結果_YYYY-MM-DD.md`として実デプロイ後に新規作成する。

### T-A1 多様なログソース集約
- [ ] `deploy.sh deploy`後、`curl -s http://localhost:3101/ready`が200を返すことを確認する。
- [ ] `deploy.sh logs`後、`{job="alog"}`でLoki検索し、syslog由来のログが取得できることを確認する。
- [ ] `deploy.sh attack`のSYNスキャン部分実行後、`{job="suricata"}`でLoki検索し、eve.json由来のログが取得できることを確認する。

### T-A2 翻訳変換
- [ ] `{job="alog"} | json | event="auth_fail"`で検索し、`user`/`src_ip`フィールドが正しく抽出されていることを確認する。
- [ ] `{job="alog"} | json | event="priv_exec"`で検索し、`command`フィールドが正しく抽出されていることを確認する。
- [ ] `message_ja`フィールドが日本語の自然文になっていることを目視確認する。

### T-A3 件数変化検知
- [ ] `deploy.sh query`の(a)クエリで、直近5分のauth_fail件数が10を超えていることを確認する。
- [ ] `deploy.sh alert`で`alog-count-change-auth-fail`がfiring状態であることを確認する。

### T-A4 新規出現検知
- [ ] `deploy.sh query`の(b)クエリ結果に`src_ip=203.0.113.200`のみが含まれ、既知IP（203.0.113.10/11, 198.51.100.20）が含まれないことを確認する。
- [ ] `deploy.sh alert`で`alog-new-appearance-src-ip`がfiring状態であることを確認する。

### T-A5 アラート発報・Webhook通知
- [ ] `webhook/log/webhook.log`に少なくとも1件のPOSTリクエストが記録されていることを確認する。
- [ ] 記録内容に`"axis":"count_change"`または`"axis":"new_appearance"`または`"axis":"value_change"`のいずれかが含まれることを確認する。

### T-A6 MinIO長期保管
- [ ] MinIO Console(http://localhost:9002)にログインし、bucket=loki配下にオブジェクトが存在することを確認する。
- [ ] `sudo docker logs loki`にS3接続エラー（`AccessDenied`/`NoSuchBucket`等）が出力されていないことを確認する。

## 合格基準

- T-A1〜T-A6の必須項目が全て合格していること。
- T-A3・T-A4は「`logs`実行から5分以上待ってから`attack`を実行する」という時刻依存の手順を守った上で判定すること（[要件定義書](../01_要件定義/要件定義書.md) 制約参照）。
- 上記が確認でき次第、`05_試験/試験結果_YYYY-MM-DD.md`を新規作成し実測値を記録する（本テーマの範囲外、実デプロイ後にメインセッションが実施）。
