Theme 26 · Dynamic Routing Deep Dive · Phase 7

3つの言葉と、
2つの関所。

Phase 7 で何をしているのか。ひとことで言うと「別々のルーティングの言葉をつなぐ翻訳者を置き、その翻訳者に関所を設ける」作業です。設定コマンドの前に、その狙いを図で掴みましょう。

社内には EIGRP・OSPF・BGP という3つの経路の言葉があり、互いに通じません。Phase 7 は、その境目に「翻訳者（再配信）」を置き、翻訳者に「関所（route-map / prefix-list）」を設ける作業です。

関所が選り分けるのは「通信」ではなく「経路情報（＝宛先までの道案内）」です。余計な道案内が漏れたり、ぐるぐる回ったりしないように絞るのが目的です。

!

最初に押さえること：route-map / prefix-list は「通信フィルタ」ではありません。 ここで絞るのはルータ同士が交換する道案内です。PC の ping や HTTP を直接止めるのは ACL やファイアウォールの役目で、Phase 7 の対象外です。

Q. なぜ設定するの？

そもそも、なぜ再配信が要るのか

3つの言葉はお互いの経路を自動では共有しません。だから hq-dist（EIGRP しか話さない）は、放っておくと DMZ（10.26.30.0/24）への道を一生知りません。そこをつなぐのが「再配信（redistribute）」＝翻訳者です。

E

EIGRP（AS 100）

HQ の一角だけで動く Cisco 系の言葉。hq-dist ⇄ hq-core 間で HQ 内勤ネット 10.26.10.0/24 を運ぶ。

O

OSPF（Process 10）

HQ の Area 0 と支社の Area 1。同じ OSPF でも WAN で分断され、実は2つの島。DMZ 10.26.30.0/24 や支社 10.26.20.0/24 を運ぶ。

B

BGP（eBGP）

拠点間 WAN の言葉。hq-edge(65000) ⇄ isp(65100) ⇄ br-edge(65001)。誰に何を見せるかを細かく制御できる。

Phase 7 のゴール（このために翻訳する）

① hq-pc（10.26.10.10）から dmz-srv（10.26.30.10）へ社内で到達できる。② hq-pc から br-pc（10.26.20.10）へ WAN 越しに到達できる。翻訳が無ければ、この2つは経路が無くて届きません。

Q. どのIPを、どこからどこへ？

2つの関所を開けてみる

関所ごとに「どの言葉から、どの言葉へ」「どのプレフィックスを通し、何を止めるか」を並べます。ここが Phase 7 の実体です。

①hq-core

EIGRP ⇄ OSPF の相互再配信

EIGRP AS100 ⇄ OSPF Area0
通す 10.26.0.0/16 le 32 配下すべて 社内で使う 10.26.x.x の実ネット（/24・/30・/32）を双方向に翻訳する。
止める 上記に一致しない経路（外部・想定外） route-map の permit に載らない経路は末尾で暗黙 deny。誤って外部経路が EIGRP へ逆流するのを防ぐ。
なぜここが一番厳重か： 唯一「IGP どうしを双方向で」翻訳する場所だから。行きと帰りが同じルータに集まり、フィルタが甘いと経路がぐるっと戻ってループになりやすい。だから route-map ＋ prefix-list で対象を明示的に絞る。

le 32 が要る理由：permit 10.26.0.0/16 だけだと /16 ちょうどにしか一致しない。le 32 を足して、内側の /24・/30・/32 まで対象に含める。

②br-edge

支社が ISP へ見せる経路を限定

OSPF Area1 → ISP (BGP)
通す 10.26.20.0/24 のみ 支社の利用者ネットだけを ISP へ広告。外から支社に届くための最小限。
止める 10.26.2.0/30・Loopback /32 など Edge–Core リンクやルータ Loopback といった内部基盤は ISP に見せない（暗黙 deny）。
なぜ絞るか： 内部の設備アドレスを外に晒さないため。加えて、支社が HQ 行きの経路まで広告すると「支社が HQ への中継地（トランジット）」に化けてしまう。出す経路を 10.26.20.0/24 だけに固定して、その事故を防ぐ。

向きは片方向（社内 → ISP の OUT）。IGP 相互再配信ほど複雑な route-map は要らず、prefix-list 一枚で足りる。

Q. なぜ EIGRP だけ特別なの？

「EIGRP だけ特別」ではない

route-map / prefix-list は EIGRP 専用の道具ではありません。境目ごとに使う道具が違うのは、そこで起きうる事故（リスク）の大きさが違うから。道具の重さは、リスクの重さに合わせています。

境目向きいちばんの心配ごと使う道具
hq-core
EIGRP ⇄ OSPF		 双方向
（IGP ⇄ IGP）		 再配信ループ・経路の逆流。行き帰りが1台に集まる最難所。 route-map + prefix-list
br-edge
OSPF → BGP		 片方向
（社内 → ISP）		 内部経路の漏洩・意図しないトランジット化。 prefix-list（OUT）
hq-edge / br-edge
BGP → OSPF		 片方向
（default 注入）		 ほぼ無い。入れるのは 0.0.0.0/0 の1本だけ。 default-information originate

結論：道具の重さ ＝ その関所のリスクの重さ。 一番危ない hq-core だけが route-map + prefix-list の“フル装備”に見えるので「EIGRP だけ特別」と感じるだけです。

Q. なぜループしないの？

ループはなぜ起きて、なぜ止まるか

再配信ループは「ある言葉に出した経路が、別の境目を通って元の言葉に戻ってくる」ときに起きます。今回の構成で止まる理由は2つです。

✕もしフィルタが無ければ

  • OSPF に入った経路が、関所を素通りして EIGRP へ。
  • EIGRP 側でメトリックが付き直り、別の道として OSPF に戻る。
  • 「どっちが本物？」をルータが決められず、経路がバタつく／最悪ぐるぐる回る。

✓今回止まる理由

  • 再配信点は hq-core ただ1台。 EIGRP 側の隣は hq-dist だけで、戻り口が他に無い。
  • route-map で対象を明示。 通すのは 10.26.0.0/16 配下に限定し、想定外の経路は暗黙 deny。
正直な注意： これは「絶対にループしない」保証ではありません。10.26.0.0/16 le 32 は HQ・DMZ・支社をまとめて広く通します。将来、再配信点を2台に増やすなら、再配信時に route tag を付け、反対方向では自分のタグを deny する方式を足すのが定石です（今の Phase 7 の必須設定ではなく、拡張時の設計候補）。
Q. どのIPが、どこからどこへ？

経路情報の行き先を一覧で

prefix-list に書くのは「端末1台の IP」ではなく、ルーティングテーブルに載る宛先ネットワークです。どのネットが、どの言葉からどの言葉へ渡るか。

10.26.10.0/24HQ内勤 · hq-pc が所属 EIGRP →〔関所①〕→ OSPF。hq-edge や hq-dmz にも「内勤ネットへの道」が伝わる。
10.26.30.0/24DMZ · dmz-srv が所属 OSPF →〔関所①〕→ EIGRP。hq-dist が DMZ への道を知り、hq-pc→dmz-srv が成立。
10.26.20.0/24支社 · br-pc が所属 OSPF Area1 →〔関所②〕→ BGP → ISP。支社ネットだけを外へ広告。
0.0.0.0/0デフォルト経路（それ以外ぜんぶ） ISP → BGP → 両Edgeが OSPF へ注入 → HQ では〔関所①〕→ EIGRP。個別に知らない宛先の受け皿。

読み方のコツ： 10.26.20.10（端末）ではなく 10.26.20.0/24（そのネット）を書く。フィルタが見るのは「宛先ネットワークとマスク長」で、個々の PC ではありません。

確認の勘所

投入後は show コマンドで“翻訳”を確かめる

この教材は設定コマンドの答えは載せません。 変換して投入するのは学習者の役目です。ここでは「投入後に何を確認すればよいか」の勘所（show 系のヒント）だけを置きます。
show ip route

再配信された経路は種別で見分ける。O E2（EIGRP由来がOSPFへ）・D EX（OSPF由来がEIGRPへ）・B*（BGPのデフォルト）。期待どおりの種別と next-hop か。

show ip prefix-list

permit 条件と hit count を確認。0 のままなら一致していない＝マスク長や le 32 の付け忘れを疑う。暗黙 deny の影響も意識する。

show route-map

sequence 番号・match 条件・permit/deny の並びを確認。参照している prefix-list 名が合っているか（打ち間違いは全落ちの原因）。

show ip protocols

再配信元・適用中の route-map 名・EIGRP のシード metric を一望。意図した再配信が有効になっているか。

show ip bgp

支社が広告しているのは 10.26.20.0/24 だけか。0.0.0.0/0 を受信できているか。関所②と default 注入の前提を確認。

show ip eigrp topology

再配信された外部経路が EIGRP トポロジに載っているか。hq-dist から DMZ / default が見えるかを最終確認。
