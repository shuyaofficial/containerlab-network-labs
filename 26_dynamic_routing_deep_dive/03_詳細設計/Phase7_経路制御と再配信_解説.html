最初に押さえること：経路フィルタと通信フィルタは別物です。

hq-core

OSPF ↔ EIGRPの再配信

現行Phase 7をIOSの構造に落とした学習例です。先にprefix-listとroute-mapを作り、その後で再配信へ適用します。

  • OSPF → EIGRPDMZやデフォルト経路などをEIGRP外部経路（D EX）としてhq-distへ渡す。
  • EIGRP → OSPFhq-pc側の経路をOSPF外部経路（O E2）としてhq-edge・hq-dmzへ渡す。
HQ-CORE · 学習用CLI例
ip prefix-list PL-EIGRP-TO-OSPF seq 10 permit 10.26.0.0/16 le 32

route-map RM-EIGRP-TO-OSPF permit 10
 match ip address prefix-list PL-EIGRP-TO-OSPF

router eigrp 100
 redistribute ospf 10 metric 1000000 10 255 1 1500

router ospf 10
 redistribute eigrp 100 subnets route-map RM-EIGRP-TO-OSPF
le 32 が必要な理由： permit 10.26.0.0/16 だけなら原則として /16そのものだけに一致します。le 32 を付けると、10.26.0.0/16の内側にある /24・/30・/32も対象になります。
暗黙deny： route-mapのpermit節に一致しない経路は末尾で拒否されます。名前の打ち間違い、prefix-listの未作成、条件の不足があると、必要な経路も一件も再配信されません。
br-edge / hq-edge

BGP広告とDefault Route

br-edgeのOUTフィルタは「支社からISPへ見せる経路」を限定します。ISPから受けたデフォルトは、各EdgeがOSPFへ注入します。

BR-EDGE · 支社プレフィックスだけをISPへ
ip prefix-list PL-BRANCH-OUT seq 10 permit 10.26.20.0/24

router bgp 65001
 network 10.26.20.0 mask 255.255.255.0
 neighbor 198.51.100.5 remote-as 65100
 neighbor 198.51.100.5 prefix-list PL-BRANCH-OUT out
ISP → EDGE → OSPF · Default Routeの流れ
! isp（概念例）
router bgp 65100
 neighbor 198.51.100.1 default-originate
 neighbor 198.51.100.6 default-originate

! hq-edge / br-edge
router ospf 10
 default-information originate
成立条件： 通常の default-information originate は、そのEdgeのルーティングテーブルに 0.0.0.0/0 が存在するときだけOSPFへ広告します。BGP受信を先に確認します。
現行資料の未確定点： hq-edgeの 10.26.0.0/16 aggregate広告は、BGPテーブル内に構成要素となる経路が必要です。Phase 7だけでは生成方法が確定しないため、このHTMLでは設計上の前提として扱います。

Loop prevention

なぜ現行構成ではループしにくいか

再配信ループは「ある領域から出した経路が、別の境界を通って元の領域へ戻る」ときに起きやすくなります。

  • OSPFとEIGRPの再配信点は hq-core 1台だけ。
  • hq-coreのEIGRP側ネイバーもhq-dist 1台で、別の戻り口がない。
  • EIGRP→OSPFにはroute-mapがあり、少なくとも対象を10.26.0.0/16配下へ限定する。
ただし完全なループ防止策ではありません。 10.26.0.0/16 le 32 はHQ・DMZ・Branchの多くをまとめて許可します。将来、再配信点を2台に増やすと、OSPF由来の10.26.x経路がEIGRPから戻ってきても一致する可能性があります。

将来の複数境界構成

再配信時にroute tagを付け、反対方向では「自分が付けたtagをdeny」する方式を追加します。これは現行Phase 7の主設定ではなく、拡張時の設計候補です。

Address reading

どのIPアドレスを指定しているのか

prefix-listに書くのは原則として「端末1台のIP」ではなく、ルーティングテーブルに載る宛先ネットワークです。

  • 10.26.10.0/24hq-pc 10.26.10.10 が所属するHQ LAN。EIGRP側からOSPF側へ知らせたい。
  • 10.26.30.0/24dmz-srv 10.26.30.10 が所属するDMZ LAN。OSPF側からEIGRP側へ知らせたい。
  • 10.26.20.0/24br-pc 10.26.20.10 が所属するBranch LAN。br-edgeからISPへ広告を許可する。
  • 0.0.0.0/0個別に知らない宛先をEdgeへ送るためのデフォルト経路。ISP→BGP→OSPF→EIGRPの順で届く。
Safe sequence & verification

投入順と確認ポイント

「ポリシーを作ってから適用する」がPhase 7の肝です。適用後は保存より先に、経路種別とフィルタのヒット状況を確認します。

  1. 既存経路を記録show ip route でOSPF・EIGRP・BGPの現状を残す。
  2. prefix-list作成許可するプレフィックスとマスク長を先に定義する。
  3. route-map作成prefix-listをmatchし、permit/denyの順序を確認する。
  4. 再配信へ適用hq-coreでOSPF↔EIGRP、EdgeでDefault Routeを注入する。
  5. 検証後に保存期待経路と疎通を確認してからrunning-configを保存する。
show ip routeO E2、D EX、B* の有無とnext-hop
show route-mapsequence、match条件、ポリシー適用状況
show ip prefix-listpermit条件とhit count、暗黙denyの影響
show ip protocols再配信元、route-map名、EIGRPメトリック
show ip bgpDefaultと10.26.20.0/24の受信・広告状況
このCLIはCisco IOL 15.7系ラボ向けの学習例です。実機へ投入する場合は、現行config、管理アクセス、ロールバック手順を確認してから最小単位で適用してください。
