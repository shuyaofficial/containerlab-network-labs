# 基本設計

## 設計方針

- Theme 22を汚さないため、Containerlab名、管理ネットワーク、作業フォルダを完全に分ける。
- ShowNet 2026の物理的な大規模性は、少数ノードの役割分担で抽象化する。
- Cisco IOLで再現しやすいBGP/OSPF/ACL/経路観測を中心にする。
- Media over IPは、マルチキャストやPTPの完全再現ではなく、UDP/RTP風の疎通と遅延観測の練習として扱う。

## 論理構成

```text
             ext-bbix AS65010              ext-jpix AS65020
                    |                             |
              edge-n1 AS290  ===== iBGP ==== edge-n2 AS290
                 /      \                     /      \
            core-n3a ---- core-n3b ---- core-n3a/core-n3b
                 \          /                \
                  \        /                  sec-s4 -- remote-s5
                   transport-n4
                  /      |       \
             media-x  yokohama  hamamatsu

             monitor-ai は core-n3a 配下の観測点
```

## ノード役割

| ノード | 役割 |
|---|---|
| `ext-bbix` | 外部IX/ISP相当。AS65010。edge-n1とeBGP |
| `ext-jpix` | 外部IX/ISP相当。AS65020。edge-n2とeBGP |
| `edge-n1` | ShowNet AS290対外接続ルータ #N-1相当 |
| `edge-n2` | ShowNet AS290対外接続ルータ #N-2相当 |
| `core-n3a` | ShowNetコア #N-3相当 |
| `core-n3b` | ShowNetコア #N-3相当 |
| `transport-n4` | IOWN/APN/大容量トランスポート #N-4相当 |
| `media-x` | 幕張Media-X拠点 |
| `yokohama-foh` | 横浜FoH/ミキシング拠点 |
| `hamamatsu-mocap` | 浜松モーションキャプチャ拠点 |
| `monitor-ai` | AI/統合監視の観測点 |
| `sec-s4` | サイバー脅威検出/制御点 |
| `remote-s5` | セキュアリモートアクセス入口 |

## ルーティング

- AS290内部はOSPF process `290` を使う。
- `edge-n1` と `edge-n2` はloopback間でiBGPを張る。
- `edge-n1` は `ext-bbix`、`edge-n2` は `ext-jpix` とeBGPを張る。
- AS290から外部へは `10.90.0.0/16` のラボ集約を広告する。
- 外部ASからAS290へはデフォルトルートを広告する。

## セキュリティ

- `sec-s4` に `REMOTE-S5-IN` ACLを置く。
- `remote-s5` からは管理/監視系 `10.90.0.0/24`, `10.90.60.0/24` を許可する。
- `remote-s5` からMedia over IP系 `10.90.40.0/22` への直接到達は拒否する。

## 制約

- Cisco IOLのSRv6 uSID対応は期待しない。
- PTPやST 2110の完全再現はしない。
- LinuxコンテナとCisco IOLのL2相性問題を避けるため、L3接続中心にする。
- 起動リソースが重い場合は、Theme 22を止めずに、このラボのdeployを後回しにする。

## 拡張案

- CSR1000v/C8000vを使える余力がある場合、SRv6やより実機に近いBGP policyを後続拡張する。
- `monitor-ai` にPrometheus/Zabbix相当を追加する。
- `media-x` 系にiperf3やPTPツールを追加した専用Linuxイメージを使う。
