# 基本設計書 — Theme 25: Multi-Campus

> 対応要件: [要件定義書](../01_要件定義/要件定義書.md)

## 1. アーキテクチャ概要

Multi-Campusでは、各キャンパスを独立したIGPドメイン（OSPF）で構成し、境界ルーター間をeBGPで接続する。キャンパス内部の詳細経路は相手拠点へ漏らさず、`/16`集約のみを広告する。

```text
   [Main Campus]                     [Branch Campus]
                                 
   [main-pc]                         [branch-pc]
      |                                  |
   [main-core]                       [branch-core]
      |  OSPF 10                         |  OSPF 20
   [main-edge]------- WAN -------[branch-edge]
    AS 65050        eBGP           AS 65060

   Main: 10.50.0.0/16              Branch: 10.60.0.0/16
```

詳細図は[ネットワーク物理構成図](ネットワーク物理構成図.mermaid)を参照する。

## 2. 各層の責務

| 層 | ノード | 責務 | この構成で使用しないもの |
|---|---|---|---|
| Edge / BGP境界 | main-edge、branch-edge | 拠点間eBGP、集約広告、再配信制御 | VLAN、STP、HSRP |
| Core / 内部 | main-core、branch-core | キャンパス内OSPF、利用者GW | eBGP（直接は話さない） |
| Endpoint | main-pc、branch-pc | E2E疎通と障害試験の送受信 | 動的ルーティング |

## 3. 物理・リンク設計

| リンク種別 | 本数 | アドレス長 | 用途 |
|---|---:|---:|---|
| キャンパス内（Core—Edge） | 2 | /30 | OSPF隣接 |
| WAN（Edge—Edge） | 1 | /30 | eBGP |
| 端末収容（Core—PC） | 2 | /24 | 利用者セグメント |

## 5. OSPF設計

### 5.1 Main Campus — OSPF Process 10

| 項目 | 設計値 |
|---|---|
| Process ID | 10 |
| Area | 0 |
| 参加IF | main-core—main-edge間P2P、各Loopback、利用者セグメント |
| Router ID | 各ノードのLoopback0 |
| WANリンク | OSPF参加させない |

### 5.2 Branch Campus — OSPF Process 20

| 項目 | 設計値 |
|---|---|
| Process ID | 20 |
| Area | 0 |
| 参加IF | branch-edge—branch-core間P2P、各Loopback、利用者セグメント |
| Router ID | 各ノードのLoopback0 |
| WANリンク | OSPF参加させない |

> 両キャンパスでarea 0を使用するが、BGP境界を越えて同じOSPFドメインにはならない。プロセス番号は識別用であり、対向とは無関係。

## 7. 集約・再配信設計

### 7.1 OSPF → BGP方向

1. OSPFで学習した自拠点プレフィックスをBGPへ渡す。
2. prefix-listで自拠点範囲（Main: `10.50.0.0/16`、Branch: `10.60.0.0/16`）だけを許可する。
3. route-mapでprefix-listを適用する。
4. BGP `aggregate-address`で`/16`集約を生成し、`summary-only`で`/24`の個別広告を抑止する。

### 7.2 BGP → OSPF方向

1. BGPで受信した相手拠点の`/16`をOSPFへ再配信する。
2. prefix-listで相手拠点の`/16`だけを許可する。
3. route-mapでtagを付けて再配信する（Main: tag 65060、Branch: tag 65050など）。
4. tagにより、再配信された経路がBGPへ再注入されることを防止する。

### 7.3 ループ防止

- 片方向だけ再配信すると到達性が欠ける。双方向で制御する。
- route-mapとtagで「相手から受けた経路を相手へ戻さない」ことを保証する。
- `show ip route`の経路種別（O E2 vs O、BGP vs connected）で経路の出自を判別する。

## 8. 管理設計

- containerlab管理ネットワークは`campus-multi-site-mgmt`とする。
- 管理IPは`172.25.25.0/24`から固定する。
- 管理ネットワークはOSPF/BGPへ広告しない。
- IOLをPID1にするentrypointを使用して`docker attach`可能にする。
- deploy後、仮想リンクの橋渡しに必要な`iouyap 513`だけを環境側で起動する。
- PC（Linux端末）のIPとスタティックルートは環境側で自動設定する。
- 学習対象のCisco IOLへのコンフィグ投入は学習者が行う。

## 9. 構築境界

### 環境側で準備するもの

- ノード6台
- 仮想リンク5本
- ノード名
- attach可能なIOL entrypointと`iouyap`起動
- Cisco IOLとLinux端末の起動
- PCのデータプレーンIP・静的経路

### 学習者が設定するもの

- Cisco IOLのインフラリンクIP
- Loopback0
- OSPF（process 10 / 20）
- eBGP（AS 65050 / 65060）
- prefix-list、route-map
- 集約経路（aggregate-address）
- 再配信設定（redistribute）
- tagによるループ防止
- 追加Mission: IPsec
