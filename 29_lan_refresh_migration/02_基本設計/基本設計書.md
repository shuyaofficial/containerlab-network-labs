# 基本設計書: 社内LAN更改・移行・移設

## 1. 新環境設計の方針
本テーマの更改は、老朽LAN（old-rt/old-core/old-sw1/old-sw2）を移行元とし、新環境（new-core1/new-core2/new-rt）を新設したうえで、VLAN単位の段階移行によってGWを新環境へ切り替えていく設計とする。フロアSW（old-sw1/old-sw2）と監視基盤（mon-core/cap-sw）はそのまま存置し、コアSWとWANルータのみを更改対象とする。

要件定義書§2の更改内容を、以下の3点で実現する。

1. **冗長コア設計**: old-coreのSPOFを、new-core1/new-core2の物理2台構成＋HSRP＋STPルート設計で解消する。
2. **動的ルーティング設計**: old-rt⇔old-core間のstatic routeを、new-core⇔new-rt間のOSPF（area 0）に置き換える。
3. **新旧並行稼働設計**: 移行完了までの間、旧環境（old-core）と新環境（new-core1/2）を同一フロアSW配下で並行稼働させ、VLANごとに順次GWを切り替える。並行稼働期間中のL2ループはSTPで防止する。

## 2. 冗長コア設計（new-core1/new-core2）
- new-core1・new-core2の2台を新コアとして新設し、両者間をEthernet0/1同士でトランク接続（VLAN10,20,30,90）する。
- new-core1・new-core2それぞれから、フロアSW（old-sw1・old-sw2）へ二重ホーム接続する。
  - new-core1: Ethernet0/2 ⇔ old-sw1 Ethernet1/0、Ethernet0/3 ⇔ old-sw2 Ethernet0/3
  - new-core2: Ethernet0/2 ⇔ old-sw1 Ethernet1/1、Ethernet0/3 ⇔ old-sw2 Ethernet1/0
  - いずれもトランク（VLAN10,20,30,90）とし、カットオーバーまではshutdownで待機する（§6参照）。
- old-coreのような単一機への集約ではなく、物理的に独立した2台へVLAN SVIとHSRPを分散配置することで、片系停止時も残る1台でサービスを継続できる構成とする。

> 参考: 書籍『ネットワーク・デザインパターン』のTrustゾーン設計パターン（`参考資料/ネットワーク技術/ネットワーク・デザインパターン/_index/text/p026.md`、`p051.md`）は、社員用VLANとゲスト用VLANの分離や、コアスイッチの冗長化（VSS/スタック構成）とアップリンクのLAG構成といった、コアスイッチを冗長化しつつVLAN設計を維持するパターンを扱っている。本テーマではVSS/スタックではなくHSRPペア構成を採用するが、「コアを物理的に複数台へ分散し、片系障害の影響範囲を局所化する」という設計思想は共通する。

## 3. HSRP設計
- 各VLAN（10/20/30/90）のSVIを、new-core1・new-core2の両方に設定する。
  - new-core1: `<VLAN>.2/24`、new-core2: `<VLAN>.3/24`
  - 仮想IP（VIP）は現行のGWアドレス（`<VLAN>.1`）をそのまま用いる（`standby <VLAN番号> ip <VLAN>.1`）＝**IP温存によるGW移行**。
- new-core1を優先系（priority 110、preempt有効）、new-core2を予備系（priority 100、preempt有効）とし、常にnew-core1がActiveになるよう設計する。
- preemptを両系で有効にすることで、new-core1が復旧・再起動した際に自動的にActiveへ復帰する。これにより、新環境単体試験（試験計画書Mission3）でnew-core1停止→new-core2 Active昇格→new-core1復旧→Active復帰、という一連のフェイルオーバー動作を検証できる。
- 端末側のGW設定・ARPは変更不要である。移行の瞬間（old-core SVIのshutdown・new-core SVIのno shutdown＋standby preempt）にVIPの実体がold-coreからnew-core側へ切り替わるのみであり、端末は同一IP（`<VLAN>.1`）に対してGratuitous ARPで新しいMACアドレスを学習し直す。

## 4. STPルート設計
- new-core1に`spanning-tree vlan 10,20,30,90 root primary`、new-core2に`spanning-tree vlan 10,20,30,90 root secondary`を設定し、新環境をSTPのルートブリッジとする。
- **設計根拠**: カットオーバー前のP1（新旧トランク開通）フェーズでは、old-core・old-sw1・old-sw2・new-core1・new-core2の間に複数の物理経路（旧トランク＋新トランク＋新コア間トランク）が存在し、L2ループが形成され得る。old-core・old-sw1・old-sw2はブリッジ優先度を変更せずデフォルト（32768）のままとするため、new-core1（root primary）・new-core2（root secondary）を明示的にルートブリッジ／セカンダリルートブリッジとすることで、STPの再収束後にどちらが論理的な頂点になるかを設計として固定する。
- old-coreを含む区間でも、STPが自律的に冗長経路を検出してブロッキングポートを選定し、ループを防止する。ただし、具体的にどの物理ポートがブロッキングになるかは、実機のトポロジ計算に依存するため、カットオーバーMOP P1で`show spanning-tree vlan <VLAN>`により実際の収束結果を確認する手順としている（このタイミングで再収束に伴う瞬断が起こり得ることも、MOPの確認項目としている）。
- old-coreの撤去（P8）後は、new-core1がルートブリッジのまま新環境のみのシンプルなツリー構成に収束する。

## 5. OSPF設計
- new-rt⇔new-core1間（10.29.1.0/30）、new-rt⇔new-core2間（10.29.2.0/30）をOSPFプロセス1・area 0で接続する。
- **new-rt**: 10.29.1.0/30・10.29.2.0/30・Loopback0（203.0.113.2）をarea 0で広報し、`default-information originate always`によりデフォルトルートを常時OSPFへ注入する。これにより、new-core1/new-core2は静的なdefault routeを持たずとも、OSPF経由でWAN方向のデフォルトルートを学習できる。
- **new-core1/new-core2**: 10.29系リンクでOSPFを有効化し、ユーザーVLAN SVI（VLAN10/20/30/90）は`passive-interface`としたうえで`network`文でOSPFへ広報する（SVI配下に隣接を形成する必要はなく、経路広報のみが目的のため）。
- **new-core1**: mon-core向けのEthernet1/1（10.29.0.0/30）はmon-core側がOSPFを話さないため、OSPFのnetwork文には含めない。代わりに`ip route 10.28.100.0 255.255.255.0 10.29.0.1`（監視戻り経路）を静的に設定し、`redistribute static`でOSPFへ再配布する。これにより、new-core2・new-rtも含めた新環境全体から監視セグメント（10.28.100.0/24、nms収容）への経路が確保される。
- static routeがOSPFに置き換わることで、WANルータ更改後も経路情報を手動で更新する必要がなくなり、new-rt・new-core1・new-core2いずれかの追加・変更時の運用負荷を下げる設計とした。

## 6. 新旧並行稼働の設計（事前配線＋shutdown待機）
- 新環境（new-core1/new-core2/new-rt、および新コア⇔フロアSW間の4本のトランクリンク）は、カットオーバーの前段階（Mission3: 新環境事前構築）で物理的に配線・設定を投入しておく。
- ただし、フロアSW向けの4本のトランク（new-core1/2 ⇔ old-sw1/old-sw2）と、各VLANのSVIは、カットオーバー本番（Mission4以降）まで**shutdown状態で待機**させる。
- この設計により、以下を両立する。
  - 新環境の単体試験（HSRPフェイルオーバー等）を、老朽LANのユーザートラフィックに一切影響を与えずに、フロアSWと切り離した状態で先行実施できる。
  - カットオーバー当日は、`no shutdown`によるポート起動という単純な操作のみでトランクを開通させられる（配線・VLANトランク設定など時間のかかる作業を、断時間の外に追い出せる）。
- トランクを`no shutdown`で開通させた瞬間（MOP P1）から、旧環境と新環境が同一フロアSW配下で並行稼働する状態になる。この期間のL2ループ対策が§4のSTPルート設計である。
- 並行稼働期間中は、VLANごとに「GWの実体がold-core側にあるか、new-core側にあるか」が異なる状態が生じる（例: VLAN10のみカットオーバー済みでVLAN30は未了、等）。カットオーバー未了のVLANはold-coreのSVIがGWのまま稼働を継続するため、当該VLANの通信には影響がない。

## 7. 監視を移行判定に使う設計
- テーマ28で構築した監視基盤（mon-core/nms、SNMPv3ポーリング、v2cトラップ、cap-swのSPAN）を、移行の判定装置としてそのまま活用する。
- **断時間の定量測定**: nmsから対象VLANのGW（`<VLAN>.1`）へ`ping -i 0.2`で連続ping中の損失数を数え、損失数×0.2秒でカットオーバーの実断時間を算出する。これにより、要件定義書§3の「60秒以内」を定量的に判定できる。
- **トラップによるイベント証跡**: old-core側SVIのshutdown、new-core側インターフェースのno shutdown、トランクのlink状態変化などはlinkDown/linkUpトラップとしてnmsへ通知される。これをカットオーバーMOPのエビデンスとして保存する。
- **HSRP状態確認**: `show standby brief`により、Active/Standbyが設計どおり（new-core1=Active、new-core2=Standby）になっているかを判定基準に組み込む。
- **監視対象の追加**: new-core1（10.28.90.2）・new-core2（10.28.90.3）・new-rt（10.29.1.1）をnmsのSNMPv3ポーリング対象に追加する。ポーリング対象・OID・v3認証パラメータの詳細はパラメータシート§を参照。
- 移行完了後（Mission6）は、mon-coreのdefault routeをold-core向け（10.28.0.1）からnew-core1向け（10.29.0.2）へ切り替え、老朽機器への依存を監視経路からも完全に排除する。
