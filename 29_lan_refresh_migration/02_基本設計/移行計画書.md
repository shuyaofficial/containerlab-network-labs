# 移行計画書: 社内LAN更改・移行・移設

本書は、要件定義書・基本設計書で定めた更改内容を、どのような移行方式・フェーズ・体制で実施するかを定める計画書である。

## 1. 移行方式の比較
コアSW・WANルータの更改にあたり、以下3方式を比較検討する。

| 方式 | 概要 | 断時間 | 切り戻し容易性 | 作業量 | リスク |
|---|---|---|---|---|---|
| 一括切替 | 旧環境を全停止し、新環境へ一度に全VLANのGW・経路を切り替える | 全VLANが同時に断（切替作業全体の時間＝断時間となり長時間化しやすい） | 低い（切り戻しも全VLAN同時となり、失敗時の影響範囲が最大） | 少ない（作業自体は1回で完結） | 高い（1回の作業ミスが全社影響になる。障害の切り分けも困難） |
| 段階移行 | VLAN単位で順次GWを切り替える。VLANごとにカットオーバー・判定・（必要なら）切り戻しを実施する | VLANごとに短時間（GW切替の瞬間のみ、60秒以内を目標） | 高い（該当VLANのみを個別に切り戻せる。他VLANへの影響がない） | 多い（VLAN数分のカットオーバー・判定作業が発生） | 中程度（1VLANあたりの影響範囲は小さいが、並行稼働期間中の旧新混在状態の管理が必要） |
| 並行稼働（恒久） | 新旧環境を恒久的に並行稼働させ、新規VLANのみ新環境で運用する | 既存VLANへの影響なし（切替自体を行わない） | 該当なし（切替を行わないため） | 少ない（新環境の追加のみ） | 中程度（老朽機器のSPOFがいつまでも解消されない。当初の更改目的を達成できない） |

## 2. 採用方式と根拠
**VLAN単位の段階移行**を採用する。

- 一括切替は作業量こそ少ないが、要件定義書§3で定めた「ユーザーVLANあたり60秒以内」という許容断時間を満たすことが困難であり、かつ失敗時の影響が全VLANに及ぶためリスクが大きい。
- 並行稼働（恒久）は、そもそも本テーマの更改要件（SPOF解消・旧IOS更改・老朽機器の撤去）を満たせず、要件定義書§2の更改要件と矛盾する。
- 段階移行は、VLANごとに「カットオーバー→判定→（必要なら切り戻し）」という単位で作業を区切れるため、要件定義書§4の切り戻し要件（断時間超過・判定NG・HSRP/STP異常）に対して、VLAN単位で機動的に対応できる。作業量は増えるが、1回あたりの作業単位が小さいため、切り戻しの判断・実施が容易であり、監視基盤による断時間測定・エビデンス取得もVLANごとに明確に対応づけられる。

以上より、本テーマでは段階移行を採用し、VLAN10→VLAN20→VLAN30の順にGWを移行したうえで、WAN切替・移設・撤去を行う計画とする。

## 3. 移行フェーズ計画
| フェーズ | 名称 | 主な作業 | 断時間の発生有無 |
|---|---|---|---|
| P0 | 移行前確認 | 現行環境の稼働確認、移行前ベースライン（疎通マトリクス・経路・ARP）の再確認、新環境の事前構築・単体試験の合格確認、監視（連続ping・トラップ受信）の起動 | なし |
| P1 | 新旧トランク開通 | new-core1/2⇔old-sw1/2間の4トランクを`no shutdown`、STP再収束の確認 | なし（ユーザーVLANのGWは未移行のため） |
| P2 | VLAN10 GW移行 | old-core VLAN10 SVI shutdown、new-core1/2 VLAN10 SVI no shutdown＋HSRP preempt、断時間測定 | あり（VLAN10のみ、目標60秒以内） |
| P3 | カットオーバー判定（VLAN10） | 疎通マトリクス・HSRP状態・トラップ受信の判定 | なし |
| P4 | VLAN20 GW移行 | P2と同様の手順をVLAN20に対して実施（切り戻し訓練を含む） | あり（VLAN20のみ） |
| P5 | VLAN30 GW移行 | P2と同様の手順をVLAN30に対して実施 | あり（VLAN30のみ） |
| P6 | WAN切替 | old-coreのdefault route依存解消の確認、VLAN90（機器管理）GW移行、203.0.113.2への到達性確認 | あり（VLAN90のみ、機器管理トラフィックが対象） |
| P7 | pc-b移設・VLAN20整理 | pc-bをold-sw1からold-sw2へ移設し、VLAN20をold-sw2のみに整理 | あり（pc-b単体、短時間） |
| P8 | 旧機器撤去・監視切替・完了判定 | old-core/old-rtのshutdown・docker stop撤去、mon-core defaultのnew-core1向け切替、全点疎通マトリクスによる完了判定 | なし（旧機器は既に無負荷状態） |

各フェーズの詳細な投入コマンド・確認コマンド・切り戻し判断基準は、カットオーバーMOPを参照する。P6でVLAN90（機器管理）のGWを移行するのは、VLAN90がユーザーVLANではなく（要件定義書§3の60秒要件の直接対象ではない）、かつold-core撤去（P8）に先立ってold-sw1/old-sw2の管理疎通（`ip default-gateway`）を新環境側へ切り替えておく必要があるためである。

## 4. 体制と役割
実務における移行作業は、通常以下3つの役割に分かれて実施する。

| 役割 | 責務 |
|---|---|
| 作業者 | MOPに従いコマンドを投入する |
| 確認者 | 作業者の投入内容・実行結果をダブルチェックし、確認コマンドの結果を判定する |
| 判定者 | 断時間測定結果・疎通マトリクス・HSRP/STP状態を踏まえ、次フェーズへ進むか切り戻すかを最終判断する |

本ラボでは、学習者1名がこの3役を兼務する（**1人3役**）。ただし、役割を意識的に切り替えながら作業することで、実務での相互チェック・承認プロセスの重要性を体感する。具体的には、コマンド投入（作業者視点）と、その直後の確認コマンド実行・結果の読み合わせ（確認者視点）、断時間測定結果を要件定義書の基準と照合したうえでの続行／切り戻し判断（判定者視点）を、MOPの各フェーズで明示的に区切って実施する。

## 5. リスク一覧と対策
| No | リスク | 発生し得るフェーズ | 対策 |
|---|---|---|---|
| R1 | STP再収束が想定より遅延し、トランク開通時に瞬断・ループが発生する | P1 | `show spanning-tree vlan <VLAN>`で収束状態を確認してから次フェーズへ進む。異常時は切り戻し手順書のP1手順（該当トランクの再shutdown）を実施する |
| R2 | HSRPが誤ってnew-core2側でActiveのままになり、preemptが機能しない | P2/P4/P5/P6 | `show standby brief`でActive/Standbyを確認し、期待状態（new-core1=Active）でなければ切り戻し手順書の該当フェーズ手順を実施する |
| R3 | 切り戻し（旧GWの再有効化）を行っても、端末が新GWのMACアドレスをARPキャッシュに保持しており通信が復旧しない | P2/P4/P5/P6の切り戻し時 | 切り戻し手順書の「戻し後確認」でGratuitous ARPの発生・端末側ARPキャッシュの更新を確認項目に含める |
| R4 | 移行作業中に監視経路自体（mon-core⇔監視対象間）が切れ、断時間測定・トラップ受信ができなくなる | 全フェーズ | P0で監視経路の冗長性（mon-coreの並行route）を確認してから着手する。監視断を検知した場合は測定を中断し、原因切り分けを優先する |
| R5 | OSPFの隣接形成・経路収束が想定より遅く、new-rt経由の経路が引けない状態でP6（WAN切替）を実施してしまう | P1〜P6 | P1完了後にOSPF隣接状態（`show ip ospf neighbor`）・経路（`show ip route ospf`）を確認するチェックをMOPに組み込む |
| R6 | 作業者の操作ミス（対象インターフェース・VLAN番号の誤り等）によりコマンドを誤投入する | 全フェーズ | 各フェーズの「確認」ステップを必ず経てから次工程へ進む運用とし、単独のコマンド投入だけでフェーズを完了させない |

## 6. スケジュール
本ラボにおける移行作業の目安スケジュール（作業開始をT+0とする）を以下に示す。実際の所要時間は、各フェーズの確認・判定に要する時間によって前後する。

| 時刻目安 | フェーズ | 内容 |
|---|---|---|
| T+0 | P0 | 移行前確認・監視起動 |
| T+10分 | P1 | 新旧トランク開通・STP収束確認 |
| T+20分 | P2 | VLAN10 GW移行・断時間測定 |
| T+25分 | P3 | カットオーバー判定（VLAN10） |
| T+35分 | P4 | VLAN20 GW移行（切り戻し訓練を含む） |
| T+45分 | P5 | VLAN30 GW移行 |
| T+55分 | P6 | WAN切替・VLAN90 GW移行 |
| T+70分 | P7 | pc-b移設・VLAN20整理 |
| T+85分 | P8 | 旧機器撤去・監視切替・完了判定 |
| T+100分 | ― | 完了 |

各フェーズ内のGW切替そのもの（HSRP preemptの反映）に要する時間は数秒〜数十秒であり、上記の時間幅の大半は確認・判定・記録に充てられる。
