# Theme 29 構築手順書

対象トポロジ: `refresh.clab.yml`（ラボ名 `lan-refresh-lab`、mgmt `172.29.29.0/24`、コンテナ名prefix `clab-lan-refresh-lab-`）

## 1. リソース確認（デプロイ前）

```bash
free -h
```

- IOLの実メモリ使用量は1台あたり約170〜220MiB（2026-07-16のテーマ28検証実測）です。本テーマはIOL 9台構成（old-rt / old-core / old-sw1 / old-sw2 / mon-core / cap-sw / new-core1 / new-core2 / new-rt）のため、9台同時稼働時の実使用量はおよそ2GB程度になります。OrbStack VMのメモリが15GBであれば十分な余裕があります。
- 任意の省メモリ運用として、new-core1・new-core2・new-rtの3台はMission 3で本格的に使うまで停止しておいても構いません。
  ```bash
  sudo docker stop clab-lan-refresh-lab-new-core1 clab-lan-refresh-lab-new-core2 clab-lan-refresh-lab-new-rt
  ```
  Mission 3で使う際は`docker start`で再開します。`docker stop`/`docker start`はコンテナを削除しないため、`write memory`済みの設定はnvramから復元されます（未保存のrunning-configはIOS再起動で失われるため、停止前に必要な設定は保存してください）。
  ```bash
  sudo docker start clab-lan-refresh-lab-new-core1 clab-lan-refresh-lab-new-core2 clab-lan-refresh-lab-new-rt
  ```

## 2. デプロイ

```bash
ssh clab@orb
cd /Users/shuya/Documents/claude/Mac仮想環境構築/29_lan_refresh_migration/04_構築
./deploy.sh deploy
```

- containerlabがトポロジをデプロイし、IOL 9台（old-rt / old-core / old-sw1 / old-sw2 / mon-core / cap-sw / new-core1 / new-core2 / new-rt）＋ Linux 5台（nms / cap / pc-a / pc-b / srv-file）の計14ノードが起動します。
- `entrypoint_attach.sh`がIOLブート完了後（+約60秒）に`iouyap`を自動起動します。デプロイ直後は配線がまだ完成していないため、**60秒程度待ってから**次の確認に進んでください。
- 自動起動が働かない場合のみ、手動リカバリとして`./deploy.sh iouyap`を実行します。

### IOLライセンスの扱い（deploy.shが自動処理）

- 本テーマのL2系イメージ（`L2-15.2` / `L2-advipservices-2017`）と新旧IOS（`15.6.3M3a` / `15.7.3M2`）は、ライセンスが`/iol/.iourc`ファイルではなく**イメージ側`/entrypoint.sh`内**に埋め込まれています。本ラボは`entrypoint_attach.sh`をbindで上書きするため、そのままではライセンスが渡らずIOLが`License Error`で起動しません（テーマ27 ISS-08の亜種）。
- 対策として`deploy.sh deploy`は、デプロイ前にVM内のIOLイメージからライセンス値を抽出し、VMローカルの`/opt/clab/.iourc`（mode 600）へ書き出します。`refresh.clab.yml`が全IOLノードへ`/opt/clab/.iourc:/iol/.iourc:ro`をbindし、`entrypoint_attach.sh`の「有効値なら尊重」ロジックでそのまま使われます。
- **必ず`./deploy.sh deploy`経由でデプロイしてください**（`containerlab deploy`を直接実行すると、初回は`/opt/clab/.iourc`が無くbindが失敗します）。
- ライセンス値はVM内（Dockerイメージと`/opt/clab/.iourc`）にのみ存在します。**gitリポジトリ・Mac側へコピーしないこと**。

|---|---|---|---|
| nms | 10.28.100.10/24 | defaultはeth0(mgmt)側に残置。個別経路`10.28.0.0/16 via 10.28.100.1`・`10.29.0.0/16 via 10.28.100.1`をeth1へ追加 | 監視サーバ。`apk`等の外部到達をmgmt経由で維持しつつ、現行(10.28.0.0/16)・新環境(10.29.0.0/16)の両方をeth1(mon-core)経由で監視する |

## 5. nmsへのnet-snmp導入

`wbitt/network-multitool`イメージには`snmpwalk`/`snmpget`/`snmptranslate`/`iperf3`が同梱済みです。**`snmptrapd`のみ未同梱**のため、**Mission 1の前（デプロイ直後）**に以下でインストールします。本テーマでは移行作業中のlinkDownトラップ受信がカットオーバーのエビデンスになるため、テーマ28（Mission 6でよい）と異なり最初から必要です。

```bash
sudo docker exec -it clab-lan-refresh-lab-nms bash
apk add --no-cache net-snmp
```

- multitoolイメージはAlpineベースのため`apk`を使用します（`snmptrapd`は`net-snmp`パッケージに含まれ、`/usr/sbin/snmptrapd`に入ります）。
- nmsのdefaultルートはmgmt側（eth0）に残してあるため、ラボ稼働中でも`apk`のインターネット到達は生きています（clab.ymlの`exec`はラボ向けに`10.28.0.0/16 via 10.28.100.1`・`10.29.0.0/16 via 10.28.100.1`の個別経路のみ追加する設計。defaultをeth1へ向けるとDNS/apkが不通になるので変更しないこと）。
- インターネットに出られない環境の場合は、Mac側で取得済みのnet-snmp入りイメージ・パッケージを`docker cp`でコンテナに持ち込む等の代替手段を検討してください。

## 6. snmptrapd起動手順（Mission 1〜。移行作業中は常時稼働させる）

nms上で以下を実行し、トラップ受信の受け皿を用意します。カットオーバーMOPのP0（移行前確認）はこの受信画面が稼働していることを前提とします。

```bash
cat > /tmp/snmptrapd.conf <<'EOF'
authCommunity log public
disableAuthorization yes
EOF

snmptrapd -f -Lo -c /tmp/snmptrapd.conf
```

- `-f`: フォアグラウンド起動（Ctrl-Cで停止するまで画面にログを流し続けます）。
- `-Lo`: 標準出力へログ出力。
- `authCommunity log public`: community `public`のトラップを受理してログに記録します。
- `disableAuthorization yes`: v1/v2cトラップの認可チェックを簡略化します（ラボ用途）。

old-coreのVLAN SVI shutdown等でlinkDownトラップがこの画面に着信します。

## 7. Wiresharkキャプチャ（2系統）

SPAN先（cap）での観測は、用途に応じて以下の2系統があります。

### 7.1 capコンテナ内でtcpdump

```bash
sudo docker exec -it clab-lan-refresh-lab-cap sh
tcpdump -nni eth1 -w /tmp/span.pcap
```

### 7.2 Mac側Wiresharkへ直結

```bash
ssh clab@orb "sudo nsenter -t \$(docker inspect -f '{{.State.Pid}}' clab-lan-refresh-lab-cap) -n tcpdump -U -nni eth1 -w -" | /Applications/Wireshark.app/Contents/MacOS/Wireshark -k -i -
```

- OrbStack VM内のcapコンテナのネットワーク名前空間へ`nsenter`で入り、tcpdumpの標準出力をパイプでMac側のWiresharkに渡します。
- キャプチャ対象はcap-swのSPAN destinationポート（cap:eth1）なので、old-core⇔mon-core間を通過する監視系トラフィック（トラップ・SNMPポーリング）がここに写ります。カットオーバー時のlinkDownトラップの実着信確認にも使えます。

## 8. pc-b移設コマンド（Mission 6付録）

pc-b内でeth1からeth2へIPを付け替える作業です（old-sw1側ポートshut・old-sw2側ポートのaccess VLAN20化はIOS側の作業のため、README_Lab_Challenge.mdのMission 6・カットオーバーMOP側を参照してください）。

```bash
sudo docker exec -it clab-lan-refresh-lab-pc-b bash
ip addr flush dev eth1 && ip link set eth1 down
ip addr add 10.28.20.10/24 dev eth2 && ip route replace default via 10.28.20.1 dev eth2
```

## 9. destroy時の注意

```bash
./deploy.sh destroy
```

- 内部で`--cleanup`付きの`containerlab destroy`を実行するため、**nvram（保存済みIOSコンフィグ）ごと削除されます**。
- 投入したIOS設定を残したままラボを一旦壊したい場合は、`--cleanup`を付けずに直接実行してください。
  ```bash
  sudo containerlab destroy -t refresh.clab.yml
  ```
