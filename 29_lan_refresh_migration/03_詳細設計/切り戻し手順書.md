# 切り戻し手順書: 社内LAN更改・移行・移設

## 1. 発動条件
以下のいずれかに該当する場合、該当フェーズの切り戻し手順を実施する（要件定義書§4より再掲）。

1. 断時間が60秒を超過した場合
2. カットオーバー判定試験（疎通マトリクス）にNG項目がある場合
3. HSRP・STPが期待状態にならない場合（例: Active/Standbyの逆転、STPが期待したルートブリッジ／ブロッキングポートにならない等）

## 2. 原則
- **切り戻し後は、原因究明が完了するまで同一フェーズの再カットオーバーを行わない。** 原因を特定しないまま再度カットオーバーを試みると、同じ事象を繰り返すだけでなく、切り戻し自体の正当性の検証が不十分なまま次のフェーズへ進んでしまうリスクがある。
- 切り戻しは、カットオーバーMOPで投入した設定を**逆順に戻す**ことを基本とする。GW移行フェーズ（P2/P4/P5/P6/P7）の切り戻しは、いずれも「new-core側のSVI（またはポート）をshutdown → old-core側（または元の収容先）をno shutdown」という共通パターンを取る。
- 切り戻し完了後は、切り戻し前（カットオーバー実施前）のベースラインと同等の状態に復旧していることを、疎通試験で確認してから完了とする。
- 本書のP4（VLAN20 GW移行）の手順は、Mission5「切り戻し訓練→再カットオーバー」で実際に使用する。Mission5では、意図的にP4のカットオーバーを実施した後、本書のP4手順で切り戻しを行い、正常に復旧することを確認したうえで、カットオーバーMOP P4を再実行して成功させる。

## 3. フェーズ対応表

### P1: 新旧トランク開通の切り戻し
| 項目 | 内容 |
|---|---|
| 戻しコマンド | new-core1: `interface Ethernet0/2` → `shutdown`、`interface Ethernet0/3` → `shutdown`<br>new-core2: `interface Ethernet0/2` → `shutdown`、`interface Ethernet0/3` → `shutdown`<br>old-sw1: `interface Ethernet1/0`・`interface Ethernet1/1` → それぞれ`shutdown`<br>old-sw2: `interface Ethernet0/3`・`interface Ethernet1/0` → それぞれ`shutdown` |
| 戻し後確認 | 各機器で`show interfaces status`→該当ポートがdisabledになっている。old-core・old-sw1・old-sw2で`show spanning-tree vlan 10`（20/30/90も同様）→new-core側を含まない、移行前と同じ単純なトポロジに収束している |
| 戻し完了判定 | P0-4で開始した監視pingのロスがゼロに戻っていること（old-core配下の既存VLANトラフィックへの影響がないこと）を確認して完了とする |

### P2: VLAN10 GW移行の切り戻し
| 項目 | 内容 |
|---|---|
| 戻しコマンド | new-core1: `interface Vlan10` → `shutdown`<br>new-core2: `interface Vlan10` → `shutdown`<br>old-core: `interface Vlan10` → `no shutdown` |
| 戻し後確認 | old-coreで`show ip interface brief \| include Vlan10`→up/up。new-core1・new-core2で`show standby vlan 10 brief`→SVIがshutdownのためstandbyグループも非活性であることを確認。pc-aから`ping -c 4 10.28.10.1`→4/4成功 |
| 戻し完了判定 | pc-aからのGW疎通・WAN疎通（`ping -c 4 203.0.113.1`、旧経路）が移行前ベースラインと同等に復旧していることを確認して完了とする |

### P3: カットオーバー判定（VLAN10）の切り戻し
| 項目 | 内容 |
|---|---|
| 戻しコマンド | なし（P3は判定のみのフェーズであり、設定投入を行わない） |
| 戻し後確認 | P3の判定でNGと判断した場合は、本書のP2手順を実施する |
| 戻し完了判定 | P2の戻し完了判定に準じる |

### P7: pc-b移設・VLAN20整理の切り戻し
| 項目 | 内容 |
|---|---|
| 戻しコマンド | pc-b: `ip link set eth2 down`、`ip link set eth1 up`（元のIP設定10.28.20.10/24・GW 10.28.20.1はeth1側に残置済みのため再設定は不要）<br>old-sw2: `interface Ethernet1/1` → `shutdown`<br>old-sw1: `interface Ethernet0/3` → `no shutdown` |

### P8: 旧機器撤去・監視切替の切り戻し
| 項目 | 内容 |
|---|---|
| 戻しコマンド | （containerlabホストで）`docker start clab-lan-refresh-lab-old-core`、`docker start clab-lan-refresh-lab-old-rt`<br>mon-core: `no ip route 0.0.0.0 0.0.0.0 10.29.0.2` → `ip route 0.0.0.0 0.0.0.0 10.28.0.1`<br>nmsのポーリング対象にold-rt（10.28.1.1）・old-core（10.28.90.1）を再度追加する |
| 戻し後確認 | `docker ps`でold-core・old-rtのコンテナが起動していることを確認する。ただしold-coreの各VLAN SVIはP2/P4/P5/P6で既に役目を終えているため、**再度shutdown状態のまま維持し、GWとして機能させない**（VLAN GWの実体は既にnew-core側にあるため、old-core復旧はコンテナ・監視対象としての復旧のみを意味する）。mon-coreで`show ip route static`→defaultが10.28.0.1経由に戻っている。nmsから9台へのSNMPv3ポーリングが成功する |
| 戻し完了判定 | 全点疎通マトリクスがP7完了時点（GW移行は完了済み、旧機器はまだ稼働中）の状態と同等になっていることを確認して完了とする |
