# 図面作成・プレビュー表示に関する検証ログ

本ドキュメントは、基本設計フェーズにおける「ネットワーク構成図」の最適な表示方法を模索した際の、検証とトラブルシューティングの記録です。

## 1. 検証の背景
VSCode上でネットワーク構成図（物理構成図・論理構成図）を作成し、左右に分割して同時にプレビューしながら設計・構築を進めたいという要件が発生した。

## 2. 試行したアプローチと結果

### アプローチ①：Markdown標準のMermaidブロック（`.md`）
- **結果**: ❌（環境依存のレンダリングエラー）
- **詳細**: `.md` ファイル内に ` ```mermaid ` ブロックを記述。VSCode標準機能の「プレビューのロック」を使えば複数タブ表示が可能になるはずだったが、インストールされている複数のMarkdown/Mermaid拡張機能が干渉し、図面が描画されず真っ黒になる事象が発生。

### アプローチ②：専用拡張子（`.mermaid`）＋ Mermaid Preview拡張機能
- **結果**: 🔺（表示は最も綺麗だが1画面制限あり）
- **詳細**: 拡張子を `.mermaid` とし、`Mermaid Preview` 拡張機能を使用。文字色（`color:#000;`）を調整することでダークテーマでも視認性が良く、最もスタイリッシュな図面（添付のスクリーンショット参照）が出力された。
- **課題**: 拡張機能の仕様上、プレビュー画面を1つしか保持できず、物理・論理の2画面同時表示ができない。

### アプローチ③：アスキーアートによるテキスト描画（`.md`）
- **結果**: 💡（同時表示用としての代替案）
- **詳細**: 拡張機能に依存しないよう、プレーンテキストの罫線で物理・論理構成図を作成。視覚的なリッチさには欠けるが、VSCode環境に依存せず確実に2画面表示ができる。

### アプローチ④：Python `diagrams` ライブラリによる画像生成（PNG）
- **結果**: 🟢（安定表示可能だが、見た目の好みが分かれる）
- **詳細**: 
  1. バックグラウンドで `brew install graphviz` を実行してシステム要件を満たした。
  2. 仮想環境（`venv`）を構築し、`pip install diagrams` でライブラリを導入。
  3. PythonスクリプトからPNG画像を生成。画像ファイルのため、VSCode上で何枚でも同時に開くことが可能となった。
- **評価**: 安定性は最も高いが、出力されるアイコンやデザインのテイストが、Mermaidのスッキリしたデザインに比べて好みに合わない場合がある。

## 3. 結論・今後の対応方針
現時点でのユーザー（設計者）の個人的な好みとしては、**「アプローチ②（Mermaid Preview）の見た目が一番見やすい」**という結論に至った。
今後はこの検証結果を踏まえ、状況に応じて「美しさ（Mermaid）」を取るか、「複数画面での並行作業（画像やテキスト）」を取るか、用途に合わせて最適な手法を選択・模索していく方針とする。

## 4. VSCode操作ナレッジ（備忘録）
本検証の過程で確認・習得したVSCodeの便利な操作方法を以下に記録する。

### ① 画面の分割（スプリット）方法
設計書を見ながらコンフィグを書く際などに必須の機能。
- **ドラッグ＆ドロップ**: タブをクリックしたまま画面の右端まで引っ張ってドロップする。
- **メニュー操作**: タブを右クリックして「右に分割（Split Right）」を選択、または画面右上の「見開き本アイコン」をクリックする。

### ② Mermaidプレビューの開き方（`.mermaid`ファイル）
- **コマンドパレット**: `Cmd + Shift + P` でコマンドパレットを開き、`mermaid` と入力して `Mermaid: Preview` を選択する。
- **右クリック**: コードが書かれているエディタ画面上で右クリックし、`Preview Diagram` を選択する。

### ③ プレビュー画面の複数表示（`.md`ファイル）
VSCodeの標準Markdownプレビューは「1つのタブを使い回す（上書きされる）」仕様だが、以下の方法で固定（ロック）できる。
- **ダブルクリック**: プレビュータブの名前（*斜体* になっているもの）をダブルクリックし、通常の真っ直ぐな文字にする。
- **ロック機能**: プレビュー画面右上の `...` から「プレビューをロックする（Lock Preview）」を選択する。
※これにより、新しい `.md` ファイルを開いた際に別タブとしてプレビューが追加される。

## 5. 今後の環境構築におけるベストプラクティス
本プロジェクトを通じて得られた、**「構成図のデザインからエミュレータ画面の整形まで」**の一連のベストプラクティスを定義する。
今後の新規ネットワーク環境構築時にも、このフローを適用することで、設計と検証のギャップを無くし、美しく分かりやすい環境を構築できる。

### 【推奨ワークフロー】
1. **基本設計図面の作成（Mermaid）**
   - まず `.mermaid` ファイルで論理・物理構成図を作成する。
   - ノードの階層構造（上下・左右）をここで美しく整理し、設計の全体像を固める。
2. **Containerlabトポロジの作成（`.clab.yml`）**
   - Mermaidで定義した構成に基づいて機器の接続コードを書く。
3. **エミュレータ画面（NeXt UI）の配置整形**
   - Containerlab展開後、自動生成されるトポロジ画面はノードが乱雑に配置される（クモの巣状）。
   - これを放置せず、**Mermaidの図面構成と全く同じ配置になるよう、座標データを整形する**。
   - **やり方**:
     - マウスでドラッグ＆ドロップして画面右上の `💾（Save Layout）` を押す。
     - または、生成される `*.clab.yml.annotations.json` ファイルの `x` `y` 座標を直接書き換える。
   - **効果**: 設計書（Mermaid）と検証環境（Containerlab）の見た目が完全に一致するため、トラブルシューティング時の思考の切り替えストレスがなくなり、直感的なオペレーションが可能となる。
