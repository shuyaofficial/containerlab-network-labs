## 2. 直面した問題と「できなかった原因」（根本原因）

### 【主因】L2スイッチに管理IPが付かず SSH 不可
- **原因**: トポロジで hq-core1/hq-core2 に **`type: L2` を指定していなかった**。
- **メカニズム**: containerlab の day0 設定テンプレート `nodes/iol/iol.cfg.tmpl` は、管理ポート `Ethernet0/0` に対し
  ```
  interface Ethernet0/0
  {{ if .IsL2Node }}
   no switchport
  {{ end }}
   vrf forwarding clab-mgmt
   ip address <mgmt-ip> ...
  ```
  となっており、**`no switchport` は `type: L2`（=`.IsL2Node`）のときだけ**出力される。
  未指定だと L2 イメージの `Ethernet0/0` は **switchport（L2モード）のまま**で、`ip address` / `vrf forwarding` が無効化され、管理IPが付与されず SSH 不能。
- **証跡**: 修正前の生成config `clab-capstone/hq-core1/boot_config.txt` は router 形式（`no switchport` 無し）。SSHは `Connection timed out`。

### 併発バグ① 管理IPのドリフト + 誤指定
- containerlab は管理IPを毎回自動採番するため、deploy/destroy のたびに各機器のIPが変動。
- 旧 `configure_cores.sh` は hq-core2 の宛先に **`172.20.20.8`** を指定していたが、これは実際には **hq-edge1** の管理IP。SSHが通っても **edge1 を誤設定**する状態だった。
- destroy/deploy の短時間反復で `%IP-4-DUPADDR`（アドレス重複）も誘発。

### 併発バグ② SSH host key の残留（再デプロイで弾かれる）
- 旧スクリプトのSSHオプションに `UserKnownHostsFile=/dev/null` が無く、`StrictHostKeyChecking=no` だけでは
  再デプロイで鍵が変わったとき **`REMOTE HOST IDENTIFICATION HAS CHANGED!`** で接続拒否。これが「不安定」の一因。

### 設計ギャップ（疎通ゴールに必要だが未実装だった点）
- 旧 `configure_all.sh` の **br-edge に NAT とデフォルトルートが無く**、支社PCがインターネット到達不可だった。
- 疎通確認用の「インターネット」ターゲットが存在しなかった。

## 3. 実施した解決策

### Step 1 — 完全クリーンアップ
```bash
sudo containerlab destroy -t capstone.clab.yml --cleanup
sudo docker network prune -f
```
→ コンテナ・管理ネットワーク(`clab`)・生成dir(`clab-capstone/`)を一掃。IP重複と host key churn を解消。

### Step 2 — トポロジ修正 `capstone.clab.yml`
- hq-core1 / hq-core2 に **`type: L2`** を追加（本丸の修正）。
- **管理IPを静的固定**（ドリフト根絶）。`mgmt:` ブロック + 各ノード `mgmt-ipv4:`。

| node | mgmt-ipv4 | | node | mgmt-ipv4 |
| :--- | :--- | :--- | :--- | :--- |
| isp | 172.20.20.11 | | hq-core1 | 172.20.20.21 |
| hq-edge1 | 172.20.20.12 | | hq-core2 | 172.20.20.22 |
| hq-edge2 | 172.20.20.13 | | hq-pc-sales | 172.20.20.101 |
| br-edge | 172.20.20.14 | | br-pc | 172.20.20.102 |

```yaml
mgmt:
  network: clab
  ipv4-subnet: 172.20.20.0/24
  ipv6-subnet: 3fff:172:20:20::/64
```

### Step 3 — 再デプロイ + 検証
```bash
sudo containerlab deploy -t capstone.clab.yml
```
→ 生成された `clab-capstone/hq-core1/boot_config.txt` の `Ethernet0/0` に **`no switchport` が入った**ことを確認（修正が効いた証拠）。

### Step 4 — 全6台の管理SSH疎通確認
- 堅牢なSSHオプション（次ステップと共通）で6台すべて（特に core1=.21 / core2=.22）にSSH成功。

### Step 5 — 設定投入（`configure_all.sh` を1本に統合・堅牢化）
- ルーター4台 + コア2台のスクリプトを**統合**。主な堅牢化:
  - **管理IPはコンテナ名から動的解決**（`docker inspect`）→ ドリフト無効化。
  - **SSHオプションに `UserKnownHostsFile=/dev/null` を追加** → host key churn 対策。
  - **br-edge に NAT overload + デフォルトルート追加**:
    `access-list 1 permit 10.2.40.0 0.0.0.255` / `ip nat inside source list 1 interface Ethernet0/1 overload` / `ip route 0.0.0.0 0.0.0.0 200.0.3.254`
  - **ISP に Loopback 8.8.8.8/32 を追加**（疎通確認用の「インターネット」役）。
- 設定本文は `03_詳細設計` のパラメータシート（IP / ルーティング / NAT）に準拠。

### Step 6 — 疎通・正常性確認
→ 結果は `05_試験/試験結果_2026-06-08.md` を参照。すべて GREEN。

#### 使用した堅牢なSSHオプション（再掲）
```
-o StrictHostKeyChecking=no -o UserKnownHostsFile=/dev/null -o ConnectTimeout=10 \
-o KexAlgorithms=+diffie-hellman-group14-sha1 -o HostKeyAlgorithms=+ssh-rsa -o PubkeyAcceptedAlgorithms=+ssh-rsa
```

## 5. 残課題 / 対象外

- **IPsec / GRE VPN（Tunnel0, IKEv2, HQ-Edge1 ↔ BR-Edge）**: パラメータシート03編に定義があるが、**ユーザー本人が後で実施**するため本作業では未着手。
  - HQエッジの NAT 除外ACL（`access-list 100 deny ip 10.0.0.0 0.255.255.255 10.2.40.0 0.0.0.255`）は **VPN前提のため温存**。VPN追加時にそのまま機能する（インターネット通信には無害）。
  - 注意: IOL イメージは crypto(IPsec) 非対応のことが多く、その場合は GRE-only へのフォールバックを検討。
