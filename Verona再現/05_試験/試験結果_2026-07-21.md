# 試験結果: Verona再現（実機検証 2026-07-21）

商用SASE「Verona」の機能を、arm64 OSSで再現したラボを OrbStack VM `clab`（aarch64、10コア/19GiB）で実機デプロイし、[試験計画書](試験計画書.md)のT-V1〜T-V7を検証した。**コア4機能（T-V1〜V4）・発展3機能（T-V5〜V7）とも合格**（計画の合格基準「コア全て＋発展1つ以上」を上回る）。

## 環境
- ホスト: OrbStack VM `clab`（aarch64）、docker 29.1.3
- ネットワーク: `vsase-cloud`(172.60.10.0/24, bridge `vsase-br0`)／`vsase-branch`(172.60.20.0/24)／`vsase-dark`(172.60.30.0/24)
- コンテナ: ziti / protected-app / apptun / clienttun / squid / blocky / safe-site / branch-client / suricata（コア9）＋ step-ca / keycloak / headscale（発展3）
- 使用イメージ（全arm64ネイティブ実測）: openziti/ziti-cli, jasonish/suricata, ubuntu/squid, ghcr.io/0xerr0r/blocky v0.33.0, smallstep/step-ca, quay.io/keycloak/keycloak 26.7.0, headscale/headscale v0.29.2, nginx:alpine, wbitt/network-multitool

## 詳細エビデンス

### T-V1 ZTNA・ダークサービス（ポート不可視）
```
[1] オーバーレイ経由 clienttun→localhost:8080→protected-app : HTTP 200
[2] 直接 clienttun→protected-app:80（到達不能が正常）        : HTTP 000
```
protected-app は `vsase-dark` のみに存在し公開ポートを持たない。認証済みOpenZitiオーバーレイ経由でのみ到達可能で、直接アクセスは到達不能（Veronaのダイナミックポートコントロール＝「外部からポートが見えない設計」の再現）。

### T-V2 SWG・URLフィルタ（Squid）
```
[1] 許可ドメイン safe.verona-lab.test  経由 squid : HTTP 200
[2] ブロックドメイン malware.verona-lab.test 経由 squid : HTTP 403
```

### T-V3 SWG・DNSセキュリティ（Blocky）
```
[3] 許可ドメイン safe.verona-lab.test    → 172.60.10.13（正引き成功）
[4] ブロックドメイン malware.verona-lab.test → 0.0.0.0（denylistでシンクホール）
```
T-V2（URL層）とT-V3（DNS層）で、Verona資料p40-42の「URLフィルタリング＋DNSセキュリティの二重防御」を再現。

### T-V4 FWaaS・IPS/IDS（Suricata）
```
signature: "VERONA-IDS recon SYN scan toward SASE cloud chokepoint" (sid:2000001)
src 172.60.10.21 → dest 172.60.10.11 (vsase-br0上でhost mode監視)
```
※IDSモード（検知のみ）。インライン遮断（IPS）・アンチウイルス・アンチスパムは対象外（[OSS対応表](../02_基本設計/網屋Verona_OSS対応表.md)）。

### T-V5 デバイスポスチャー（step-ca + 判定スクリプト）
```
[A 拒否] 証明書なし → NG: クライアント証明書が見つかりません → 判定: NG（接続拒否）
[step-ca] 実クライアント証明書 verona-client を発行（/tmp/client.crt）
[B 許可] step-ca発行証明書 + EDR2.5 → [1]証明書OK [2]OS OK [3]EDR 2.5>=2.0 → 判定: OK（接続許可）
```
osqueryがarm64非対応のため簡易チェック（証明書有無・OS種別・擬似EDRバージョン）に簡略化したモックだが、**証明書は実際にstep-caが発行**しており、要件未達端末を拒否するポスチャーの判定ロジックを実証。

### T-V6 IdP連携（Keycloak）
```
kcadmで verona レルム＋OIDCクライアント verona-sase を作成
OIDC discovery: HTTP 200
  issuer: http://localhost:8081/realms/verona
  token_endpoint: .../protocol/openid-connect/token
  grant_types: authorization_code / client_credentials / implicit
```
Keycloak IdPがOIDCを提供（SSO・2要素の基盤）。OpenZiti external-jwt-signer による証明書認証×IdaaS認証の完全連携は次段階（keycloak/README.md）。

### T-V7 拠点間トンネル制御（Headscale＝SD-WAN相当）
```
Headscale v0.29.2 起動（埋め込みDERPリージョン作成）、health=200
拠点ユーザー branch-site 作成、preauthキー発行: hskey-auth-O_NqE...
```
拠点がWireGuardメッシュに参加するための制御プレーン（Verona SASEクラウドの拠点管理に相当）が動作。tailscaleクライアントの実データプレーン疎通は次段階。

## デプロイ中に判明・修正した事項
1. Headscale: `IMG serve` とすべき所を `IMG headscale serve` としており「unknown command headscale」→ エントリポイントが既に `headscale` のため引数を `serve` のみに修正
2. Headscale config: `dns.override_local_dns` の既定 true で `dns.nameservers.global` 必須 → `override_local_dns: false` ＋ `nameservers.global` を追加
3. Headscale config: v0.29 は DERPMap 非空が必須 → 埋め込みDERPサーバ（`derp.server.enabled: true`）を追加しラボ内完結
（コア4機能 OpenZiti/Squid/Blocky/Suricata は初回デプロイで全て成功、修正不要）

## 再現度の位置づけ（過大評価しない）
- 本ラボはVeronaの5大機能（ZTNA/SWG/FWaaS/CASB相当/SD-WAN）を**個別OSSで組み上げて実機実証**したもの。フルマネージド運用・Cloud Console一元管理・4800+アプリのアプリケーションコントロール(CASB)・91カテゴリの商用URLフィード等は対象外または簡略化（[OSS対応表](../02_基本設計/網屋Verona_OSS対応表.md)参照）。
- 発展3機能（ポスチャー/IdP/トンネル）は各コンポーネントの起動・基盤動作までを実証。完全な相互連携（証明書認証×IdaaS SSO、実トンネルのデータプレーン疎通）は次段階として各READMEに明記。
