# 要件定義書: Verona再現

## 1. 背景・目的

網屋の商用フルマネージドSASE製品「Verona」（`verona.pdf` 全67頁、資料バージョン Ver. 13.7.2）は、ZTNA・SWG・FWaaS・CASB・SD-WANをオールインワンで提供する。既存テーマ [36_ztna_openziti](../../36_ztna_openziti/README_Lab_Challenge.md) は、このうちZTNAの核心である「証明書認証によるダイナミックポートコントロール（ダークサービス）」を OpenZiti で実証済みだが、Veronaが謳う残り4機能（SWG・FWaaS・デバイスポスチャー・IdP連携・拠点間トンネル）は未再現である。本テーマは、その未再現ギャップをarm64のOSSで埋め、「Verona＝オールインワンSASEを1つずつOSSで組む」構成を学習可能にすることを目的とする。

## 2. 対象範囲

Verona資料が定義する5大機能（ZTNA / IDaaS連携 / SD-WAN / CASB / SWG+FWaaS）のうち、以下を対象とする。

| Verona機能 | 本ラボでの対応OSS | 区分 |
|---|---|---|
| ZTNA（ダイナミックポートコントロール・ダークサービス） | OpenZiti（36_ztna_openzitiのパターンを統合） | コア |
| SWG（URLフィルタリング） | Squid | コア |
| SWG（DNSセキュリティ） | Blocky | コア |
| FWaaS（IPS/IDS。IDSモードのみ） | Suricata | コア |
| ZTNA（デバイスポスチャー） | step-ca（クライアント証明書）＋簡易ポスチャーチェックスクリプト | 発展 |
| IDaaS連携（証明書×IDaaS認証・SSO） | Keycloak | 発展 |
| SD-WAN（拠点間トンネル） | Headscale + tailscale | 発展 |

コア（[04_構築/deploy.sh](../04_構築/deploy.sh) の `deploy-core` で確実に起動・検証できる）と発展
（config・docを用意し単独サブコマンドで起動できる形にする。連携までの完全な疎通検証は求めない）
の2段階でスコープを分ける。詳細な再現度は
[02_基本設計/網屋Verona_OSS対応表.md](../02_基本設計/網屋Verona_OSS対応表.md) を参照。

### 非目標（対象外）

- **CASB（アプリケーションコントロール、4800以上のアプリ識別DB）**: 対応するOSSの選定・実装コストが高く、本ラボの主眼（ZTNA/SWG/FWaaSの実証）から外れるため対象外とする。
- **アンチウイルス・アンチスパム（FWaaS内訳）**: FWaaSはIPS/IDSのみを対象とし、ファイル検疫・メール検査は対象外とする。
- **Cloud Console（Web管理画面での証明書一括管理・統計可視化）**: 42_ndr_flowのLoki/Grafanaパターンを応用すれば拡張可能だが、本ラボでは新規実装しない。
- **91カテゴリのURL分類データベース・脅威インテリジェンスDB**: OSSでは同等のデータベースを持たないため、単一の共有ブロックリスト（テストドメイン）による概念実証に留める。
- **WAN冗長化・IPoE・デュアルSIM(5G)等のハードウェア/回線機能**: SD-WANのうち拠点間トンネルの概念のみを対象とし、回線・ハードウェア依存機能は対象外とする。
- **実在の危険サイト・実在マルウェアの使用**: SWG/FWaaSの検証はすべて `.test`（IANA予約TLD、RFC 2606）のテスト専用ドメインと、無害な擬似スキャン（nmap SYNスキャン）で行う。

## 3. 機能要件

- branch-client（拠点端末相当）から、OpenZiti overlay 経由でのみ保護対象アプリ（`protected-app`）に到達でき、直接到達（`protected-app:80`）は失敗すること（ダイナミックポートコントロールの再現）。
- clienttun（リモート端末相当）が overlay を通じて `protected-app` に到達できること。
- branch-client が Squid の明示プロキシ（3128）経由でブロックリスト外ドメインへアクセスした場合は許可（200系）され、ブロックリスト内ドメインへアクセスした場合は拒否（403）されること。
- branch-client が Blocky（DNS）へブロックリスト内ドメインを問い合わせた場合は `0.0.0.0`（zeroIp）が返り、ブロックリスト外の固定マッピングドメインは正しいIPが返ること。
- branch-client から SASEクラウド結節点（vsase-cloud）に向けたSYNスキャンが、Suricataのカスタムルール（sid:2000001）で検知され、`eve.json` にalertとして記録されること。
- step-ca が単独で起動し、ルートCA初期化・クライアント証明書発行ができること。発行した証明書の有無に応じて `check_posture.sh` がOK/NGを返すこと。
- Keycloak が単独で起動し、Admin Console にアクセスできること。
- Headscale が単独で起動し、coordinatorとして待受できること。

## 4. 非機能要件

- 実行基盤は OrbStack VM `clab`（arm64）。全コンテナイメージはarm64実測済みのものを使用する（[04_構築/deploy.sh](../04_構築/deploy.sh) 冒頭のイメージ変数を参照）。
- containerlab は使用しない（IOLを含まない純docker構成）。`docker run` を直列に並べる方式とし、composeは使用しない（[36_ztna_openziti](../../36_ztna_openziti/04_構築/deploy.sh)・[42_ndr_flow](../../42_ndr_flow/04_構築/deploy.sh) の作法を踏襲）。
- ネットワーク名・サブネット・host公開ポートは固定値とし、他テーマ（36/42/microseg_nftables等）およびALog再現（Grafana3001/Loki3101/MinIO9001想定）と重複しないこと。
- 全ドキュメントは Markdown をソース・オブ・トゥルースとし、[規約/ドキュメント標準.md](../../規約/ドキュメント標準.md)・[規約/frontmatterスキーマ.md](../../規約/frontmatterスキーマ.md) に準拠する。

## 5. ゾーン設計方針

Verona SASEの3層構成（p6: Verona SASE＝クラウド基盤／Verona Edge＝拠点ルータ／Verona Client＝リモート端末）を、Docker network 3種にマッピングする。

- **vsase-cloud**（SASEクラウド相当）: OpenZitiコントローラ・Squid・Blocky・Suricataの監視点。IDaaS連携（Keycloak）・デバイスポスチャー（step-ca）・拠点間トンネル（Headscale）も同じクラウドゾーンに配置する。
- **vsase-branch**（拠点相当）: branch-client のホームネットワーク。Verona Edge配下の拠点LANに相当。
- **vsase-dark**（保護対象アプリ専用）: `protected-app` のみが所属し、クライアントは一切参加しない。OpenZitiのapptunだけが橋渡しする（36_ztna_openzitiのダークサービスパターンをそのまま踏襲）。

## 6. 制約

- **arm64制約**: OrbStack VM (Ubuntu 24.04 arm64) 上で実行する。すべてのイメージはarm64ネイティブビルドを使用し、x86エミュレーションは発生しない想定（IOLのような約2分起動待ちは無い）。
- **compose不使用**: `sudo docker run` の直列実行方式を厳守する（36/42の作法）。
- **host 53番ポートは使わない**: Blocky DNSのhost公開は5353とし、VM/Macホストの既存DNS設定と衝突させない。
- **本タスクはファイル作成のみ**: 本テーマの構築物（deploy.sh・setup_ziti.sh・各種config）はメインセッションが別途 `ssh clab@orb` 経由で実デプロイ・検証する。本タスク時点でdocker実行・SSH接続は一切行っていない。
- **config schemaの未検証事項**: Blocky（config.yml）・Headscale（config.yaml）・Keycloak（kcadm.sh構文）・step-ca（DOCKER_STEPCA_INIT_*）は、いずれもバージョンによりスキーマ・CLI構文が変わりうる。各READMEに「未検証事項」として明記した。
