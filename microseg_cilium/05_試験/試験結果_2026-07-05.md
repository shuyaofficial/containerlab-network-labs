# テーマ microseg_cilium マイクロセグメンテーション（Cilium/eBPF）試験結果（2026-07-05）

NW-ZT トラック N4。マイクロセグの本質「**同一セグメント（east-west）の通信を default-deny にし、許可した端末・パスだけを通す。侵害端末の横移動を遮断する**」を、Cilium(eBPF) の L3/L4/L7 ポリシーで実機実証。N3（NDR=検知）に対し N4 は enforcement（遮断）。

## 試験環境（k3d + Cilium, arm64）

| 要素 | 値 |
|---|---|
| クラスタ | k3d v5.9.0（k3s v1.35.5）、標準 CNI(flannel)/network-policy 無効化 |
| CNI | Cilium 1.19.3（eBPF、kubeProxyReplacement=false、l7Proxy=true、Hubble 有効） |
| ワークロード（ns=microseg） | frontend / other（multitool, curl 可）、backend（nginx。`/` と `/admin` を持つ） |
| ラベル設計 | `app=frontend|backend|other`（Cilium Identity の基） |

- **east-west の定義**: frontend/other/backend は同一 namespace・同一 Pod ネットワーク（10.0.0.0/24）の横並び。Pod 間通信が east-west。ここを default-deny にするのが N4。

|---|---|---|---|---|

## 実測エビデンス（要点）

### 段階別 HTTP コード（同一ワークロードにポリシーを重ねた対照）

```text
段階          frontend→/   other→/          frontend→/admin   適用ポリシー
G2 素通し      200          200              200               （なし）
G3 L4          200          000(blocked)     200               NetworkPolicy backend-allow-frontend
G4 L7          200          000(blocked)     403               CiliumNetworkPolicy backend-l7-allow-root
```

- G2→G3 で **other だけが 200→000** に落ちる＝east-west default-deny（許可リスト外の遮断）。
- G3→G4 で **/admin だけが 200→403** に落ちる＝L7 パス単位の遮断（`/` は 200 のまま）。

### G5 — Hubble verdict（L7 と L4 の遮断点の違い）

```text
# L7（HTTP メソッド/パス単位で Envoy が判定）
frontend -> backend:80 http-request DROPPED   (HTTP/1.1 GET .../admin)
frontend <- backend:80 http-response FORWARDED (HTTP/1.1 403 (GET .../admin))
frontend -> backend:80 http-request FORWARDED  (HTTP/1.1 GET .../)
frontend <- backend:80 http-response FORWARDED (HTTP/1.1 200 (GET .../))

# L4（other は SYN 段階で落ちる。L7 まで到達しない）
other <> backend:80 Policy denied DROPPED (TCP Flags: SYN)
```

- **遮断される「深さ」が違う**: other は許可 Identity でないため **L4(SYN) で DROP**。frontend は L4 は通るが `/admin` は **L7(HTTP request) で DROP → 403**。マイクロセグの L3/L4 と L7 の役割分担が verdict に現れる。

### Cilium endpoint（Identity ベース制御の裏付け）

```text
ENDPOINT  POLICY(ingress)  IDENTITY  LABELS
364       Enabled          8991      k8s:app=backend      ← backend のみ ingress enforcement
93        Disabled          62164    k8s:app=frontend
742       Disabled          35131    k8s:app=other
```

- backend endpoint のみ ingress Enforcement=Enabled。IP でなく **Identity（app=backend 等の label）** で対象を選んでいる＝Illumio/TrustSec の「役割ベース制御」を Cilium Identity が実体化。

## 総括

**マイクロセグメンテーションの核心を OSS（Cilium/eBPF, arm64）で L3/L4/L7 の 3 段で実証完了。** 同一 east-west セグメントで、(1) L3/L4 で「許可した端末以外を default-deny」（other 遮断）、(2) L7 で「許可したパス以外を遮断」（/admin=403）を、同一ワークロードにポリシーを重ねる対照で示した。Hubble により **L4 は SYN 段階 DROP、L7 は HTTP request DROP→403** と遮断の深さの違いまで可視化できた。

Illumio / VMware NSX / Cisco Secure Workload が担う「east-west 最小権限・横移動遮断」を、Cilium の Identity ベース宣言的ポリシーで 1:1 に再現。N4 解説の nftables(IP 直書き) の限界＝TrustSec の SGT が解く「IP が変わると破綻する」問題を、Cilium Identity(label ベース) が構造的に解消している点が最大の学び。詰まりどころ（k3d+Cilium の API サーバ検出、標準 NetworkPolicy と CNP の OR マージ）は [構築ログ](../04_構築/構築ログ_2026-07-05.md)。
