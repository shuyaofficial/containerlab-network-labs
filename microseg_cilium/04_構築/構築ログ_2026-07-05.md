# テーマ microseg_cilium マイクロセグメンテーション（Cilium/eBPF）構築ログ（2026-07-05）

NW-ZT トラック N4 の実機実装。k3d(k3s in docker) 上に Cilium(eBPF) を CNI として導入し、east-west 通信を L3/L4（標準 NetworkPolicy）と L7（CiliumNetworkPolicy）の 2 段で default-deny 化した。N3（NDR＝検知）に対し N4 は enforcement（遮断）。

## 設計判断: k3d + Cilium + kube-proxy 併用

- **k3d を採用**（kind ではなく）。arm64/docker で軽量、標準 CNI(flannel) の無効化が `--k3s-arg` で完結する。VM は docker グループ所属のため **k3d は sudo 不要**（前テーマ42 の `sudo docker` とは異なる）。
- **Cilium は cilium CLI で導入**（helm 相当を CLI がラップ）。arm64 対応の 1.19.3 が入った。
- **kubeProxyReplacement=false（kube-proxy 併用）**。k3s の kube-proxy を残し Cilium は CNI+ポリシー+L7 に専念。フル置換だと k3d の LB/DNS と競合しやすいため保守側に倒した。

## 到達点（実証済み）

- 全要素 arm64: kubectl v1.36.2 / k3d v5.9.0 / cilium CLI v0.19.4 / Cilium 1.19.3 / hubble v1.19.4。
- Cilium Ready（DaemonSet/Operator/Envoy/Hubble Relay すべて OK、Cluster Pods 4/4 managed）。
- **L3/L4 east-west default-deny**: frontend→backend=200、other→backend=遮断（000）。
- **L7 パス制御**: frontend の `GET /`=200、`GET /admin`=403（Cilium/Envoy が遮断）。
- **Hubble verdict**: L7 `http-request DROPPED`→403、L4 `Policy denied DROPPED (SYN)` を観測。

## 詰まりどころと解決

### 1. k3d を sudo で叩くと k3d not found（PATH 喪失）

- 症状: `sudo k3d ...` が `sudo: k3d: command not found`。
- 原因: k3d は `~/.local/bin` にあり、sudo は PATH を引き継がない。
- 解決: このユーザーは **docker グループ所属で sudo 不要**（`docker ps` が sudo なしで通る）。k3d を sudo なしで実行するよう変更。deploy.sh から sudo を除去。

### 2. Cilium が CrashLoop（`dial tcp 0.0.0.0:PORT connection refused`）— 最大の詰まり

- 症状: cilium-agent が `Unable to contact k8s api-server https://0.0.0.0:40571` で Init:Error/CrashLoopBackOff。node は NotReady のまま。
- 切り分け: cilium CLI が「kube-proxy has not been installed」と**誤検知**し、`k8sServiceHost` を空にした → Pod が `0.0.0.0` へ API 接続を試みて失敗。`cilium-config` の `k8s-service-host` が空だった。
- 解決: k3d が hostAlias に注入する **server コンテナ名を明示**（`--set k8sServiceHost=k3d-microseg-server-0 --set k8sServicePort=6443`）。→ 再インストールで全 Pod Ready。
- **教訓**: k3d + Cilium では API サーバアドレスの自動検出が効かない。server コンテナ名か k8s Service の ClusterIP を明示する。

### 3. cilium-secrets namespace が Terminating で固まる（再インストール阻害）

- 症状: 再 install が `namespace cilium-secrets is being terminated` / `release name still in use` で失敗。
- 原因: metrics-server が CNI 未確立で Pending → `metrics.k8s.io/v1beta1: stale GroupVersion discovery` が namespace 削除の discovery を失敗させ、`kubernetes` finalizer が外れず Terminating で停止。
- 解決: `kubectl replace --raw /api/v1/namespaces/cilium-secrets/finalize`（finalizers を空にする）で強制削除。ただし本質的には **クラスタごと作り直すのが確実**（deploy.sh を通しで再実行）だったため、最終的にはクラスタ再作成で回避。

### 4. L7 が効かない（`/admin` が 403 でなく 200）— 設計の要

- 症状: L4 の標準 NetworkPolicy と L7 の CiliumNetworkPolicy を**両方適用**すると、`/admin` が 200 のまま（L7 が無効化）。
- 切り分け: CNP には http rule が正しく入り endpoint は Enforcement=Enabled。だが Pod IP 直アクセスでも 403 にならない。標準 NetworkPolicy を**外すと 403 になった**。
- 原因: Cilium は標準 NetworkPolicy と CNP を **OR（和集合）でマージ**する。標準 NetworkPolicy の「L7 指定なしの :80 全許可」が CNP の L7 制限を上書きして全通させていた。
- 解決: **L7 段階では CNP に一本化**（標準 NetworkPolicy を撤去）。CNP は `fromEndpoints` で L3/L4 の default-deny も同時に表現できるため、other→backend の遮断も維持される。deploy.sh の `policy-l7` で標準 NetworkPolicy を delete してから CNP を apply するよう修正。

## 再現手順（後から自分で構築する場合）

1. `./deploy.sh deploy`（tools 取得→k3d クラスタ→Cilium 導入＝G1）
2. `./deploy.sh workload`（microseg ns に frontend/backend/other＝G2）
3. `./deploy.sh test`（素通し疎通＝全 200）
4. `./deploy.sh policy-l4` → `./deploy.sh test`（L3/L4 default-deny＝other 遮断＝G3）
5. `./deploy.sh policy-l7` → `./deploy.sh test`（L7 パス制御＝/admin 403＝G4）
6. `./deploy.sh hubble`（verdict 観測＝G5）
7. 片付け: `./deploy.sh destroy`

## 残課題（次段階）

- **kubeProxyReplacement 完全版**: 今回は kube-proxy 併用。フル eBPF 置換（`kubeProxyReplacement=true`）にすると DSR/直接ルーティング等 eBPF の真価が見える。
- **IP でなくラベル/Identity ベースの徹底**: Cilium は既に Identity ベース（labels）で制御しており、N4 解説の nftables(IP 直書き) の限界＝TrustSec の SGT が解く問題を、Cilium Identity が実体として解決している点を対比できる。
- **egress 方向の制御**: 本ラボは ingress default-deny のみ。egress default-deny（backend から外部への通信制限）で多層化。
- **Hubble UI**: CLI observe まで。UI（Web）は未構築。

## 商用/OSS・解説との関係

N4 解説（[nwzt_N4_解説](../ZERO_zero_trust/解説/nwzt_N4_解説.md)）は Cisco IOL VLAN/ACL + ホスト nftables で「二層μセグ」を説明している。本テーマはその**思想を Kubernetes/Cilium で実装**した OSS 版（テーマ42 が N3 解説の OSS 版だったのと同じ位置づけ）。nftables の「IP 直書き許可リスト・中央管理なし」の限界を、Cilium の **Identity ベース・宣言的ポリシー・eBPF enforcement** が解消している点が学びの核心。
