構築ログ 2026-07-08 — 統合ラボ実デプロイ

build-log done 最終更新 2026-07-08

OrbStack VM clab（arm64）へ統合ラボ（N1+N4 を単一 IOL clab、N2+N3 を docker 併走）を実デプロイし、B1〜B6 を検証した際の記録と詰まりどころ。

分担方針の注記: 通常このワークスペースでは「構築はユーザー本人、Claude は設計書・部材担当」だが、本テーマはユーザーの明示指示により Claude が実デプロイ検証まで実施した（今回限りの上書き）。

1. 事前判断: IOL の 802.1X 受理を probe#

  • L2-advipservices-2017（N4 で L3 SVI+ACL 実績）が 802.1X も受理するかを probe/dot1x_probe.exp で確認。
  • 結果: 全 dot1x コマンド受理、show dot1x all で PAE=AUTHENTICATOR。
確定。 単一 IOL に 802.1X + L3 SVI + inter-VLAN ACL を集約（案B）を確定。IOL 2台構成は回避。

2. 詰まりどころと対処#

1. IOL のインターフェース採番（最重要）

症状。 静的ポートの Vlan20/30（pc-dev/guest-pc）は疎通するのに、radius・サーバ室ブリッジ・pc-unauth が全断。pc-sales の 802.1X が EAP=FAILURE（RADIUS 到達不能）。

原因。 IOL はスロットあたり4ポート。/iol/NETMAP は 0/0 0/1 0/2 0/3 1/0 1/1 1/2 を持ち、Et0/4・Et0/5・Et0/6 は存在しない。clab の eth4/eth5/eth6 は Et1/0・Et1/1・Et1/2 にマップされる。初版 cfg が Et0/4/0/5/0/6 を設定していたため、radius(eth5)/bridge(eth6)/pc-unauth(eth4) が無設定のまま → Vlan100 SVI down → RADIUS 到達不能 → 認証失敗の連鎖。

対処。 core_sw_merged.cfg の当該インターフェースを Et1/0・Et1/1・Et1/2 に修正。nwzt.clab.yml にも採番の注記を追記。修正後 RADIUS 到達回復 → 802.1X 認証成功。

docker exec <node> cat /iol/NETMAP

教訓: clab cisco_iol で eth4 以降を使う場合、IOL 側は Et1/0 から始まる。cfg のインターフェース名は上記コマンドで確認するのが確実。

2. iouyap 未起動でデータプレーン全断

症状／原因。 既知（テーマ22/26/31/microseg 共通）の問題。iouyap が起動していないとデータプレーンが全断する。

対処。 deploy-all.sh の start_iouyap() が下記コマンドを deploy 直後に実行して回避。

docker exec -d -w /iol clab-nwzt-lan-core-sw /usr/bin/iouyap 513

3. docker attach（コンソール）の競合

症状。 expect の docker attach が多重・残留するとコンソールが詰まり、以後の投入が無反応化する。

注意。 pkill -f "docker attach" は自分の親シェル（コマンド行に "docker attach" を含む）まで kill してしまう。下記のように comm=docker に限定して kill する。

ps -eo pid,comm,args | awk '$2=="docker" && /attach/'

4. ZTNA ダークサービス性の担保

症状。 overlay 経由（HTTP 200）は成立するが、clienttun からの直接アクセスも 200 になり「ダーク」でない。

原因。 サーバ室ブリッジ nwzt-srv0 を docker network で作ると ホスト GW（172.31.50.254）が付き、docker ホストが zn-ziti→VLAN50 を L3 転送してしまう。

対処。 apptun を docker network connect した後、下記コマンドでホスト GW を除去。apptun は同一ブリッジ L2 で srv-app1 到達を維持、リモート client の直接経路のみ遮断。→ 直接 = HTTP 000（unreachable）、overlay = 200 を両立。deploy-all.sh ziti に恒久化。

ip addr del 172.31.50.254/24 dev nwzt-srv0

3. リソース共存#

検証時は theme27（clab-qos-acl-policy-lab-*、IOL 7台）が稼働中だったが、本ラボは IOL 1台に集約したため共存できた（メモリ 15Gi、IOL コールドスタートは競合で延伸するが完走）。lab名 nwzt-lan・mgmt 172.20.40.0/24・bridge nwzt-srv0/zn-ziti で既存テーマと分離。

4. 結果#

B1〜B6 合格（B2b 未認証→VLAN99 自動割当のみ ◐ 残課題）。詳細は 08_試験結果報告書 を参照。
