# 要件定義書: ゼロトラスト_ネットワーク特化

## 1. 背景・目的

[ゼロトラスト完全版](../../ゼロトラスト完全版/README_Lab_Challenge.md)は仮想企業（1拠点・約100名）の全社アーキテクチャを設計したが、統合ポイント（I-1〜I-8）・E2Eフロー（F-A〜F-E）の大半は「◐設計のみ（接続未実施）」に留まった。本テーマは、そのうちネットワーク層に閉じる部分（N1 NAC・N2 ZTNA・N3 NDR・N4 μセグ）を選び、**単一の実在するcontainerlabトポロジ上に統合し、実デプロイまで行う**ことで、完全版の「接続◐」を「実際に動く」段階へ進める。L7層（Keycloak/IAP/SIEM/mTLS等 P0-P6）は本テーマのスコープ外とし、完全版の設計のままとする。

`04_構築/`（[nwzt.clab.yml](../04_構築/nwzt.clab.yml)・[core_sw_merged.cfg](../04_構築/core_sw_merged.cfg)・[deploy-all.sh](../04_構築/deploy-all.sh)・[ztna/setup_ziti.sh](../04_構築/ztna/setup_ziti.sh)・[ndr/suricata/local.rules](../04_構築/ndr/suricata/local.rules)）は構築部材として完成済みであり、本書はそれと整合する設計書として書く。

## 2. 対象範囲

- **N1 NAC（802.1X）**: core-sw を認証者、radius(FreeRADIUS) を認証サーバとし、認証成功端末を動的VLAN、未認証端末を隔離VLANへ振り分ける。
- **N4 μセグ**: core-sw の inter-VLAN ACL（層1・粗い分離）と、サーバ側ホスト nftables（層2・同一VLAN内の細かい分離）の二層構成。
- **N3 NDR**: サーバ室ブリッジ `nwzt-srv0` を Suricata が host mode で監視し、east-west の SYN スキャンを検知して Loki/Grafana に集約する。
- **N2 ZTNA（SDP/OpenZiti）**: 保護対象サーバ `srv-app1` をダークサービス化し、overlay 経由のみ到達可能にする。
- 対象外: L7層（ID統制/IAP/SWG/DLP/デバイス統制、[ゼロトラスト完全版](../../ゼロトラスト完全版/README_Lab_Challenge.md) の P0-P6）、I-4（検知→対応の自動CoA連携）の自動化実装。

## 3. 機能要件

- pc-sales の 802.1X 認証成功で VLAN10 へ動的割当されること（RADIUS属性 `Tunnel-Private-Group-Id=10`）。
- pc-unauth（サプリカント未起動＝no-response）が隔離VLAN99へ自動割当されること。
- pc-dev / guest-pc は静的アクセスVLAN（20/30）に収容されること。
- core-sw の SVI 間 ACL `SEG10-OUT` により、VLAN10→srv-app1(172.31.50.11) は tcp/80 のみ許可、tcp/22 は拒否されること。
- 同一VLAN内（サーバ室VLAN50）の host-infected→srv-app1 の東西通信は、ホスト nftables 投入後に遮断されること（投入前は到達可能なことを対照として示す）。
- host-infected→srv-app1 への SYN スキャンが Suricata のカスタムルール（sid:1000001）で検知され、eve.json 経由で Loki/Grafana に集約されること。
- srv-app1 は OpenZiti overlay（apptun→ziti→clienttun）経由でのみ到達可能で、clienttun からの直接到達（172.31.50.11:80）は失敗すること。

## 4. 非機能要件

- 実行基盤は OrbStack VM `clab`（arm64）、containerlab は core-sw のみを管理し、N2/N3 は `deploy-all.sh` による docker 併走とする。
- コアスイッチは単一の IOL（`vrnetlab/cisco_iol:L2-advipservices-2017`）に集約し、認証者(L2)とL3 SVI/ACLを1台に統合する（[基本設計書](../02_基本設計/基本設計書.md) D-1参照）。
- 全ドキュメントは Markdown をソース・オブ・トゥルースとし、絶対パス（`/Users/…`）を含まない。
- frontmatter・章立て・命名は [規約/ドキュメント標準.md](../../規約/ドキュメント標準.md)・[規約/frontmatterスキーマ.md](../../規約/frontmatterスキーマ.md) に準拠する。

## 5. ゾーン設計方針

社内LANは `172.31.0.0/16`、第3オクテット＝VLAN ID とする（完全版のIPアドレス管理表を無変更で踏襲）。VLAN100（コア・認証管理）を中核に、営業(VLAN10)・開発(VLAN20)・ゲスト(VLAN30)・サーバ室(VLAN50)・隔離(VLAN99)へ分離し、監視SOC（論理VLAN90、host mode）が全体を横断監視する。詳細は [IPアドレス管理表](../02_基本設計/IPアドレス管理表.md) を参照。

## 6. 制約

- **arm64制約**: OrbStack VM (Ubuntu 24.04 arm64) 上で実行する。IOLはx86イメージのarm64エミュレーションのため起動に約2分かかる。
- **iouyap必須**: IOLのデータプレーンは deploy 直後に `iouyap`(513) を起動しないと全断する。
- **config投入は expect の1 attach直列**: `docker attach` の多重起動は禁止（[clab運用規約](../../規約/clab運用規約.md)）。
- **サーバ室ブリッジの事前作成**: `nwzt-srv0`(docker network, GW .254) は `containerlab deploy` より前に作成しておく必要がある（`kind:bridge` が既存ブリッジを要求するため）。
- **分担方針の一時的な上書き**: 通常運用の分担方針（構築はユーザー本人、Claudeは設計書・部材担当）を、本テーマに限りユーザー指示により上書きし、Claudeが実デプロイ検証まで行う。
- **I-4は対象外**: 検知→対応の自動CoA隔離は完全版同様に◐設計のまま。本テーマではホスト nftables の手動投入によるデモに留める。

## 7. 受け入れ基準

- README・00〜05のドキュメント一式が frontmatter必須7項目・H1単一・章立て規約を満たすこと。
- 記載するIP・ノード名・ポート割当が実ファイル（[nwzt.clab.yml](../04_構築/nwzt.clab.yml)・[core_sw_merged.cfg](../04_構築/core_sw_merged.cfg)）と一致すること。
- [05_試験/試験計画書.md](../05_試験/試験計画書.md) の B1〜B6 ゲートが定義されていること（判定は実デプロイ後に記録するため、本テーマ作成時点では空欄／未実施表記で良い）。
- `05_試験/試験結果_*.md` は本タスクでは作成しない（実デプロイ後にメインセッションが実測値で記録する）。
- `node 規約/ビルド/build.mjs --check` で本テーマ分のNFD/H1複数/frontmatter欠落エラーが無いこと。
