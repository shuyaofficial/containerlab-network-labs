# テーマ42 NDR（east-west 可視化）試験結果（2026-07-05）

NW-ZT トラック N3。NDR の本質「**同一サブネットの端末同士（east-west）の通信を、経路に割り込まず観測し、横方向の偵察・スキャンを検知してフローとして可視化・集約する**」を実機で実証。

## 試験環境（docker, arm64）

| コンテナ | ネットワーク | 役割 |
|---|---|---|
| attacker | ndr-lan 172.40.0.21 | wbitt/network-multitool。nmap でスキャンする側 |
| victim | ndr-lan 172.40.0.22 | nginx:alpine。狙われる側 |
| suricata | **host mode（ndr-br0 監視）** | jasonish/suricata 8.0.5。DPI + flow 記録 |
| loki / promtail / grafana | host / host / host | eve.json 集約・可視化（grafana 13.1.0） |
| goflow2 / softflowd | host / host | NetFlow 受信/エクスポート試行（ボーナス、代替判断あり） |

- **east-west の定義**: attacker と victim は同一サブネット 172.40.0.0/24（L2 隣接）。両者間の通信は docker bridge `ndr-br0` を横方向に流れる。Suricata はその bridge を promiscuous 監視し、north-south（境界越え）ではなく east-west を捉える。

## 実測エビデンス（要点）

### G2 — east-west SYN scan alert（最重要ゲート）

```json
{"in_iface":"ndr-br0","event_type":"alert","src_ip":"172.40.0.21","src_port":56435,
 "dest_ip":"172.40.0.22","dest_port":121,"proto":"TCP",
 "alert":{"signature_id":1000001,"rev":1,"signature":"ZT-NDR east-west SYN scan",
          "category":"Attempted Information Leak","severity":2},"direction":"to_server"}
```

- threshold（15 回/10 秒で 1 発報）が効き、スキャン全体に対し sid:1000001 は 1 件＝誤爆なし。
- 補助ルール sid:1000002（HTTP to victim）も east-west HTTP を台帳として発報。

### G3 — east-west flow（NetFlow 相当の 5-tuple 統計）

```json
{"in_iface":"ndr-br0","event_type":"flow","src_ip":"172.40.0.21","dest_ip":"172.40.0.22",
 "proto":"TCP","flow":{"pkts_toserver":1,"pkts_toclient":1,"bytes_toserver":58,
 "bytes_toclient":54,"state":"closed","reason":"timeout"},
 "tcp":{"syn":true,"rst":true,"ack":true,"state":"closed"}}
```

- 172.40.0.21 発の flow は 300 件超＝「1 送信元→多数宛先」というスキャンの形が 5-tuple で現れる。
- Suricata capture 統計: `kernel_packets=3492 kernel_drops=0`（bridge から取りこぼしなくキャプチャ）。

### G4 — Loki/Grafana 集約

```text
# Promtail が push したラベル
GET /loki/api/v1/label/job/values        → {"data":["suricata"]}
GET /loki/api/v1/label/event_type/values → {"data":["alert","fileinfo","flow","http","stats"]}

# LogQL で east-west alert を絞る
{job="suricata",event_type="alert"}      → 10 行、全て 172.40.0.21->172.40.0.22

# Grafana
GET /api/health       → {"database":"ok","version":"13.1.0"}
GET /api/datasources  → name=Loki type=loki url=http://localhost:3100 isDefault=True
```

### G5 — NetFlow 代替判断

```text
goflow2 :  msg="starting collection" scheme=netflow port=2055   ← arm64 で受信待受 起動可
softflowd: Packets received by libpcap: 156                     ← bridge から受信はできる
           Packets processed: 0 / Flows exported: 0             ← が、フロー化できず未送出
→ east-west フロー統計は Suricata flow（G3, 300 件超）で代替。goflow2 の受信能力は確認済み。
```

## 総括

**NDR の核心を OSS（Suricata + Loki/Grafana, arm64）で east-west にフォーカスして実証完了。** docker bridge を L2 セグメントに見立て、Suricata を host mode で bridge に張り付けることで、SPAN 配線なしに east-west フレームを直接監視できた。SYN スキャン（横方向の偵察）が **DPI（sid:1000001 alert）とフロー（5-tuple flow 300 件超）の両面**で捉えられ、それが Loki/Grafana に集約されて LogQL で絞り込める。これは Darktrace（DPI 振る舞い）と Cisco Secure Network Analytics/Stealthwatch（NetFlow ベース挙動分析）の 2 系統を 1 センサで追体験する構成。

north-south の IDS（境界の関所で外向きを見る）と違い、N3 は**内部端末同士の横方向を見る**点が本質。「侵害は起きる前提、起きた後に横移動に気づく」という NDR の思想を実機で確認した。NetFlow の完全パイプライン（softflowd→goflow2）は arm64 softflowd の制限で Suricata flow に代替したが、フロー統計としての情報（誰が誰とどれだけ）は等価に取得できている。詳細な詰まりどころは [構築ログ](../04_構築/構築ログ_2026-07-05.md)。
