構築ログ：テーマ16 SNMP/Syslogによる監視基盤の構築

1. 構築概要 #

  • 目的: 企業ネットワークにおいて必須となる「死活監視・トラフィック監視・ログ収集」の概念を学ぶため、Linuxサーバー(NMS)とルーター(R1)の間でSyslogとSNMPを用いた監視基盤を構築する。
  • 使用技術: Syslog (UDP 514), SNMPv2c (UDP 161/162), Zabbix (途中発展課題として導入)
  • 構成: PC1 (cisco_iol) --- R1 (cisco_iol) --- NMS (linux: network-multitool / zabbix-appliance)

2. 実施内容 #

  1. Syslogの実装と検証

    • R1ルーターに logging host と logging trap を設定し、発生したイベントログをNMSへ転送。
    • NMS（Linux）側で nc -ul -p 514 を用いてログをリアルタイム受信するテストを実施。
    • PC1側のインターフェース（R1と対向）をシャットダウンさせることで、「リンクダウン」という物理的イベントがどのようにSyslogとして生成され、伝達されるかのメカニズムを確認した。

  2. SNMPv2cの実装と検証（ポーリングとトラップ）

    • R1にSNMPエージェント機能（snmp-server community public RO）と、トラップ送信機能（snmp-server host）を設定。
    • NMSから snmpwalk コマンドを用いてR1の情報を手動で引き抜く（ポーリング）実験を実施。

  3. 【発展】Zabbixサーバーの導入と可視化

    • LinuxのNMSコンテナを一時的に破壊し、実務で広く使われている統合監視ソフトウェア「Zabbix」の公式コンテナ（zabbix/zabbix-appliance:ubuntu-latest）にすげ替えて再デプロイ。
    • ブラウザからZabbixのWeb GUIにアクセスし、R1ルーターを監視対象ホストとして登録。
    • ディスカバリ機能（LLD）によるインターフェースの自動探索を経て、通信トラフィックやICMP応答速度が視覚的なグラフとして自動描画されることを確認した。
    • 意図的にポートを遮断することでZabbixが「重度の障害」としてリアルタイム検知する様を体験。

3. トラブルシューティングと学び #

  • 監視経路の断絶問題: PC1のインターフェースをシャットダウンした際、「ダウンした」というログ自体がPC1からNMSへ届かない（通信経路自体が遮断されるため）という事象を自ら発見。これにより、ネットワーク設計における「冗長化」と「Loopbackインターフェースの活用」の必要性を深く理解した。
  • Zabbixのグラフスケール問題: 全ての監視項目（10Gbpsのポート速度制限など）を1つのグラフに重畳させた結果、実際のトラフィックがゼロに張り付いて見える現象（スケール問題）を体験。適切な項目のみをフィルタリングして表示する運用の基本を学んだ。
  • L3 vs L7監視: SNMP（L3〜ハードウェア中心の監視）とZabbix Agent等の専用エージェント（L7/アプリケーション層の監視）の違いや、それぞれの得意分野・業務での使い分けについて理解を深めた。
