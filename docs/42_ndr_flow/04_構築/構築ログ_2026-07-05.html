テーマ42 NDR（east-west 可視化）構築ログ（2026-07-05）

2026-07-05 zero-trustndrsuricatanetflowdocker

NW-ZT トラック N3 の実機実装。docker bridge を east-west セグメントに見立て、Suricata を host mode で bridge に張り付けて配下コンテナ間フレームを DPI 監視し、eve.json（alert/flow）を Loki/Promtail/Grafana に集約した。

設計判断: clab でなく docker で構築 #

N3 の主眼は「同一サブネット（L2 隣接）の端末同士＝east-west をセンサでどう捉えるか」。IOL の SPAN を使う本来構成もあるが、本ラボは docker network の bridge そのものを監視対象 L2 セグメントとみなし、Suricata を host mode でその bridge インターフェースに直接張り付ける方式にした。ミラー（SPAN）配線の代わりに bridge の promiscuous キャプチャが east-west フレームを複製なしで直接見られる。N1（IOL の L2）と違い NOS 不要で再現性が高い。

到達点（実証済み） #

  • 全イメージ arm64 対応（suricata 8.0.5 / loki / promtail / grafana 13.1.0 / goflow2、2026-07-05 実測 pull 成功）。
  • east-west 検知: attacker(172.40.0.21)→victim(172.40.0.22) の SYN スキャンで sid:1000001 が発報。
  • flow 可視化: 同 east-west を 5-tuple のフローとして eve.json に記録（172.40.0.21 発 300 件超）。
  • 集約: Promtail が eve.json を Loki へ push、LogQL {job="suricata",event_type="alert"} で alert 抽出。Grafana に Loki を自動プロビジョン。

構成の要点 #

  • bridge 名を固定: docker network create --opt com.docker.network.bridge.name=ndr-br0 で bridge if 名を ndr-br0 に固定。inspect でも bridge=ndr-br0 を確認。名前を固定しないと Suricata の -i 指定先が起動ごとに変わる。
  • Suricata は host mode + -i ndr-br0: --network host --cap-add NET_ADMIN --cap-add NET_RAW --cap-add SYS_NICE。ホストの他 IF には触れず ndr-br0 のみ監視。
  • HOME_NET/ルールはコマンドラインで上書き: 巨大な suricata.yaml（2362 行）を丸ごと bind せず、-S /rules/local.rules（追加ロード）＋ --set vars.address-groups.HOME_NET="[172.40.0.0/24]" で最小差分にした。eve-log の alert/flow はデフォルトで有効なので触らない。
  • eve.json を Mac 側に bind: suricata/log/ を /var/log/suricata に bind し、Mac から直接 grep 可能に。

詰まりどころと解決 #

1. Suricata のルールが確実にロードされたかの確認（最重要ゲート前チェック） #

  • 症状候補: ルール 0 件だと SYN スキャンしても sid:1000001 が出ない。
  • 確認: suricata.log に 2 rules successfully loaded, 0 rules failed、runmodes: ndr-br0: creating 10 threads を確認してから G2 に進んだ。「検知が出ない」を切り分ける前に、まずルール読込と監視 IF を確認するのが定石。

2. Grafana から host の Loki(3100) に届かない問題 #

  • 症状想定: Grafana を bridge（-p 3000:3000）で起動すると、datasource url=localhost:3100 はコンテナ内 localhost を指し Loki に届かない。
  • 解決: Grafana も host mode で起動。Loki は -p 3100:3100 で host に publish 済みなので、host mode の Grafana から localhost:3100 で確実に届く。datasource は provisioning で自動投入（isDefault=True を API で確認）。

3. Suricata flow はタイムアウト後に出る #

  • 症状: スキャン直後は flow イベントがほぼ出ない（alert は即時）。
  • 原因: Suricata の flow イベントはフローが閉じる/タイムアウトしてから eve.json に書かれる。
  • 対応: 試験では until grep ...; do sleep 5; done でポーリング待機。数十秒でフローがタイムアウトし 300 件超の east-west flow が出た。

4. NetFlow エクスポータ（softflowd）が arm64/bridge でフロー化できない（G5 代替判断） #

  • 症状: softflowd v1.1.0(alpine/arm64) は Packets received by libpcap: 156 と ndr-br0 のパケットを受けるが Packets processed: 0／Flows exported: 0。goflow2 に何も届かない。
  • 切り分け: goflow2 自体は host mode で netflow://:2055 待受を起動できる（arm64 実証済み）。softflowd 側でフロー化が起きていない（libpcap 受信はできるが処理 0）ため、bridge の DLT 扱いに起因する softflowd 側の問題と判断。BPF フィルタ ip 付与・短命フロー（maxlife=3）でも processed=0 のまま。
  • 代替判断: east-west のフロー統計は Suricata の flow イベントで代替（5-tuple・バイト/パケット数を持ち NetFlow と等価な情報。G3 で 300 件超を実証済み）。無理に softflowd 深追いはせず、deploy.sh netflow にコメントで制限を明記した。goflow2 の受信能力は確認済みなので、正常に吐くエクスポータ（IOL の flow exporter 等）があればそのまま受けられる。

再現手順（後から自分で構築する場合） #

  1. ./deploy.sh deploy（ndr-lan 作成＋attacker/victim/suricata 起動＝G1）
  2. ./deploy.sh scan（attacker→victim へ SYN スキャン＝G2 トリガ）
  3. ./deploy.sh eve（eve.json の alert/flow 抜粋＝G2/G3 確認）
  4. ./deploy.sh monitoring（Loki+Promtail+Grafana＝G4 集約）
  5. ./deploy.sh netflow（goflow2+softflowd＝G5 ボーナス。制限あり）
  6. 片付け: ./deploy.sh destroy

残課題（次段階） #

  • NetFlow の完全パイプライン: softflowd の代替（fprobe / pmacct / IOL の flow exporter）で goflow2 まで通す。goflow2 の受信側は動作確認済み。
  • Grafana ダッシュボード自動プロビジョン: 現状は datasource のみ自動投入。alert 件数・トップトーカーのパネルを JSON でプロビジョン化。
  • N1 との SOAR 連携（発展）: Suricata の sid:1000001 発報 → webhook → FreeRADIUS CoA で attacker 相当端末を隔離 VLAN へ、の自動封じ込め。
  • Zeek の併設: 本ラボは Suricata（シグネチャ＋flow）のみ。Zeek の conn.log による「通信の台帳」を足すと DPI 2 系統の対比ができる。
