NW-ZT トラック N2 の実機実装。SDP のダークサービスを OpenZiti で成立させた。

設計判断: clab でなく docker で構築 #

OpenZiti は L7 オーバーレイであり NOS（IOL）を要しない。N1（NAC）が IOL の L2 を扱うため clab だったのに対し、N2 は素の docker（deploy.sh による docker run オーケストレーション）が自然で再現性も高い。ネットワーク分離（zn-ziti / zn-app）で「client から app が見えない」を物理的に担保した。

到達点（実証済み） #

  • OpenZiti 全イメージ arm64 対応（controller/router/cli/tunnel、2026-07-04 実測）。
  • ziti edge quickstart（controller+router 一体）が arm64 で起動、router ONLINE。
  • サービス webapp を Bind/Dial ポリシーで認可し、2 tunneler を enroll。
  • ダークサービス実証: client は localhost:8080（overlay）で darkweb に HTTP 200、darkweb:80（直接）は HTTP 000。

詰まりどころと解決 #

1. quickstart の --home 権限エラー #

  • 症状: mkdir /persist: permission denied でコンテナ即終了。
  • 原因: ziti-cli イメージは非 root ユーザー（ziggy）で動くため、root 直下 /persist を作れない。
  • 解決: --home /tmp/ziti（書込可能）に変更。

2. enrollment JWT の権限（重要） #

  • 症状: failed to parse JWT: token contains an invalid number of segments。
  • 切り分け: JWT ファイル自体は正常（994 バイト・eyJ...）。だが docker cp が root 所有 0600 で配置 → 非 root の ziti プロセスが読めず Permission denied。
  • 解決: 配布後に docker exec -u root <c> chmod 644 <jwt>。→ enroll 成功。
  • 教訓: 非 root コンテナへ docker cp した機密ファイルは、実行ユーザーが読めるよう chmod する。

3. compose 不在 #

  • VM に docker compose/docker-compose が無いため、docker run ベースの deploy.sh に統一（移植性も向上）。

再現手順（後から自分で構築する場合） #

  1. ./deploy.sh deploy（zn-ziti/zn-app 作成＋ziti/darkweb/apptun/clienttun 起動）
  2. ./deploy.sh setup（＝setup_ziti.sh。サービス/ポリシー/enrollment/tunneler 起動＋実証まで自動）
  3. 片付け: ./deploy.sh destroy

残課題（次段階） #

  • 未認可 identity の拒否デモ: dial ポリシー外の identity では webapp に到達できないことを明示（現状は認可済みのみ検証）。
  • posture 連動: ZPA は端末コンプライアンスを認可に組込む。OpenZiti の posture check（MFA/OS/プロセス）で拡張。
  • tproxy/DNS 傍受: 本デモは proxy モード（localhost:8080）。実運用に近い webapp.ziti の DNS 傍受（tproxy）へ発展。
