  • G2→G3 で other だけが 200→000 に落ちる＝east-west default-deny（許可リスト外の遮断）。
  • G3→G4 で /admin だけが 200→403 に落ちる＝L7 パス単位の遮断（/ は 200 のまま）。

G5 — Hubble verdict（L7 と L4 の遮断点の違い） #

# L7（HTTP メソッド/パス単位で Envoy が判定）
frontend -> backend:80 http-request DROPPED   (HTTP/1.1 GET .../admin)
frontend <- backend:80 http-response FORWARDED (HTTP/1.1 403 (GET .../admin))
frontend -> backend:80 http-request FORWARDED  (HTTP/1.1 GET .../)
frontend <- backend:80 http-response FORWARDED (HTTP/1.1 200 (GET .../))

# L4（other は SYN 段階で落ちる。L7 まで到達しない）
other <> backend:80 Policy denied DROPPED (TCP Flags: SYN)
  • 遮断される「深さ」が違う: other は許可 Identity でないため L4(SYN) で DROP。frontend は L4 は通るが /admin は L7(HTTP request) で DROP → 403。マイクロセグの L3/L4 と L7 の役割分担が verdict に現れる。

総括 #

マイクロセグメンテーションの核心を OSS（Cilium/eBPF, arm64）で L3/L4/L7 の 3 段で実証完了。 同一 east-west セグメントで、(1) L3/L4 で「許可した端末以外を default-deny」（other 遮断）、(2) L7 で「許可したパス以外を遮断」（/admin=403）を、同一ワークロードにポリシーを重ねる対照で示した。Hubble により L4 は SYN 段階 DROP、L7 は HTTP request DROP→403 と遮断の深さの違いまで可視化できた。

Illumio / VMware NSX / Cisco Secure Workload が担う「east-west 最小権限・横移動遮断」を、Cilium の Identity ベース宣言的ポリシーで 1:1 に再現。N4 解説の nftables(IP 直書き) の限界＝TrustSec の SGT が解く「IP が変わると破綻する」問題を、Cilium Identity(label ベース) が構造的に解消している点が最大の学び。詰まりどころ（k3d+Cilium の API サーバ検出、標準 NetworkPolicy と CNP の OR マージ）は 構築ログ。
