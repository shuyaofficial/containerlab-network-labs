B-2. Git/GitHub公開手順 #

課題 #

Mac仮想環境構築/ は現状未git管理（132MB）。公開判断は個別テーマ単位で発生しうるが、以下がリポジトリ直下に混在している。

  • 実行時生成物: clab-*/ ランタイムディレクトリ（TLS CA秘密鍵を含む。2026-07-03時点で12箇所の clab-*/.tls/ca/ca.key を確認済み）
  • 大容量バイナリ: Ciscoの .bin（IOL）・.qcow2（FortiGate等）
  • ライセンス関連: IOURC* およびその展開物
  • 生成された構成図・ビルド出力（規約/ビルド/ のHTML出力想定）

これらを未サニタイズのままpushすると、秘密鍵の漏洩・ライセンス条項違反・リポジトリ肥大化のいずれかが起きる。

提案: .gitignore 方針 #

パターン 理由
clab-*/ TLS CA秘密鍵・実行時状態を含むランタイムディレクトリ全体を除外
*.bin Cisco IOLバイナリ（配布不可のライセンス素材）
*.qcow2 FortiGate等のVMイメージ本体
IOURC* Cisco IOLライセンスファイルおよびzip展開物
*license* ライセンス関連ファイルの汎用除外（大文字小文字問わず運用）
node_modules/ ビルドツール（規約/ビルド/build.mjs）の依存物
docs/assets/ などビルド生成物 MD→HTMLビルドの出力物（再生成可能なため）

提案: サニタイズ工程（公開前の走査手順） #

  1. 秘密鍵走査: find . -path "*clab-*/.tls*" で残存確認 → 上記gitignoreが効いていれば0件のはず。念のため git status --ignored で除外漏れがないか確認。
  2. ライセンス文字列走査: grep -ril "iourc\|hostid\|license" --include='*.md' --include='*.yml' でライセンス関連の記述を洗い出し、実際のキー・シリアル値が平文で書かれていないか確認（手順書としての言及は可、実値の埋め込みは不可）。
  3. 個人情報・認証情報走査: grep -rilE "password|passwd|secret|token" --include='*.md' --include='*.yml' --include='*.sh' でハードコード認証情報を確認。ラボ環境の初期パスワード（例: FortiGateの初回未設定パスワード）は「未設定である旨の説明」であればドキュメント上問題ないが、実際に設定した本番相当の値は除外する。
  4. ユーザー名・パス走査: grep -r "/Users/shuya" で絶対パスの露出を確認し、相対パス・プレースホルダに置換する。

公開前チェックリスト #

  • [ ] .gitignore を上記方針で設置済み
  • [ ] git status --ignored で clab-*/ ・バイナリ・ライセンス類が除外リストに乗っている
  • [ ] 上記3種のgrep走査を実施し、ヒットをレビュー済み
  • [ ] 絶対パス（/Users/shuya/...）を相対パスまたはプレースホルダに置換済み
  • [ ] READMEやCLAUDE.mdに含まれる内部限定情報（社内ネットワーク構成の実IP帯など）を確認済み

B-3. 試験・切り分け教訓のテンプレ化 #

課題 #

障害切り分けは各テーマで都度その場で記録しており、様式が揺れている（例: 試験結果_YYYY-MM-DD.md と 試験完了記録_YYYY-MM-DD.md の混在、構築ログ_テンプレート.md の有無がテーマにより異なる）。過去の運用教訓（Obsidian Mistakes.md）として蓄積されている以下の5点は、個別テーマの記録に埋もれると再利用されにくい。

  1. 外部リソースは手持ちストレージ網羅検索が先
  2. ライブ状態（show系コマンド）を設定解析より先に採取
  3. 同一症状に複数独立原因を想定し層別切り分け
  4. 日本語ファイル名はNFC正規化
  5. iCloudロックは brctl evict/brctl download で回避

提案 #

  • 05_試験/切り分けシート.md の雛形（type: troubleshoot）を 規約/雛形/ に追加し、新規テーマは必ずこの雛形からコピーする。
  • 雛形の冒頭に上記5教訓を「着手前チェック」として埋め込み、都度Obsidianを参照しなくても切り分け時に自然に思い出せる形にする。
  • 個別テーマの切り分けシートで新たな教訓が見つかった場合、そのテーマ内に留めず、意味のある区切りでObsidian Mistakes.md へも追記する（グローバル運用ルールに準拠）。テンプレへの反映は四半期棚卸し（B-1）のタイミングでまとめて検討する。

B-4. 進捗・AAR接続 #

課題 #

進捗の状態は3箇所に分散している。

  1. 各テーマMDの frontmatter status（draft/review/done/superseded）
  2. ロードマップ/進捗管理.md の状態凡例（✅/◐/⏹/⚠/⬜）
  3. Obsidian長期記憶（Daily/・Decisions.md・Mistakes.md）のAAR記録

これらは現状、更新のタイミングも粒度も独立しており、片方を更新してももう片方が追随しない（例: テーマ22は ⏹部分達成で終了 だが、フォルダ内の個別MDの status は未統一）。

提案: 双方向リンク運用 #

更新元 更新すべき先
テーマ内MDの status を done に変更 進捗管理.md の該当行の状態記号を更新し、可能なら振り返りファイルへのリンクを追加
進捗管理.md の状態を更新 該当テーマの主要MD（README・基本設計書など）の frontmatter status も揃える
意味のある区切り／タスク完了 Obsidian Daily/YYYY-MM-DD.md に該当テーマ名を明記して追記（グローバル運用ルールに準拠、本フォルダ内には書かない）
AARで「予想と実際のズレ」が学びだった場合 Obsidian Mistakes.md へ。テーマ固有の切り分けシートにも該当があれば併記
  • 3箇所の同期は自動化しない（過剰な仕組み化を避ける）。「テーマの状態が変わったら3箇所を思い出して見る」をセルフ監査チェックリストの項目として明文化するに留める。
