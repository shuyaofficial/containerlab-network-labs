2. 必須ドキュメントと章立て #

各文書の必須章（H2）。詳細な小見出しは 雛形/ の各テンプレートが定義する。

文書 必須章（H2）
README_Lab_Challenge 概要／トポロジ（```mermaid）／前提環境／開始手順／Mission（段階課題・H3でゲート条件）／禁止事項／参照
要件定義書 背景・目的／対象範囲／機能要件／非機能要件／ゾーン設計方針／制約（arm64・IOL起動2分等）／受け入れ基準
基本設計書 設計方針／全体構成／アドレス設計方針／冗長性・拡張性／セキュリティ方針／設計判断の記録（考えどころ）
IPアドレス管理表 管理サブネット／ノード一覧表（ノード名・役割・mgmt-ipv4・Loopback・備考）
パラメータシート 機器別パラメータ表（インターフェース／L2／ルーティング／セキュリティの順）
試験計画書 試験方針／試験項目表（ID・項目・手順・期待結果・判定）／合格基準
試験結果 実施サマリ／結果表（試験IDと対応）／不合格項目の分析／残課題
構築ログ 日時・作業・コマンド・結果・エラーと対処・学習ポイント
切り分けシート 症状記録／ライブ状態採取（設定解析より先）／層別切り分け表／複数原因の並行仮説／根本原因・恒久対策／AAR
ログインコマンド VM接続／機器コンソール一覧（docker attach等）／エンドポイント接続／確認済み日付

切り分けシートの設計意図（運用教訓の構造化） #

過去の教訓（Obsidian Mistakes.md）をチェック欄として強制する：

  1. ライブ状態を先に採取（show ip route 等）— 設定ファイル解析より先
  2. 層別切り分け — L2到達 → VLAN存在 → SVI Up の順で表を埋める
  3. 複数独立原因の並行仮説 — 同一症状に独立原因が重複する前提で仮説を2つ以上立てる欄

3. 記法規律 #

  • H1はファイルに1つ。frontmatter直後に置く。見出しレベルのスキップ禁止（H2の次にH4を置かない）
  • 表: ヘッダ行必須。セル内改行は <br>。10列を超える表は分割する
  • リンク: テーマ内は相対パス。テーマ跨ぎは ../ 起点の相対パス。絶対パス（/Users/…）埋め込み禁止（GitHub移植性）
  • コードブロック: 言語指定必須（bash / yaml / text。IOS CLIは text）
  • 構成図: Mermaid第一（構成図規約.md）。README内に ```mermaid で直接埋める
  • 設定コマンドの答え: 学習テーマでは書かない（mentor_guidelines.md準拠。showコマンドのヒントは可）。テーマZERO等のClaude実装テーマは対象外

4. 命名規約 #

  • ファイル・ディレクトリ名は NFC正規化必須（NFD混入はNFD/NFC濁点問題で検索・ビルドの偽陰性を生む。build.mjs --check が検出する）
  • テーマ番号は NN_ ゼロ埋め2桁（特例: ZERO_）。番号重複禁止
  • 日付入りファイル名は _YYYY-MM-DD サフィックス
  • 雛形は <対象ファイル名>.tmpl

5. MD＋HTML両生成 #

  • MDがソース・オブ・トゥルース。HTMLは 規約/ビルド/build.mjs による生成ビューで、手編集禁止
  • 出力先は Mac仮想環境構築/docs/（GitHub Pages公開単位）。テーマフォルダ内にHTMLを散らさない（例外: 02_基本設計/ネットワーク構成図.html のような手作りの図HTML）
  • 品質ゲート: コミット前に node 規約/ビルド/build.mjs --check をpassさせる
このHTMLは 規約/ビルド/build.mjs による自動生成ビューです。ソースはMarkdown（手編集禁止）。
