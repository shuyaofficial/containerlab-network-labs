ADR: ドキュメント形式としてMarkdownを継続する

2026-07-03 standardadrmarkdownformat
  • ステータス: 採択（2026-07-03）
  • 決定者: ユーザー（形式再検討の指示）＋ Claude（検討・起案）

背景 #

テーマ02〜27の設計文書はすべてMarkdownで作成してきたが、テーマ数の増加とGitHub公開・HTML出力の要件を機に「本当にMDのままでよいのか」を徹底検討した。

決定 #

Markdownをソース・オブ・トゥルースとして継続する。 ただし下記の頑健化策をセットで義務付ける（素のMD運用は継続しない）。

根拠 #

  1. git差分が意味単位で読める — 設計変更のレビュー・履歴追跡が行単位で成立する。バイナリ形式（xlsx/docx/drawio）では不可能
  2. CLIログ・設定がそのまま貼れる — NW検証はテキストの塊であり、変換工程ゼロで一次情報を保存できる
  3. GitHubでそのまま読める＋Mermaidがネイティブ描画される — 公開時に追加変換なしで閲覧品質が確保される
  4. 全既存資産（26テーマ分）がMD — 形式移行はコストのみ大きく、学習フローも壊す
  5. AIエージェントが最も安定して生成・監査できる形式 — 雛形生成・規約検査・HTML変換の全パイプラインが単純になる

頑健化策（素のMDとの差分） #

弱点 対策
メタデータがなく機械検索に弱い YAML frontmatter必須（frontmatterスキーマ.md）
図が別ファイル・別形式に散る Mermaid第一（構成図規約.md）。テキストで差分管理、GitHub/HTML両描画
表の書式が崩れやすい 表の規律（ヘッダ必須・<br>改行・巨大表分割）を標準化
リンク切れ・移植性 相対リンク規約＋build.mjs --check の自動検査
日本語ファイル名のNFD/NFC問題 NFC正規化必須＋ビルド時検出
「見せられる品質」にならない build.mjs でHTML生成（紙系デザイントークン・印刷対応・TOC）

却下した代替案 #

代替案 却下理由
AsciiDoc / reStructuredText 表現力は上だがGitHubネイティブ描画・AIツール適合・既存資産の面でMDに劣後。学習コストに見合わない
Word / Excel（docx/xlsx） git差分不能・CLIログ貼付に不向き・自動検査不能。PJ管理資料（WBS等）の数値管理用途に限定して併用は継続
Notion / Confluence 等SaaS ローカル完結・オフライン・git管理の原則に反する。エクスポート形式のロックインリスク
drawio中心の図管理 バイナリ同然のXMLで差分不能、GitHub非描画。既存図は残すが新規は非推奨
YAML/データ全面化（パラメータをすべて構造化） 学習時の手記入フロー（詳細設計を自分で書く）を壊す。ユーザー決定（2026-07-03）によりMD＋HTML両生成の方針を採用

リスクと緩和 #

  • frontmatter記入漏れ → --check ゲートで機械検出、雛形に既定値を埋め込み
  • Mermaidで表現しきれない詳細図 → 手作りHTML図（テーマ26方式）を例外として許容
  • HTML生成物とMDの乖離 → HTMLは常に再生成可能な派生物とし、手編集を禁止

再検討の条件 #

以下のいずれかが発生したら本ADRを見直す： パラメータ転記ミスがテーマ3回連続で発生（→データソース化を再検討）／GitHub以外への移行／チーム共同編集の開始。

このHTMLは 規約/ビルド/build.mjs による自動生成ビューです。ソースはMarkdown（手編集禁止）。
