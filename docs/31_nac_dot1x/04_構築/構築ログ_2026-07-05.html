テーマ31 NAC/802.1X 構築ログ（2026-07-05）

NW-ZT トラック N1 の実機実装。IOL L2 スイッチ＋FreeRADIUS＋サプリカントで 802.1X 認証と動的 VLAN を成立させた。

到達点（実証済み） #

  • IOL L2-15.2 は 802.1X を実装している（dot1x system-auth-control / authentication port-control auto / dot1x pae authenticator / mab すべて受理、show dot1x all でポートが AUTHENTICATOR）。
  • FreeRADIUS（arm64、apt 導入）が動的 VLAN 属性を返す（Access-Accept + Tunnel-Private-Group-Id=10）。
  • 802.1X エンドツーエンド成功: pc1 の wpa_supplicant（EAP-MD5, alice）→ スイッチ → FreeRADIUS で認証成功し、Et0/1 が動的に VLAN 10（BUSINESS）へ割当（show authentication sessions = alice / Authorized / Vlan: 10）。

詰まりどころと解決（重要な教訓） #

1. 最大の落とし穴: iouyap 未起動でデータプレーンが全断 #

  • 症状: スイッチ→RADIUS ping 0%、test aaa = User rejected、FreeRADIUS に要求が届かない。IOL の Et0/3 は 入力 0 パケット（出力はある）。
  • 切り分け: radius→pc1（linux 同士）は疎通 OK。radius 送信フレームは sw1 名前空間の veth までは届く（RX カウンタ +5）が、IOL の RX に入らない。
  • 根本原因: clab の cisco_iol は iouyap が IOL の UDP イーサネットセグメントと Linux veth を中継する方式。lab-dir に iouyap.ini/NETMAP があるのに iouyap プロセスが未起動だった（containerlab deploy を直接叩き、iouyap を起動する deploy.sh を使わなかったため）。
  • 解決: sudo docker exec -d -w /iol clab-nac-sw1 /usr/bin/iouyap 513 で起動 → 即疎通（ping 100%、test aaa 成功）。
  • 恒久対策: deploy.sh に start_iouyap() を組み込み（テーマ22/26 と同じ思想）。これが N1 再現の必須手順。

2. RADIUS 到達は「管理IF」でなく「データポート＋SVI」で #

  • IOL L2 の管理 IF（Et0/0）経由での RADIUS 到達は不安定だったため、RADIUS をデータポート Et0/3（VLAN100）に接続し、SVI interface Vlan100（172.31.0.1）から到達する構成にした。実機スイッチが RADIUS に到達する標準的なやり方で、教育的にも良い。

3. コンソール自動化の癖 #

  • clab cisco_iol のコンソールは「コンテナ主プロセスの stdio」＝ docker attach 方式。スクリプト化は expect で駆動。
  • 長い設定行（radius-server host ...）はコンソールの行折返しで化ける → radius server ブロック形式の短い行に分割して回避。
  • attach は多重起動するとコンソール I/O が壊れる。1 セッション完結＋実行後 pkill docker attach で掃除する運用にした。

再現手順（後から自分で構築する場合） #

  1. 04_構築/freeradius/ と client/ のイメージをビルド（docker build -t nac-freeradius:local / nac-client:local）。
  2. ./deploy.sh deploy（clab deploy → iouyap 起動 → オフロード無効化まで自動）。
  3. sudo expect run_nac.exp clab-nac-sw1 sw1_dot1x.cfg でスイッチ設定投入。
  4. pc1 で wpa_supplicant -i eth1 -D wired -c /etc/wpa_supplicant.conf を起動。
  5. sudo expect verify_final.exp clab-nac-sw1 で show authentication sessions を確認（Et0/1 が VLAN 10）。
  6. 片付け: ./deploy.sh destroy。

残課題（次段階） #

  • pc2（未認証）→ 隔離 VLAN 99 の自動割当: IBNS 1.0 の authentication event no-response/fail action authorize vlan 99 のタイミング/構文を 15.2 で詰める（現状 pc1 の認証成功→VLAN10 は確実。未認証ポートは Unauthorized で通信不可＝拒否は成立）。
  • CoA（Change of Authorization）: radclient から Disconnect/CoA を送って稼働セッションの強制再認証を実演。
  • Keycloak を LDAP/OIDC バックエンドに連携（発展）。
