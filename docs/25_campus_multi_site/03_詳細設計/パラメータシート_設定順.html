なぜ「順番」で並べ替えたか #

ノード別に値だけ並んでいると、投入順が見えず次の事故が起きやすい：

  • Router ID 事故: Loopback を作る前に OSPF を有効化すると、意図しない Router ID が選ばれ、 直すのに clear ip ospf process（＝隣接の張り直し）が必要になる。
  • 参照先不在: prefix-list / route-map を作る前に redistribute を書くと、参照先が無く挙動が崩れる。

これを避けるため、Cisco のベストプラクティスに沿って次の3原則で並べています：

  1. Router ID はプロセス有効化の前に確定 — Loopback を先に作り、プロセス内で router-id を明示（自動選出に頼らない）。
  2. prefix-list → route-map → redistribute の順 — ポリシーオブジェクトを先に用意してから参照させる。
  3. 相互再配信はタグでループ防止 — 片方向で tag 付与、逆方向で tag 一致を deny。

参照: Redistribution of OSPF Routes into BGP (Cisco) ／ IOS XR Deployment Best Practices for OSPF/IS-IS and BGP (Cisco) ／ Redistribution of iBGP Routes Into OSPF (Cisco)

Phase 1. Loopback0 を最初に作る（4台） #

先に作る理由: Loopback0 は Router ID の供給源。プロセス起動前に up させておけば、 OSPF/BGP が意図どおりの Router ID を選び、clear ip ospf process を回避できる。

Phase 3. OSPF 有効化（Router ID 明示 → area 0 参加） #

この順の理由: Loopback 作成後・隣接形成前に Router ID を明示固定するのが最も安全（Phase1 の値をそのまま使う）。 Main は process 10、Branch は process 20、いずれも area 0。参加対象は内部P2P・Loopback0・利用者セグメントのみ。

ノード OSPF Process Area Router ID（明示） 参加IF 投入
main-core 10 0 10.50.255.1 E0/1・E0/2・Loopback0 ⬜
main-edge 10 0 10.50.255.254 E0/1・Loopback0（E0/2 WANは除外） ⬜
branch-edge 20 0 10.60.255.254 E0/1・Loopback0（E0/2 WANは除外） ⬜
branch-core 20 0 10.60.255.1 E0/1・E0/2・Loopback0 ⬜

両キャンパスとも area 0 だが、BGP 境界を越えて同一 OSPF ドメインにはならない。プロセス番号は識別用で対向とは無関係。

順序を守る理由（再掲）

  1. Router ID 先出し … Loopback→RID明示で clear ip ospf process を回避。

  3. タグでループ防止 … 片方向で tag 付与、逆方向で tag deny。
