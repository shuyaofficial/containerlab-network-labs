要件定義書 — Theme 24: Routed Access Campus LAN

1. プロジェクトの目的 #

株式会社XXX（架空・社員約100名）は、フロアアクセス層までL2を延伸する従来構成から、アクセス層までL3ルーティングを適用するRouted Accessへ移行する。

目的は、STPのブロッキングポートと再計算への依存を減らし、OSPFとECMPを用いて全アップリンクを利用しながら、単一リンク・単一コア障害時の高速な経路切替を実現することである。

2. 対象範囲 #

2.1 対象 #

  • Core/Distribution役2台
  • L3 Access役2台
  • Access配下の試験端末2台
  • インフラ間のRouted Port
  • OSPF area 0
  • 等コスト経路によるECMP
  • 単一リンク・単一コア障害試験

2.2 対象外 #

  • STP、VLAN Trunk、HSRPによる冗長化
  • インターネット接続、NAT、Firewall
  • BGP、拠点間WAN、IPsec
  • VXLAN/EVPN Overlay
  • 実機Catalyst固有の機能・性能評価

3. 業務・通信要件 #

ID 送信元 宛先 要件
R-01 pc-a pc-b 双方向にIP疎通できること
R-02 access1 access2の利用者セグメント core1/core2の等コスト2経路を利用できること
R-03 全L3ノード 全Loopback OSPF学習後に疎通できること
R-04 pc-a pc-b 単一アップリンク障害後に残存経路で復旧すること

4. 機能要件 #

4.1 物理・論理構成 #

  • core1/core2とaccess1/access2を相互に二重接続する。
  • core1とcore2間にもL3リンクを設ける。
  • インフラ間5リンクはすべて/30のPoint-to-Pointネットワークとする。
  • 利用者セグメントはaccess1配下とaccess2配下に1つずつ設ける。

4.2 ルーティング #

  • 全L3ノードをOSPF area 0へ収容する。
  • Router IDは各ノードのLoopback0を使用する。
  • 利用者セグメントはOSPFで広告するが、端末側へOSPF Helloを送らない設計とする。
  • access1/access2から対向利用者セグメントへの経路を等コストにする。

4.3 障害対応 #

  • 単一Access-Coreリンク障害で、もう一方のCore経由へ自動収束すること。
  • core1停止時もcore2経由でAccess間通信を復旧できること。
  • 障害復旧後にOSPF隣接とECMPが正常状態へ戻ること。

5. 非機能要件 #

5.1 可用性 #

  • 単一のインフラリンク障害でネットワーク全体が停止しないこと。
  • 単一のCore障害でAccess間通信経路が完全に失われないこと。
  • 通信断時間とping欠落数を試験結果へ記録できること。

5.2 運用性 #

  • STPではなく、OSPFネイバー、RIB、CEFの情報で障害を切り分けられること。
  • 機器名、管理IP、データプレーンIP、接続先が文書間で一致していること。
  • 第三者がREADMEから要件、設計、構築、試験の順に追えること。

5.3 学習上の要件 #

  • IOSコンフィグは自動投入しない。
  • startup-configや模範解答を同梱しない。
  • 端末のデータプレーンIPと静的経路も自動投入しない。
  • 学習者自身が設計値をIOS/Linuxコマンドへ変換する。
