2. ライブ状態採取（設定解析より先） #

まず「今どうなっているか」を採る。設定ファイルの読み込みは後回し。

# コンテナ状態
docker ps --format 'table {{.Names}}\t{{.Image}}\t{{.Status}}'
docker logs --tail 50 <container>

# ネットワーク到達
docker exec <src> ping -c3 <dst>
docker exec <src> curl -sv http://<dst>/    # ステータス・ヘッダを見る
docker network inspect zt-dmz               # ゾーン所属・IP

# TLS
docker exec <src> curl -vk https://<iap>/   # 証明書チェーン・handshake
docker exec <src> openssl s_client -connect <iap>:443 -showcerts </dev/null

# 認証（IdP）
docker logs --tail 50 clab-zero-keycloak    # ログイン失敗・realm/client エラー

# 認可（ポリシー）
docker logs --tail 50 clab-zero-pomerium    # allow/deny の判定行

# ログ（SIEM）
docker exec clab-zero-grafana wget -qO- http://loki:3100/ready   # Loki 準備状態

3. 層別切り分け（ZT 6層） #

下から順に埋める。ある層が NG ならその層で止め、上位層は見ない（根本原因が下層にある）。

層 確認内容 確認コマンド/観点 結果 (OK/NG) メモ
L1 コンテナ起動 対象コンテナが Up か・crash ループしていないか docker ps / docker logs の起動完了行
L2 ネットワーク到達 送信元→宛先に L3 到達できるか・ゾーン所属は正しいか ping / docker network inspect
L3 TLS 証明書 handshake 成功か・CA を端末が信頼しているか・SSL bump が成立するか curl -vk / openssl s_client
L4 認証 (IdP) Keycloak がトークンを発行するか・realm/client/redirect が一致するか Keycloak ログ / token endpoint 応答
L5 認可 (ポリシー) IAP が allow/deny を正しく判定するか・posture claim が渡っているか Pomerium/oauth2-proxy ログの判定行
L6 ログ (SIEM) Promtail が収集し Loki に届き Grafana で見えるか Loki ready / Grafana クエリ

層別の読み方:

  • 下層 NG が根本: L2 が NG なら L4/L5 の失敗はその二次症状。まず L2 を直す。
  • TLS は認証の前提: SSL bump や mTLS が成立しないと L4/L5 は評価できない。L3 を先に確定する。
  • ログ層は独立: L6 は他層と独立に壊れうる（Promtail 設定・Loki 未 ready）。他が OK でも L6 単独で NG になりうる。

よくある重複パターン（ZT 特有）:

  • 「未認証拒否されない」= IAP 未経路（L2）＋ ポリシー未設定（L5）の二重。
  • 「ログイン後も app に行けない」= redirect URL 不一致（L4）＋ Trust 側転送経路欠落（L2）の二重。
  • 「ログが出ない」= Promtail のラベル設定（L6）＋ 対象コンテナのログ出力先（L1）の二重。
