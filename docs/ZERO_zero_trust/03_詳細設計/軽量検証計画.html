本計画は L7 トラック（Phase 0-6）の OSS が対象。 NW-ZT トラック（N1-N4: NAC/SDP-ZTNA/NDR/μセグ）の arm64 検証は別計画 軽量検証計画_nwzt に分離している。

検証の目的とスコープ #

  • 目的: OSS 各イメージの arm64 マニフェスト有無 と、要となる機能（起動・SSL bump 等）の実現見込みを確認する。
  • スコープ内: manifest 確認（静的判定）＋ 必要に応じた pull/起動の実検証（軽量）。
  • スコープ外: Phase 1-6 の本格デプロイ・機能結合試験（見込み確認で十分というユーザー決定）。
  • 前提: VM clab（Ubuntu 24.04 / arm64）。IOL 非連携。

判定基準 #

判定 条件 次のアクション
High 公式 arm64 manifest が有る（docker manifest inspect に arm64 が含まれる） 採用確定。実検証は起動確認のみ
Med manifest は有るが起動/機能に不確実性（SSL bump・重量・posture 等） pull して起動・軽量機能確認を実検証
Low arm64 manifest が無い or 明示非対応 代替差し替えルールを適用

manifest 確認コマンド（設計値。実行はT7）:

# arm64 対応の静的確認（イメージを pull せずに判定）

代替差し替えルール #

Low（arm64 非対応）または重量ブロッカーが出た場合の差し替え方針。基本設計書の設計判断（D-2 / D-4）と整合させる。

対象 事象 差し替え 根拠
SIEM Wazuh が arm64 不可 or 重すぎる Loki 一本（Grafana+Loki+Promtail のみ） D-4。SIEM は早期投入が目的で、軽量が優先
WEB squid の SSL bump が arm64 で成立しない mitmproxy 一本化（SWG を mitmproxy に集約） D-2。同一経路で検査・DLP を兼務
WEB IDS suricata が arm64 不可 mitmproxy のログ + Loki 側の検知ルールで代替 IDS 発報の目的（SIEM へ通知）は維持
DLP c-icap + ClamAV が arm64 不可 mitmproxy DLP アドオンに一本化 D-2。DLP は SWG の通信を検査する同一経路上
ID発展 OpenLDAP が不可 Keycloak 内蔵 DB のまま（OpenLDAP 連携は見送り） 発展課題。今回は内蔵 DB で完結
デバイス posture の実収集が難しい posture claim を モック（固定属性で分岐を示す） 要件 FR-2。実装可能性の確認が目的

差し替えの優先原則:

  1. arm64 で動くことを最優先（NFR-1）。動かないものは代替する。
  2. 観点の目的を維持する（例: IDS 不可でも「検知→SIEM 通知」の導線は残す）。
  3. 軽量・集約を選ぶ（重量級は Loki 一本・mitmproxy 一本化へ寄せる）。

検証手順（T7 が実施） #

  1. 上表の各イメージに docker manifest inspect を実行し、arm64 有無を記録する。
  2. Med 判定のものは pull → 起動 → 要機能の軽量確認（Keycloak 起動 / mitmproxy SSL bump / step-ca 発行 等）。
