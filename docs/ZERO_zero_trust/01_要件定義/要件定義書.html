背景・目的 #

商用ゼロトラスト（ZT）製品は個人検証が難しい。代表例と制約は次のとおり。

商用製品カテゴリ 代表製品 個人検証の制約
SSE / IAP Cisco Secure Access, Zscaler ZPA ライセンス契約前提。無償評価も法人審査あり
SWG / CASB Zscaler ZIA, Netskope クラウド配信型で個人テナント不可
NGFW / IAP Palo Alto Prisma Access サブスクリプション必須。x86 前提
EDR / MDM CrowdStrike, Intune 法人契約・実端末管理が前提

そこで本ラボは OSS × arm64 だけでゼロトラストの6観点が成立するかを先行検証する。 狙いは「完成した ZT 環境の構築」ではなく、各観点が OSS で代替可能かの実装可能性を確かめること（PoC の前段）。

  • 位置づけ: ロードマップ PHASE2（テーマ28-44）の前哨。ここで見込みが立った観点を、後続テーマで本格実装する。
  • ユーザー決定: 見込みが立てば実デプロイは不要。今回スコープは設計書一式 ＋ Phase 0 土台 ＋ arm64 軽量検証まで。

対象範囲 #

ゼロトラスト6観点の定義と商用製品対応 #

# 観点 定義（本ラボでの意味） 代替する商用カテゴリ
1 ID統制 誰であるかを IdP で一元検証（SSO / OIDC） IDaaS（Okta, Entra ID）
2 デバイス統制 端末の正当性・状態（posture）を検証 MDM / EPP / EDR
3 NWセキュリティ アプリ手前で認証・認可を強制（IAP） ZTNA / IAP（ZPA 等）
4 WEBセキュリティ 外向き Web 通信を検査・制御 SWG / CASB
5 DLP / IRM 機密データの持ち出しを検知・ブロック DLP
6 SIEM 全観点のログを集約・可視化・相関 SIEM / SOAR

  • 各観点を代替する OSS の実装可能性マトリクスと段階ロードマップ。
  • Phase 0 土台（docker network + multitool client / app / external）の設計。
  • arm64 可用性の軽量検証計画（実施と結果は別タスク）。
  • 各 Phase の試験計画・切り分け手順・構築ログ雛形・Phase 実装スタブ。

  • IOL（IOS-XE エミュレータ）との連携（発展課題として明記）。

機能要件 #

観点ごとに、代替 OSS が満たすべき機能を定義する。詳細な OSS 選定根拠は 実装可能性マトリクス。

FR-1 ID統制（IdP / SSO 代替） #

  • OIDC の認可コードフローで ID トークン・アクセストークンを発行できること。
  • 内蔵ユーザー DB でユーザー・ロールを管理できること（発展: OpenLDAP 連携）。
  • 採用 OSS: Keycloak。

FR-2 デバイス統制（MDM / EPP / EDR 代替） #

  • 端末証明書（mTLS）で「管理された端末か」を検証できること。
  • posture claim（端末状態を表す属性）をモックし、認可判断の入力にできること。
  • 採用 OSS: step-ca（mTLS + posture claim モック）＋ osquery 補助。発展: Wazuh エージェント。

FR-3 NWセキュリティ（IAP / ZTNA 代替） #

  • 保護対象アプリの手前で未認証アクセスを拒否できること。
  • IdP へ認証を委譲し、認可済みリクエストのみ転送できること。
  • 採用 OSS: oauth2-proxy → Pomerium（発展: OpenZiti）。

FR-4 WEBセキュリティ（SWG / CASB 代替） #

  • 外向き Web 通信を proxy 経由に強制できること。
  • SSL bump（TLS 復号）で通信内容を可視化できること。
  • IDS で不審通信を検知し、SIEM に発報できること。
  • 採用 OSS: mitmproxy または Squid（SSL bump）＋ Suricata（IDS）。

FR-5 DLP / IRM 代替 #

  • アップロード内容を検査し、機密キーワードを含むデータの持ち出しを検知・ブロックできること。
  • 検知イベントを SIEM に記録できること。
  • 採用 OSS: mitmproxy Python アドオン（SWG と兼務）。

FR-6 SIEM 代替 #

  • 各観点のログを集約・保存・検索できること。
  • ダッシュボードで「セキュリティ機構が効いている証拠」を可視化できること。
  • 採用 OSS: Grafana + Loki + Promtail（軽量先行）。発展: Wazuh（統合・4GB 級、要検証）。

非機能要件 #

ID 分類 要件
NFR-1 アーキ制約 全コンポーネントが arm64 マニフェストを持つこと。x86 専用は不可
NFR-2 リソース VM 全体のメモリ上限を意識し、共通基盤（mitmproxy/Keycloak/Pomerium）に集約すること。目安は 実装可能性マトリクス のリソース見積を参照
NFR-3 再構築性 deploy.sh と設定ファイルから冪等に再構築できること。手作業の秘伝は残さない
NFR-4 移植性 絶対パス埋め込み禁止・NFC ファイル名・相対リンクで GitHub 移植可能なこと
NFR-5 可観測性 各 Phase の効果がログで観測できること（Phase 3 SIEM を早期投入）
NFR-6 分離 docker bridge でゾーンを分離し、ゾーン間通信は設計した経路のみとすること

ゾーン設計方針 #

4ゾーンに分割し、docker bridge で分離する。詳細は 基本設計書・IPアドレス管理表。

ゾーン 役割 サブネット（予定） 主なコンポーネント
Untrust 信頼しない端末側 172.30.0.0/24 client, external
DMZ（認可層） 認証・認可・検査の関所 172.30.10.0/24 Keycloak, Pomerium, mitmproxy, step-ca
Trust（業務） 保護対象サービス 172.30.20.0/24 app
可観測 ログ集約・可視化 172.30.30.0/24 Loki, Promtail, Grafana

方針:

  • Untrust から Trust への 直接通信は禁止。必ず DMZ（認可層）を経由する。
  • 外向き Web 通信は DMZ の SWG を強制経由する。
  • 全ゾーンのログは可観測ゾーンへ集約する。

制約 #

  • arm64 必須（NFR-1）。arm64 マニフェスト非対応の OSS は代替差し替えルール（軽量検証計画）に従う。
  • IOL 非連携。x86 エミュで重く（1ノード起動 約2分）、ZT の主眼（L7 認可）と噛み合わないため docker network で閉じる。
  • docker network のみで完結。L2/L3 の作り込みは行わない。
  • Phase 1-6 は今回デプロイしない。
