管理サブネット #

プライベート 172.30.0.0/16 をラボ専用に確保し、ゾーンごとに /24 を切る。第3オクテット（10 の位）でゾーンを識別する。

ゾーン bridge 名 サブネット GW 用途
Untrust zt-untrust 172.30.0.0/24 172.30.0.1 端末側。信頼しない
DMZ（認可層） zt-dmz 172.30.10.0/24 172.30.10.1 認証・認可・検査の関所
Trust（業務） zt-trust 172.30.20.0/24 172.30.20.1 保護対象サービス
可観測 zt-obs 172.30.30.0/24 172.30.30.1 ログ集約・可視化

アドレス採番規則:

  • 第4オクテットの 10 番台=ID/認可系、20 番台=IAP/アプリ系、30 番台=検査/CA系、40 番台=SIEM系（ゾーン内で役割を識別）。
  • 名前解決は docker 組み込み DNS（サービス名）を基本とし、IP は補助。
  • Phase 1-6 のアドレスは予定値。実装時に本表を確定値へ更新する。

ホストポート公開（VM から到達させる予定ポート） #

VM clab のホストポートにマップして管理 UI 等へ到達させる。ポート・認証情報は各 Phase 実装時に確定（ログインコマンド.md と同期）。

Phase ノード 予定ホストポート 用途

2 pomerium / oauth2-proxy 実装時に確定 保護対象 app への入口

4 mitmproxy 8081 mitmweb UI
6 step-ca 9000 ACME / 証明書発行

割当ルール（今後の追加ノード向け） #

  • 新ノードはまず所属ゾーンを決め、その /24 内で役割帯（10/20/30/40 番台）から採番する。
  • 1ノードが複数ゾーンに足を持つ場合、主所属ゾーンの IP を代表とし、補助 IP は備考に記す。
  • 予定値から確定値に変わったら、備考の「設計値」を消し、date を更新する。
