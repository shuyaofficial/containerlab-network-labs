N. トラブルシューティングメモ #

  • 問題1: zt-base.clab.yml の links で両方の bridge ノードに eth1 を使ったところ、containerlab がホストのルートネームスペース上でエンドポイント名の重複を検出しエラー（Root network namespace endpoint "eth1" defined by multiple nodes）。

    • 原因: kind: bridge ノードの Link エンドポイント名はホスト側インターフェース名として扱われ、ラボ内で一意である必要がある。
    • 対処: zt-untrust:untrust-eth1 / zt-untrust:untrust-eth2 / zt-trust:trust-eth1 のように bridge ごとに接頭辞を付けて一意化した。

  • 問題2（本質的な設計課題）: client → app の ping が 100% packet loss で失敗（T-0-2 初回失敗）。

    • 原因: 論理構成設計どおり zt-untrust（client/external 所属）と zt-trust（app 所属）を別々の Linux bridge（kind: bridge）として完全分離した。Linux bridge は L2 スイッチに過ぎずルーティング機能を持たないため、別サブネット（172.30.0.0/24 と 172.30.20.0/24）間は経路が存在せず疎通しない。これは設計書の「Untrust→Trust 直通禁止」を意図通り体現した状態だが、Phase 0 のゲート条件（client→app 疎通）と両立しない。
    • 設計判断: 論理構成設計に「Phase 0 では client→app 疎通はゲート確認のための設計上の許可経路」と明記されているため、Phase 0 限定でホスト L3 中継を追加した。具体的には deploy.sh の create_bridges で各 bridge に IP アドレス管理表の GW IP（zt-untrust=172.30.0.1, zt-trust=172.30.20.1）を付与し、net.ipv4.ip_forward=1 を有効化。各ノードの exec に相手ゾーンサブネットへの静的経路（ip route add <相手サブネット> via <自ゾーン GW> dev eth1）を追加した。
    • 該当箇所: deploy.sh の create_bridges()、zt-base.clab.yml の client/app ノードの exec 末尾。
    • Phase 2 以降の扱い: この暫定中継は関所（Pomerium/oauth2-proxy）が入った時点で撤去し、Untrust→Trust は関所経由の転送のみに置き換える前提（試験結果ファイルの「残課題」に記載）。

  • 問題3: deploy.sh destroy 後もホストの zt-untrust / zt-trust ブリッジが残存。

    • 原因: remove_bridges() が BRIDGES 配列（"zt-untrust:172.30.0.1/24" 形式）をパースせず、そのまま ip link show / ip link del に渡していたため対象が見つからず削除に失敗していた。
    • 対処: remove_bridges() にも create_bridges() と同じ ${entry%%:*} パースを追加し、ブリッジ名のみを抽出するよう修正。修正後の destroy で正常にブリッジが削除されることを確認。

5. 学習ポイント #

  • containerlab の kind: bridge は「既存の Linux bridge に接続する」だけの機能で、docker network create のようなサブネット管理・IP 自動割当は行わない。ゾーン間の GW/ルーティングは呼び出し側（deploy.sh や host 設定）が明示的に用意する必要がある。
  • 論理構成設計書が言う「docker bridge によるゾーン分離」を containerlab 上でそのまま再現すると、ブリッジ間は完全に非到達になる。「Untrust→Trust 直通禁止」という本番の目的と、「Phase 0 でゲート確認のため疎通させる」という一時的な要求は、設計時点では両立しないことがあり、実装時に「暫定経路として作り、次 Phase で本来の関所に置き換える」という位置づけを明示する判断が必要だった。
  • bridge kind のリンクエンドポイント名はホストのネームスペースに直接現れるため、他ラボ・他テーマとの名前衝突を避ける接頭辞規約（<bridge名>-ethN 等）が必要。
