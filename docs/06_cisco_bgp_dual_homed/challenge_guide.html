🚀 Challenge 06: BGPデュアルホーム（マルチホーム）冗長化ハンズオン

本ラボは、物理結線のみが完了した「まっさらなルータ5台」を使い、複数のプロバイダ（ISP-A, ISP-B）と接続した企業のインターネット接続冗長化（デュアルホーム）を構築する、超実践的なハンズオン課題です。

BGPのアトリビュート Local Preference（ローカルプリファレンス） を駆使して、社内から出ていくトラフィックを思い通りにコントロールする術を体験しましょう！

2. プロトコル設計 #

  • OSPF (社内AS 65002内: R1, R2, R3):
    • プロセスID: 1
    • エリア: 0
    • 社内のすべての Loopback と社内物理リンク（12.0/24, 13.0/24, 23.0/24）を広報します。
    • 重要: 192.168.11.0/24 や 192.168.22.0/24 などの eBGP 物理リンクは、OSPF に含めないでください！
  • BGP 設定:
    • ISP-A: AS 65001 (R1とeBGPを張る。8.8.8.8 を network 広報)
    • ISP-B: AS 65003 (R2とeBGPを張る。8.8.8.8 を network 広報)
    • 社内 (R1, R2): AS 65002 (Loopback0同士で iBGP を張る。next-hop-self を設定すること！)

🎯 4つのミッション #

Mission 1: 基本インフラの構築（負荷分散の観測） #

  1. 各ルータに基本IPを設定します。
  2. 社内（R1, R2, R3）で OSPF エリア0を開通させ、Loopback0同士で Ping が通ることを確認します。
  3. R1-ISP-A間、R2-ISP-B間で eBGP を確立します。
  4. R1-R2間で iBGP（Loopbackピア、next-hop-self）を確立します。
  5. デフォルトルートの広報（重要）:
    • R3（社内PC）は 8.8.8.8 へのルートを直接BGPで学習しません（R3ではBGPを動かさないため）。
    • そのため、社内境界の R1 と R2 の両方 から、OSPF 内に向けてデフォルトルートを広報します。
      • ヒント: router ospf 1 の中で default-information originate コマンドを設定します。
  6. 観測ポイント:
    • R3 から 8.8.8.8 へ traceroute（またはPing）を実行した際、R1経由とR2経由の2方向（等コストマルチパス）に負荷分散されてパケットが流れる様子を確認します！

Mission 2: Local Preference による「アクティブ / スタンバイ」経路制御 #

企業ポリシー 「通常時はすべて ISP-A（R1）をメインにし、ISP-B（R2）は完全なバックアップ（スタンバイ）にする」 を満たす制御を実装します。

  1. ルートが社内（AS65002）に入ってきた後の優先度を決定する Local Preference（初期値100） を変更します。
  2. R2（境界2） において、ISP-B（192.168.22.100）から受け取るルートの Local Preference を 50（デフォルトより低い値） に下げて受信します。
    • やり方のヒント:
      • R2で route-map SET_LP permit 10 を作成し、その中で set local-preference 50 を設定します。
      • BGP設定で、対向ISP-Bのneighborに対し、このroute-mapを in（受信方向） で適用します。
      • 設定後、ソフトクリア（clear ip bgp * soft）を実行します。
  3. 観測ポイント:
    • 設定後、R3 から 8.8.8.8 へのすべてのトラフィックが、狙い通り R1 経由（ISP-A）に一本化 されることを traceroute で目撃します！

Mission 3: 自動フェイルオーバーの観測 #

  1. R3 から 8.8.8.8 に向けて、継続的な Ping（ping 8.8.8.8 repeat 100 など）を実行します。
  2. その状態で、R1 の Ethernet0/1（ISP-A接続ポート）を shutdown し、メイン回線をわざと断線させます。
  3. 観測ポイント:
    • メイン回線が切れた瞬間、BGPがそれを検知して自動的にバックアップである R2（ISP-B経由）に切り替わり、Ping がほとんど途切れずに通信が継続するダイナミックなフェイルオーバーの瞬間を観測します！

Mission 4: 【プロフェッショナル編】OSPFコスト制御とIP SLA/Trackによる完全冗長化 #

Mission 3までの状態では、通常時にR3から見たデフォルトルートがR1とR2で等コスト（負荷分散）になっているため、R2へ送られたパケットがR1へ戻されるという「不要な遠回り（ヘアピン）」が発生してしまいます。また、ISP-Aが落ちた際もR1は無駄にデフォルトルートを社内に広報し続けてしまいます。 これを解決する実務レベルの「完全なアクティブ・スタンバイ設計」を実装しましょう。

  1. OSPFデフォルトルートのメトリック（コスト）制御:
    • 通常時はR3からのすべてのパケットがR1へダイレクトに流れるよう、OSPFのデフォルトルートにメトリックを設定します。
      • R1（メイン）: メトリック 10
      • R2（バックアップ）: メトリック 100
  2. IP SLA ＋ Track ＋ スタティックデフォルトルートによる自動停止制御:
    • メイン回線（R1-ISP-A間）が落ちた際、R1がデフォルトルートの広報を自動的に止める（R2へ切り替えさせる）仕組みを作ります。
    • R1において：
      • IP SLAを作成し、ISP-A（192.168.11.100）へ定期的にPingを打つ監視センサーを仕込みます。
      • そのIP SLAを監視する Track を作成します。
      • ISP-A（192.168.11.100）向きのスタティックデフォルトルート（0.0.0.0/0）に、この Track を紐付けます。
      • OSPFのデフォルトルート広報設定を、素の default-information originate metric 10 にします（これでR1自身のテーブルからスタティックデフォルトルートが消えると、OSPFも自動的に広報を止めます）。
  3. 検証とクリア条件:
    • 通常時: R3 からの traceroute 8.8.8.8 が R1（192.168.13.1）経由で最短2ホップで進むこと。
    • 障害時: R1の Ethernet0/1 を shutdown した際、R1からのデフォルトルート広報が自動的に消え、R3からの traceroute が R2（192.168.23.2）→ ISP-B（192.168.22.100）経由へ最短2ホップで自動切り替えされること！
