NW-ZT トラック N4 の実機実装（nftables/IOL VLAN・ACL 版）。Cisco IOL の VLAN + inter-VLAN ACL（層1）とホスト nftables（層2）で「2層マイクロセグメンテーション」を実機実証した。

到達点（実証済み） #

  • 層1（IOL inter-VLAN ACL）: VLAN10（pc10a/pc10b）→ VLAN20（srv20）の到達を tcp/80 のみ許可、tcp/22 は拒否。ACL マッチカウンタで実測（permit tcp ... eq www = 6 matches、deny tcp ... eq 22 = 1 match）。
  • 層2（ホスト nftables）: 同一 VLAN10 内の pc10a→pc10b 通信を、pc10b の nftables で ip saddr 172.50.10.11 drop。適用前=3/3到達、適用後=0/3到達（100%損失）、nft カウンタで 3 packets drop を実測。
  • 2層の役割分担を対照実証: 同一 VLAN 内の横移動は VLAN/ACL では止められない（そもそも L3 ACL は VLAN 間の SVI にしか適用していないため、同一 VLAN 内の pc10a→pc10b はスイッチのルーティングを経由せず ACL の対象にならない）。ホスト側 nftables を入れて初めて止まることを、適用前後の対照で示した。

詰まりどころと解決 #

1. iouyap 未起動によるデータプレーン全断（既知の落とし穴、想定どおり回避） #

  • N1（テーマ31）・テーマ22/23/24/25/26/27 と同じ、clab の cisco_iol は iouyap がデータプレーンを中継する方式。deploy.sh deploy に start_iouyap()（ポート 513、全テーマ共通実績）を組み込み、containerlab deploy 直後に必ず起動する運用にした。
  • 確認: sudo docker top clab-microseg-sw1 | grep iouyap で /usr/bin/iouyap /usr/bin/iouyap 513 プロセスを確認。1回で疎通成功（iouyap 起動後の追加調査は不要だった）。

2. SVI の一時的な down 表示 #

  • スイッチ設定投入直後の show ip interface brief で Vlan10/Vlan20 が down/down と表示されたため、SVI が上がっていない疑いで詳細調査した（show vlan brief、show interfaces status、show interfaces Vlan10 を追加確認）。
  • 結果、実際には Et0/1-3 は connected、Vlan10/Vlan20 は up, line protocol is up（show interfaces Vlan10 で確認）で、直後の show ip interface brief の表示が反映のタイミング差で古かっただけだった。再度 show ip interface brief を実行すると up/up に変わっていることを確認。実害なし。

3. expect の attach が別コマンドとの並列実行で失敗 #

  • 検証中、rm と ssh expect を同一メッセージ内で並列に投げたところ、sudo expect -f verify_microseg.exp ... が spawn 直後に exit 255 で終了し、ログファイルすら生成されなかった。
  • 原因は複数の ssh clab@orb セッションが同時に走ったことによる docker attach の競合と推測（31_nac_dot1x の教訓どおり、attach の多重化は避けるべき）。
  • 対策: 以降は 1メッセージ1 attach の単発実行に限定し、pgrep -f 'docker attach' で残存確認してから次を実行。再実行後は問題なく成功した。

4. ACL の eq 80 はコンソール表示上 eq www に正規化される #

  • show ip access-lists で permit tcp ... eq 80 は IOS の仕様で well-known port 名 www に自動変換されて表示される（設定自体は問題なく反映）。cfg 投入時のコンソールログで長い行が eq 8$ のように途中で折り返されて見えたのも表示上の癖で、実際の ACL 内容は正しく登録されていた（show ip access-lists で確認）。

再現手順（後から自分で構築する場合） #

  1. 04_構築/endpoint/ の Dockerfile をビルド（microseg-endpoint:local。deploy.sh が自動実行）。
  2. ./deploy.sh deploy（image build → clab deploy → iouyap 起動 → オフロード無効化 → 端末 IP/サービス設定まで自動）。
  3. sudo expect run_microseg.exp clab-microseg-sw1 sw1_microseg.cfg（./deploy.sh config でも同じ）でスイッチに VLAN/SVI/ACL 設定を投入。
  4. ./deploy.sh test で G2（inter-VLAN ACL）と G3 適用前の疎通を確認。
  5. ./deploy.sh nft で pc10b にホスト nftables（層2）を投入。
  6. ./deploy.sh test-g3-after で G3 適用後の遮断を確認（適用前後の対照）。
  7. sudo expect verify_microseg.exp clab-microseg-sw1 で ACL マッチカウンタを採取。
  8. 片付け: ./deploy.sh destroy。

環境情報 #

  • OrbStack VM（arm64）、containerlab 0.77、docker 29.1.3。
  • イメージ: vrnetlab/cisco_iol:L2-advipservices-2017（L3 SVI + inter-VLAN ACL に対応、22/23 で実績あり）、microseg-endpoint:local（ubuntu:24.04 + nftables/iproute2/iputils-ping/curl/ncat/python3 を自作）。
  • 既存稼働中の clab-dynamic-routing-lab-* には無関係（clab-microseg- 接頭辞・専用 mgmt ネットワーク clab-microseg-mgmt 172.20.30.0/24 で完全分離、衝突なしを事前に確認）。

残課題 #

  • L2 ACL / PVLAN による同一 VLAN 内制御との比較: 本ラボはホスト nftables で層2を実現したが、IOL の Private VLAN（PVLAN）や port-based ACL で同一 VLAN 内を制御できるかは未検証（IOL の PVLAN サポート状況次第）。
  • 複数端末・複数ルールでのスケール検証: 今回は pc10a→pc10b の1方向・1ルールのみ。実運用の Illumio 的な「多対多の許可リスト管理」の複雑さは未検証。
  • nftables ルールの永続化: 今回は docker exec でランタイムに投入したのみ（コンテナ再起動で消える）。永続化する場合は Dockerfile 側で nftables.conf を焼き込むか、systemd 相当の仕組みが要る（本ラボの検証目的では対象外とした）。
