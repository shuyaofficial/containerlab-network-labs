テーマ13: ACL / NAT / Port Security 構築ログと教訓

本ドキュメントは、セキュリティおよびアドレス変換機能（ACL / NAT / Port Security）構築時のトラブルシューティングノウハウ、および実務上の教訓を記録する。

1. NAT (PAT) における Inside / Outside の重要性 #

  • 概念: ルーターが「どのパケットを変換対象とするか」を認識するためには、インターフェースに対する ip nat inside（社内側）と ip nat outside（インターネット側）の明示的な宣言が必須である。
  • 動作: 内部（inside）からやってきたパケットの送信元IPが、定義したACL（許可リスト）に合致する場合のみ、外部（outside）のインターフェースが持つグローバルIPに書き換えられ（Overloadでポート変換も実施）、インターネット上のサーバーとの通信が可能となる。

2. 拡張ACLにおける「暗黙のdeny」と「ログ(log)オプション」の役割 #

  • 事象: 外部（SRV1）から内部のエッジルーター（R1）に対するTelnet（ポート23）アクセスを禁止する際、単に防ぐだけであれば permit icmp any any echo-reply だけを書き、残りは「暗黙のdeny」に任せればパケットは破棄される。
  • 実務上のテクニック: あえて deny tcp any any eq 23 log を明記することで、パケットを破棄すると同時に %SEC-6-IPACCESSLOGP というセキュリティログを出力させることができる。これにより「いつ、どこから、何を狙った攻撃が来ているか」を検知できる。
  • カウンターの可視化: さらに最後に deny ip any any を明記することで、暗黙のdenyで静かに捨てられる通信の数をヒットカウンター（show access-lists）として可視化し、想定外の通信ボリュームを把握できる。

3. L2/L3スイッチとしての端末の振る舞い（ip routingの影響） #

  • 教訓: PCの代わりにルーターイメージ（IOL）を使用する場合、no ip routing を設定することで完全に「L2ホスト（PC）」として振る舞わせることができる。この際、外の世界へのルートを示す設定は ip route 0.0.0.0 0.0.0.0（ルーティングテーブル用）ではなく、ip default-gateway（ホスト用）を使用しなければならない。これを混同するとパケットが中継されず通信不可となる。

4. Port Security による未許可MACアドレスの物理遮断 #

  • 動作: スイッチのアクセスポートに対して switchport port-security mac-address sticky を設定することで、最初に通信したPCのMACアドレスを自動学習・記憶させることができる。
  • 防御機能: 登録されていない別のPC（偽装されたMACアドレス等）が接続された瞬間、スイッチはそれを不正アクセスと見なし、psecure-violation エラーによってポートを強制的に err-disable（物理的シャットダウン）状態にする。
  • 復旧の儀式: err-disable 状態のポートは、不正PCを取り外しただけでは復旧しない。管理者が明示的に shutdown コマンドを実行して反省（確認）を示したのち、再度 no shutdown を実行しなければ再利用できない仕様となっている。
