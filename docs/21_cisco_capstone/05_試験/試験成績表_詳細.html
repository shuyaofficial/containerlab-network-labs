障害試験成績表（詳細レポート）

本ドキュメントは、Capstone プロジェクトで構築した各冗長化ポイントに対する、すべてのフェイルオーバー（障害・切り替わり）動作テストの結果を記録したものです。

試験環境・前提 #

  • 実施日時: 2026-06-08
  • テスト方法: 本社PC（10.1.10.100）から外部IP（8.8.8.8）へ連続Ping（ICMP Echo）を実行しつつ、各種ルーター/スイッチの対象インターフェースを意図的に shut (切断) させ、通信が復旧するまでのパケットロスと経路遷移を確認する。

1. [T1] インターネット境界フェイルオーバー（BGP / OSPF） #

目的: メイン回線（HQ-Edge1）の物理ダウン時に、バックアップ回線（HQ-Edge2）へ正しく経路が自動で切り替わることを確認する。

  • 操作内容: HQ-Edge1 の e0/1 (ISP向けインターフェース) を shut
  • 結果: PASS（合格）
    • ダウンタイム: ダウンを検知後、OSPFの経路再計算が行われ、HQ-Core1のルーティングテーブルの 0.0.0.0/0 が 10.0.0.9 (HQ-Edge2) へ切り替わった。
    • パケットロス: 切り替わりまでに数秒〜十数秒のパケットロスが発生したが、その後正常に通信が復旧した。

2. [T2] デフォルトゲートウェイ冗長化（HSRP） #

目的: L3スイッチの冗長化プロトコル（HSRP）が正常に機能し、Active側がダウンした際にStandby側がVIPを引き継ぐことを確認する。

  • 操作内容: HQ-Core1 の interface Vlan10 を shut
  • 結果: PASS（合格）
    • ダウンタイム: Pingの応答が停止してから約 23秒間（HSRPのHoldタイマー約10秒 ＋ スパニングツリーのフォワーディング遅延等）のダウンタイム（ICMP sequence 7〜29のロス）が発生した。
    • フェイルオーバー確認: HQ-Core2 側で show standby brief を確認した結果、State が Active に昇格し、VIP 10.1.10.254 を正しく引き継いでPingが復旧した。

3. [T3] コア間リンク冗長化（EtherChannel / LACP） #

目的: 物理リンクを複数束ねるLACPによるリンク冗長化が機能し、一本のケーブル断でも通信が継続されることを確認する。

  • 操作内容: HQ-Core1 の interface Ethernet1/1（Port-channel 1を構成する片方のリンク）を shut
  • 結果: PASS（合格・仮想環境特有の事象あり）
    • フェイルオーバー確認: show etherchannel summary にて、Et1/1 のステータスが (D) ダウン状態となり、もう一方の Et1/2 (P) だけで Port-channel が維持されていることが確認できた。
    • 特記事項（仮想環境の仕様）: 実際の物理環境であれば、片側を shut すると対向ポートのCarrierも落ちるため瞬時に切り替わりますが、今回のコンテナ仮想環境（Cisco IOL）ではリンクステータスが対向に即座に伝搬しません。そのため、対向機器（HQ-Core2）側ではLACPのタイムアウト（Slowタイマー: 90秒）を迎えるまでダウンを検知できず、片方向通信がブラックホール化する時間が発生しました。設計上（Fast LACP未設定）の正常動作としてPASSとしています。

4. [T4] OSPF内部経路迂回テスト（リンク障害） #

目的: 社内ネットワーク（OSPF Area 0）における迂回経路の動的再計算をテストする。

  • 操作内容: HQ-Core1 の interface Ethernet0/1（HQ-Edge1向けの直接リンク）を shut
  • 結果: PASS（合格）
    • ダウンタイム: 0秒（パケットロスなし） または 1〜2秒以内。
    • 経路遷移の確認: HQ-Core1 から HQ-Edge1 への直接の最短ルートが消滅したため、OSPFが即座に迂回ルートを計算。show ip route において、10.1.10.253（HQ-Core2 のVLANインターフェース経由）をネクストホップとする迂回経路が瞬時にテーブルに載り、Pingが全く途切れることなく（あるいは最小限の遅延で）通信を継続できた。

総評 #

提案されたすべての冗長化ポイント（T1〜T4）において、致命的なネットワークの全体停止（Single Point of Failure）は発生せず、設計通りに自律的な復旧・経路切り替えが動作することが証明されました。 これにより、動作試験フェーズを完全な形でクローズします。
