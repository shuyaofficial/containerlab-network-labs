要件定義書（Requirements Document）— テーマ22: 大規模キャンパス社内LAN

1. プロジェクトの目的 #

株式会社XXX（架空・社員約400名）は事業拡大により本社を2棟構成のキャンパスへ移転する。 テーマ21（本社+支社の2拠点ネットワーク）で構築した「止まらない・守られる・運用できる」設計思想を維持したまま、 規模が拡大しても破綻しない3階層キャンパスネットワークへスケールアップすることが本プロジェクトの目的である。

テーマ21との違い（本プロジェクトの新規論点）:

  • 規模拡大に耐える 階層化（コア/ディストリビューション/アクセス）
  • 棟単位の OSPFマルチエリア化と経路集約（コアの経路数を増やさない）
  • VLAN数増加に耐える MST（Multiple Spanning Tree）
  • 境界FWの 専用アプライアンス化（FortiGate） と マルチベンダー相互接続

セキュリティとトラフィック管理の観点から、以下のセグメントに分割する。

セグメント 所属棟 主な業務
営業部 A棟 インターネットアクセス・社内ポータル・ファイルサーバ利用が主体

サーバ群 B棟 社内ポータル（Web）、ファイルサーバ等の社内システムを収容

2.3 通信要件（誰がどこへアクセスできるか） #

# 送信元 宛先 方針
1 営業部・開発部・支社 インターネット 許可（境界でNAT変換）
2 営業部・開発部 サーバ群（ポータル/ファイルサーバ） 許可
3 営業部 ⇔ 開発部 相互の直接通信 制限（開発部の機密保護のため）
4 支社 本社サーバ群 許可（ただし拠点間は必ず暗号化）
5 支社 本社の開発部など業務上不要なセグメント 拒否
6 インターネット 社内（全セグメント） 拒否（戻り通信を除く）

3. 非機能要件 #

3.1 可用性（SLA） #

  • A棟（オフィス棟）: 単一の機器・ケーブル故障で業務通信が停止しないこと（ディストリビューション層の冗長化、SPOF排除）。
  • コア層: 2台構成とし、片系停止でも全棟間の通信が維持されること。
  • デフォルトゲートウェイ: 端末から見たGWを仮想化し、GW機器の障害時に自動切替されること。
  • 上位リンク障害との連動: ディストリビューション機器の上位リンクが全断した場合、GWの切替が連動して行われること。

3.2 拡張性（本テーマの主役） #

  • 棟・フロア・VLANの追加が、コア層の設計変更なしに行えること。
  • 棟内のセグメント増加がコア層の経路テーブルを肥大化させないこと（棟単位の経路集約）。
  • IPアドレス体系は「棟 → 用途(VLAN)」の階層で集約可能なように設計すること。
  • VLAN数が増えてもSTPの計算・運用負荷が増えない方式（インスタンス集約）を採用すること。

3.3 セキュリティ #

  • インターネット境界には**専用ファイアウォールアプライアンス（FortiGate）**を配置し、ステートフルなポリシー制御とNATを行うこと。
  • 拠点間（本社〜支社）の通信はインターネット経由のため、IPsecによる暗号化を必須とする。 本社側FW（FortiGate）と支社側ルーター（Cisco）のマルチベンダー構成で相互接続できること。
  • 部署間のアクセス制御（営業⇔開発の制限）を社内L3境界で実施すること。
  • アクセス層では不正端末の接続対策（ポートセキュリティ、未使用ポートの閉塞）を行うこと。

3.4 運用・保守性 #

  • 全ネットワーク機器に管理用の安定したアドレス（Loopback）を付与し、帯域内から管理アクセスできること。
  • B棟に運用管理セグメントを設け、将来の監視基盤（SNMP/Syslog集約・テーマ16の応用）を収容できる設計とすること。
  • 設計ドキュメント（基本設計書・IPアドレス管理表・構成図）だけで第三者が同一環境を再構築できること。

4. 制約事項（ラボ環境） #

  • 仮想環境（containerlab on OrbStack / arm64）のため、縮小モデルで構築する:
    • ネットワーク機器は11台（コア2・A棟ディストリ2・B棟ディストリ1・アクセス3・FortiGate1・ISP1・支社1）+ 軽量エンドポイント5台を上限とする。
    • B棟ディストリビューションは本来冗長ペアだが、ラボでは単機とする（本番設計との差分は基本設計書に明記する）。
    • 「縮小モデルで原理を観測し、設計書では本来規模を表現する」二段構えとする。
  • 使用イメージ: Cisco IOL（ルーター/L2/L3SW）、FortiGate 7.2（検証済み・メモリ2GB）、エンドポイントは wbitt/network-multitool（切り分け/containerlab_iol_quirks.md のAlpine問題回避のため）。
  • メモリ合計が約13GBに達するため、構築はフェーズごとの段階デプロイを許容する。

5. 将来要件（本プロジェクトのスコープ外・設計上の考慮のみ） #
