試験計画書: QoS/ACL/route-map応用

本試験は、設定した各種セキュリティポリシー（ACL）や品質保証（QoS/PBR）が要件通りに動作していることを確認するためのものです。

1. ゾーン間ACLの確認 (Mission 1) #

  • [ ] hq-pc から isp (10.27.255.100) へ ping が通ること（Trust→Untrust許可）。
  • [ ] hq-pc から dmz-srv (10.27.30.10) の TCP 80 / 443 へのアクセスが許可されること（curl または telnet で確認）。
  • [ ] isp から hq-pc (10.27.10.10) への ping が hq-edge で拒否（ドロップ）されること（Untrust→Trust拒否）。
  • [ ] isp から dmz-srv の TCP 80 へのアクセスが許可され、ping(ICMP)が拒否されること（Untrust→DMZ制限）。
  • [ ] dmz-srv から hq-pc への通信開始が拒否されること（DMZ→Trust拒否）。

2. ポリシーベースルーティング (PBR) の確認 (Mission 2) #

  • [ ] hq-voip (10.27.50.10) から traceroute を実行し、通常のルーティングテーブルとは異なる、PBRで指定したネクストホップ（例: 10.27.1.1）を経由しているか確認する。
  • [ ] hq-dist で show route-map コマンドを実行し、PBRのmatchカウントが増加しているか確認する。

3. DSCPマーキングの確認 (Mission 3) #

  • [ ] hq-voip から継続的にpingを実行する。
  • [ ] パケットキャプチャ（docker exec -it <container> tcpdump 等）でIPヘッダを確認し、TOS/DSCPフィールドが EF (0x2E または 46) にセットされているか確認する。

4. キューイングと優先制御の確認 (Mission 4) #

  • [ ] hq-edge の WAN側インターフェースで show policy-map interface を実行する。
  • [ ] LLQ（Priority Queue）にトラフィックが分類され、パケットカウントが増加しているか確認する。
  • [ ] 大量のビジネストラフィック（iperf等）を同時に流した際に、音声トラフィック（EF）のキューが確保され、他のクラスでドロップ（Tail Drop/WRED）が発生していることを確認する。

5. 経路広告制御の確認 (Mission 5) #

  • [ ] isp ルータで show ip bgp を実行し、br-edge (AS 65001) から 10.27.20.0/24 の経路のみを受信していること（それ以外の本社側IPなどが漏れていないこと）を確認する。

6. 総合試験 (Mission 6) #

  • [ ] 全ての設定を入れた状態で、ルーティングの障害（リンクダウン）を発生させ、迂回後もPBRやQoS、ACLが正しく適用され続けることを確認する。
