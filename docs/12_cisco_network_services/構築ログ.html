テーマ12: DHCP / DNS / NTP 構築ログと教訓

本ドキュメントは、ネットワークサービス（DHCP / DNS / NTP）構築時に発生したトラブル、切り分け手法、および実務上の教訓を記録する。

1. L2イメージにおけるDHCPクライアント機能の制限 #

  • 事象: クライアントPCとして動作させたCisco IOL L2-15.2 スイッチにて、VLAN 10（SVI）に対して ip address dhcp を設定しようとしたところ、dhcp オプションが存在せずエラーとなった。
  • 原因: 該当の古いL2スイッチイメージでは、SVIにおけるDHCPクライアント機能がサポートされていないため。
  • 解決策: PCの物理ポート（Ethernet0/1）を no switchport コマンドでルーテッドポート化し、そこに対して ip address dhcp を設定することで回避した。対向のL3スイッチ側は通常通り switchport access vlan 10 のままで、DHCPリレー検証が正常に機能することを確認した。

2. DHCPリレーにおける「戻りルート」の罠 #

  • 事象: PCから送信されたDHCP Discoverが、リレーエージェント（SW1）経由でDHCPサーバー（SRV1）に届いているにもかかわらず、PCにIPアドレスが割り当てられなかった。
  • 原因: DHCPサーバー（SRV1）のルーティングテーブルに、PC側のネットワーク（10.10.10.0/24）へ向かうルートが存在しなかった。そのため、SRV1が割り当てたIP情報をPCへ返すためのDHCP Offerパケットが破棄されていた。
  • 解決策: SRV1にスタティックルート（ip route 10.10.10.0 255.255.255.0 10.1.1.2）を追加した。DHCPサーバーには、リレーエージェントの配下にあるクライアントネットワークへのルーティングが必須である。

3. L3スイッチの「ip routing」無効化によるパケットドロップ #

  • 事象: DHCPでIPを取得したPCから、SRV1（10.1.1.1）宛てのDNSクエリやPingが失敗した（timeout）。
  • 原因: L3スイッチ（SW1）のデフォルト設定でルーティング機能（ip routing）が無効になっていた。SW1は自身のMACアドレス宛に届いたPCからのPingパケットを受け取ったものの、ルーティング機能がオフのため「自分宛てではないパケット」として破棄していた。（※DHCPリレーはコントロールプレーン通信のため、SW1自身のCPUから送信されて奇跡的に成功していた）
  • 解決策: SW1のグローバルコンフィギュレーションモードで ip routing を有効化することで、データプレーンのルーティングが正常に機能し、Pingが疎通した。

4. NTPの初回同期遅延仕様 #

  • 事象: SRV1（NTPサーバー）とSW1（NTPクライアント）の設定を正しく行ったにもかかわらず、SW1の show ntp associations で reach が 0 のまま同期（* マーク）されなかった。
  • 原因: Ciscoルーターを ntp master に設定した場合、サーバー自身の内部時計（127.127.1.1）が同期状態（Clock is synchronized）になるまで数分間の準備期間を要し、その間はクライアントからのNTP要求を無視する仕様があるため。
  • 解決策: 設定ミスではないため、数分間待機する。reach の値が 0 → 1 → 3 → 7 ... とシフトレジスタで増加し始めれば、正常に同期シーケンスに入ったと判断できる。
