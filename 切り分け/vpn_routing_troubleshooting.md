# VPNトラブルシューティング記録：非対称ルーティングと優先度問題

## 事象
サイト間VPN（IKEv2）の構築時、IPsecトンネルのステータスが `IN-NEG` から進まない状態に陥った。
Cisco（br-edge）からFortiGate（fgt-edge）宛のPingが通らず、通信が途絶している状態だった。

## 切り分け手順
「一番近いところから順番にPingを打って犯人をあぶり出す」というネットワークの基本に則り、以下のテストを実施。

1. `ping 200.0.3.254`（ISPのBranch側インターフェース） -> `!!!!!` (SUCCESS)
2. `ping 200.0.1.254`（ISPのHQ側インターフェース） -> `!!!!!` (SUCCESS)
3. `ping 200.0.1.1`（FortiGateのWANインターフェース） -> `.....` (FAIL)

### 結論
ISP内でのルーティング（1 -> 2）は正常に機能している。
問題は2から3の間、つまりFortiGateの応答にあることが特定された。

## 根本原因
FortiGateのポート1（管理用）がDHCPでIPを取得する際、同時に**「優先度（Administrative Distance）5」**のデフォルトルートを取得していた。
手動で設定したポート2（WAN）向けのデフォルトルートは優先度がデフォルトの「10」だったため、FortiGateは応答パケットをすべて優先度の高いポート1から送信し、管理ネットワーク内で破棄されていた（非対称ルーティング）。

## 解決策
手動で設定したスタティックルートの優先度（Distance）を「1」に変更し、DHCPで取得したルートより優先されるように修正した。

```text
config router static
    edit 1
        set distance 1
    next
end
```
これにより、Pingが `!!!!!` となり、通信経路が正常に開通した。
