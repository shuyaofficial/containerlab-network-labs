1. 本書の位置づけ#

基本設計書はゾーン構成と「何を・どこで・なぜ制御するか」という方針を示す。本書はその方針を、機器・プロトコルの動作仕様レベルまで落とし込み、「その設定でなぜ意図通りに動くのか」を解説する。

設定値そのもの（IP・ACL文・route-map句の一字一句）はパラメータシートを正とし、本書では重複記載を避けて動作原理の説明に専念する。実際に投入するコマンド列と完成形はコンフィグ解答編、投入手順の流れは作業手順書（MOP）を参照する。

文書役割
基本設計書ゾーン構成・制御方針（何を・どこで）
本書（詳細設計書）動作原理・評価順序・処理順序（なぜそう動くか）
パラメータシート確定値そのもの（IP・ACL文・route-map句）
コンフィグ解答編完成running-config相当（参考解）

2. ACL詳細設計#

2.1 named ACLの評価順序と適用位置

Cisco IOSのnamed ACLは、記述した行を上から順に評価し、最初にマッチした行で処理が確定する（以降の行は評価されない）。どの行にもマッチしなければ、明示していなくても末尾にdeny ip any anyという暗黙denyが存在し、パケットは破棄される。したがって「許可したい通信を先に、狭い条件から広い条件へ」書くのが定石であり、本設計もこの順序に従う。

適用位置は「境界に最も近いインバウンド」に統一する。不要な通信をネットワークへ入れる前に落とすことで、以降のルーティング・NAT・QoS処理の負荷を最小化できるためである。本設計ではhq-edge Ethernet0/2（Untrust境界）とhq-dmz Ethernet0/2（DMZ境界）のインバウンドにACLを適用する。

2.2 OUTSIDE-IN（hq-edge Ethernet0/2 in）

ISP側（Untrust）からHQへ向かう通信を制御する。ゾーン間ポリシーの「Untrust→DMZ: 80/443許可」「Untrust→Trust: 拒否（戻り通信のみ許可）」を1本のACLで実現する。

順序ACL文意図
10permit tcp any host 10.27.30.10 eq 80Untrust→DMZの HTTP を許可（dmz-srv宛のみに限定し、他のTrust宛ては一切許可しない）
20permit tcp any host 10.27.30.10 eq 443同上のHTTPS
30permit icmp any any echo-replyTrust側が送信したpingへの応答のみを許可。ICMPはステートレスで established が使えないため、応答方向（echo-reply）に限定することで実質的に「発信は許可・着信は拒否」を実現する
40permit tcp any any establishedTrust発のTCPセッションの戻りパケット（ACK/RSTが立った以降のセグメント）のみ許可。外部からの新規セッション開始（SYNのみ）はマッチせず暗黙denyへ落ちる
(暗黙)deny ip any any上記以外は全拒否。Untrust→Trustの新規通信を遮断する最終防衛線
i
学習上の割り切り: 宛先NATを設けない理由本ラボはクローズドなトポロジであり、dmz-srvの公開用静的NAT（グローバルIP↔10.27.30.10）は構成しない。そのためOUTSIDE-INは宛先をプライベートIPのまま直接記述している。実運用では、外部から見えるのは公開グローバルIPであり、インバウンドACLは静的NAT変換前のグローバルアドレスに対して評価される点に注意する。

2.3 DMZ-OUT（hq-dmz Ethernet0/2 in）

dmz-srvが送信元となりhq-dmzへ入ってくる通信（DMZ→内向き）を制御する。ゾーン間ポリシーの「DMZ→Trust: 全拒否」を実現する。

順序ACL文意図
5permit tcp host 10.27.30.10 eq 80 anyWebサーバ応答（送信元ポート80）を先に許可。ACLはステートレスのため、これが無いとTrust発HTTP の SYN-ACK が下のdenyで落ち、要件（Trust→DMZ 80/443許可）が成立しない（ISS-13）。establishedでなくポート限定にするのは、SSH等の戻りまで通してしまい「SSH禁止」要件が破れるため
6permit tcp host 10.27.30.10 eq 443 any同HTTPS応答
7permit icmp host 10.27.30.10 10.27.10.0 0.0.0.255 echo-replysrv発の echo-reply を宛先10.27.10.0/24に限り許可。pingは行き=echo/返り=echo-replyの非対称を利用し「PC→srvの片方向ping」を実現（srv発のecho開始はdenyのまま・ISS-14）
10deny ip 10.27.30.0 0.0.0.255 10.27.10.0 0.0.0.255DMZ→HQ Trust(VLAN10)を拒否。DMZが侵害されても本社LANへ横展開させない
20deny ip 10.27.30.0 0.0.0.255 10.27.20.0 0.0.0.255DMZ→支社Trust(VLAN20)を拒否。OSPFで到達可能な経路をACLで明示的に遮断する
25deny ip 10.27.30.0 0.0.0.255 10.27.50.0 0.0.0.255DMZ→VoIPセグメント(VLAN50)を拒否。Trustゾーンは10/20/50の3サブネットであり、列挙漏れは即セキュリティホールになる（ISS-12で追補）
30permit ip any any上記以外（Untrustへの戻り通信など）は許可。DMZサーバの正常な応答動作を妨げない

Trust→DMZ（80/443許可）とTrust→Untrust（全許可）は、経路上にこれを明示的に拒否するACLが存在しないため、デフォルトで許可される（積極的なpermitを書く必要はない）。

3. PBR詳細設計#

3.1 インバウンドPBRと通常ルーティングの処理順序

インターフェースにip policy route-mapを適用すると、そのインターフェースで受信したパケットは、通常の宛先ベースルーティング（ルーティングテーブル参照）より先にroute-mapで評価される。route-mapのいずれかのpermit句にマッチし、かつset句が有効であれば、その指示（ネクストホップ強制等）に従って転送される。マッチしなければ通常のルーティングにフォールバックする（詳細は3.3）。

3.2 route-map PBR_VOICEの構成

hq-dist Ethernet0/3（hq-voip側）のインバウンドに適用する。

句設定動作
matchmatch ip address VOIP-SRCACL VOIP-SRC（permit ip 10.27.50.0 0.0.0.255 any）に一致する送信元（hq-voip）からのパケットを対象にする。route-mapのmatchに使うACLの permit/deny は「この後のset句を適用するか」を意味し、通常のACLのようにパケットそのものを許可・拒否するわけではない
setset ip next-hop 10.27.1.1再帰的な経路計算を行わず、直接到達可能なネクストホップ（hq-core Ethernet0/1）へ強制転送する
!
DSCPマーキングはPBR route-mapに含めないパラメータシート.mdはこのroute-map内に set dscp ef を併記しているが、PBR用route-mapのDSCP設定は本来 set ip dscp 構文であり、これ自体をサポートしないIOSも多い（IOL 15.7.3M2でも弾かれる可能性が高い）。そのため本設計・コンフィグ解答編では、DSCPマーキングをPBRから切り離し、同じEthernet0/3インバウンドに別建てのMQC入力ポリシーとして実装する（4.1・4.2参照）。対応が確認できる環境では、route-mapへset ip dscp efを統合する構成も選択できる。

3.3 PBR未マッチ時の挙動

route-mapのどのpermit句にもマッチしないパケットは、ACLの暗黙denyのように破棄されるわけではない。PBRが適用されなかったものとして、通常のルーティングテーブル参照（本設計ではOSPFで計算された経路）に従って転送される。これはACLの「マッチしなければ破棄」という挙動との重要な違いであり、PBRの導入がルーティングの可用性を損なわないための設計上の安全弁になっている。

!
学習ノート: ネクストホップの実効性set ip next-hop 10.27.1.1 は、hq-dist から見た通常のOSPF最適経路のネクストホップ（hq-core Ethernet0/1）と同一アドレスである。そのため本ラボの構成では「別経路へ迂回させる」というPBR本来の効果は限定的で、あくまでroute-map句の書き方を学ぶための例示値として扱う（課題管理表に記録済み・オープン）。

4. QoS詳細設計#

4.1 MQCの3段階

Cisco IOSのMQC（Modular QoS CLI）は3段階で構成する。本設計ではMQCインスタンスを2箇所に持つ。1つはhq-distでの分類＋マーキング（MARKING）、もう1つはhq-edgeでの優先制御（WAN-EDGE）である。

段階hq-dist（入力・マーキング）hq-edge（出力・優先制御）
① 分類class-map match-all VOICE-SRC（match access-group name VOIP-SRC）class-map match-any VOICE（match dscp ef）
② ポリシーpolicy-map MARKING（class VOICE-SRC: set dscp ef）policy-map WAN-EDGE（class VOICE: priority 2000 / class-default: fair-queue）
③ 適用service-policy input MARKING（Ethernet0/3）service-policy output WAN-EDGE（Ethernet0/2）

hq-distのMARKINGは送信元ACL（VOIP-SRC）による分類、hq-edgeのWAN-EDGEはDSCP値による分類、と分類条件が異なる点に注意する（4.2）。

4.2 DSCPマーキング位置と信頼境界

DSCPマーキングは、トラフィックの発生源に最も近いhq-dist Ethernet0/3インバウンドのMQC入力ポリシー（policy-map MARKING）で1回だけ行う（3.2で述べた通り、PBR route-mapへは統合しない）。hq-edgeのWAN側 class-map VOICE は独自にACLで再分類せず、既にマーキング済みのDSCP値を信頼して match dscp ef で拾うだけにとどめる。これは「マーキングは境界の内側でできるだけ発生源に近い場所で1回行い、以降は信頼する」というQoS設計の定石（信頼境界の設定）であり、境界を跨ぐたびにACLで再分類する無駄な処理を避けられる。

4.3 LLQ（priority）とfair-queueの挙動

class VOICE の priority 2000（kbps）はLLQ（Low Latency Queuing）を意味する。輻輳が発生している間、VOICEクラスのトラフィックは他クラスに先立って必ず送出される「絶対的な優先権」を持つと同時に、2000kbpsを超過した分はポリシングにより破棄される（優先権の見返りとして上限が課される）。輻輳していない時間帯は、このポリシングは働かず一時的に2000kbpsを超えて送出されうる。

class-default の fair-queue（WFQ）は、優先クラスに属さない残り全トラフィックを、フローごとに公平に帯域配分する。LLQのような絶対保証はないが、特定の大容量フロー1本が他のフローの帯域を独占することを防ぐ。

5. BGP経路フィルタリング詳細#

prefix-listはACLと同様にseq番号の順に評価され、最初にマッチしたエントリで確定し、末尾に暗黙denyを持つ。neighbor <IP> prefix-list <NAME> out は、そのピアへ広告する経路だけを絞り込むフィルタであり、自分が受信・インストールする経路には影響しない（受信側の制御にはinやdistribute-listを用いる）。

機器ASprefix-list適用
hq-edge65000HQ_NET（自拠点サブネットのみpermit）neighbor 198.51.100.10 prefix-list HQ_NET out
br-edge65001BRANCH_NET（10.27.20.0/24 permit）neighbor 198.51.100.13 prefix-list BRANCH_NET out
isp65100（フィルタなし）両ピアへ default-originate でデフォルトルートのみ広告

ispは実際のインターネット接続を持たないため、neighbor <IP> default-originate を使い、自身のRIBに関わらず0.0.0.0/0を無条件に相手へ広告する。hq-edgeはこれをeBGPで受信した後、内部OSPF（Area 0）へdefault-information originateで再配布し、HQ内の他ルータが「その他すべての宛先」をhq-edge経由でISPへ向かわせられるようにする。

i
br-core / br-edgeのArea 1についてOSPFの複数エリア設計では通常バックボーン（Area 0）への直結が必須だが、支社側のArea 1はHQのArea 0と同一OSPFプロセスに属さない、独立した単一エリアOSPFドメインである。両拠点はOSPFではなくBGP（ispを介したeBGP）でのみ相互接続されるため、バックボーン接続規則の対象外となる。

6. NAT詳細#

hq-edgeの Ethernet0/1（hq-core側）を ip nat inside、Ethernet0/2（isp側）を ip nat outside とする。HQ Trustの内部プライベートIP（10.27.10.0/24, 10.27.50.0/24）がインターネット方向へ通信する際、hq-edgeの外部インターフェースIP（198.51.100.9）へ overload（PAT）変換する。dmz-srv（10.27.30.0/24）はサーバ用途で外部への発信が主目的ではないため、本設計ではNAT対象に含めない。

変換順序（処理順）は方向によって異なる。

方向順序
内→外（inside→outside）ルーティング判定 → NAT変換（inside local→inside global）→ 外部インターフェースから送出
外→内（outside→inside、戻りパケット）インバウンドACL（OUTSIDE-IN、変換前の宛先で評価）→ NAT逆変換（global→local）→ 内部へ転送

この順序を理解しておくと、「なぜOUTSIDE-INがdmz-srvの宛先を直接プライベートIPで記述できるのか（＝そもそも静的NAT変換を経由しないため）」が矛盾なく説明できる。
