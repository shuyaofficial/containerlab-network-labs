## なぜ「順番」で並べ替えたか

ノード別に値だけ並んでいると、投入順が見えず次の事故が起きやすい：

- **Router ID 事故**: Loopback を作る前に OSPF/BGP を有効化すると、意図しない Router ID が選ばれ、直すのに手戻りが発生する。
- **ACL による自縄自縛**: ルーティングの疎通確認より先にゾーン間 ACL を適用すると、「経路の問題」か「ACL の問題」かの切り分けができなくなる。
- **QoS の空振り**: マーキング（入口）より先に LLQ（出口）を入れても、DSCP EF が付いていないパケットは優先キューに分類されず、効果検証ができない。

これを避けるため、次の4原則で並べています：

1. **Router ID はプロセス有効化の前に確定** — Loopback を先に作る。
2. **ルーティングの土台 → 疎通確認 → ポリシー** — まず OSPF/BGP/NAT で全拠点疎通を作り、確認してからポリシーを載せる。
3. **ポリシーオブジェクトは「作成 → 適用」** — ACL・prefix-list・class-map・policy-map・route-map は作ってからインターフェース／ネイバーに適用する。
4. **マーキングは入口、キューイングは出口** — DSCP EF を付ける設定（hq-dist）が先、EF を優遇する設定（hq-edge LLQ）が後。

## Phase 5. eBGP 基本と経路広告制御（Mission 5 の骨格）

| ノード | BGP | ネイバー | ネットワーク広告 / 制御 | 投入 |
|---|---|---|---|---|
| hq-edge | AS 65000 | isp (198.51.100.10) | prefix-list `HQ_NET`（10.27.10.0/24, 10.27.30.0/24, 10.27.50.0/24）を作成し out 適用。同3ネットを広告（OSPF学習済みであること） | ⬜ |
| br-edge | AS 65001 | isp (198.51.100.13) | prefix-list `BRANCH_NET`（10.27.20.0/24 のみ permit）を作成し out 適用 | ⬜ |
| isp | AS 65100 | hq-edge, br-edge | 各ピアへ Default route (`0.0.0.0/0`) を広告（default-originate） | ⬜ |

> 補足（Mission 5 発展）: OSPF の `distribute-list` による特定経路のインストール防止は、全疎通確認（Phase 7）後に追加で試すこと。先に入れると切り分けが難しくなる。

## Phase 8. ゾーン間 ACL（Mission 1）

**この順の理由**: 疎通の土台が確認できたので、ここで初めて「絞る」。ACLは**作成→適用**の順。適用位置と方向はパラメータシートどおり。

### hq-edge — `OUTSIDE-IN`（E0/2 インバウンド）
| 順序 | ルール | 意図 | 投入 |
|---|---|---|---|
| 作成1 | permit tcp any → host 10.27.30.10 eq 80 | DMZ公開HTTP | ⬜ |
| 作成2 | permit tcp any → host 10.27.30.10 eq 443 | DMZ公開HTTPS | ⬜ |
| 作成3 | permit icmp echo-reply（＋ tcp established） | 内部発の戻り通信 | ⬜ |
| 作成4 | deny ip any any | その他全拒否（明示） | ⬜ |
| 適用 | E0/2 インバウンドへ適用 | Untrust→内側の入口で遮断 | ⬜ |

### hq-dmz — `DMZ-OUT`（E0/2 = to dmz-srv インバウンド）
| 順序 | ルール | 意図 | 投入 |
|---|---|---|---|
| 作成1 | permit tcp host 10.27.30.10 eq 80 → any | Webサーバ応答の戻り許可。無いとTrust発HTTPのSYN-ACKがdenyで落ちる（ISS-13） | ⬜ |
| 作成2 | permit tcp host 10.27.30.10 eq 443 → any | 同HTTPS応答（establishedを使わないのはSSH禁止要件を守るため） | ⬜ |
| 作成3 | permit icmp host 10.27.30.10 → 10.27.10.0/24 echo-reply | PC→srvの片方向pingを許可（echo-reply非対称の応用・ISS-14） | ⬜ |
| 作成4 | deny ip 10.27.30.0/24 → 10.27.10.0/24 | DMZ→HQ Trust 遮断 | ⬜ |
| 作成5 | deny ip 10.27.30.0/24 → 10.27.20.0/24 | DMZ→Branch Trust 遮断 | ⬜ |
| 作成6 | deny ip 10.27.30.0/24 → 10.27.50.0/24 | DMZ→HQ VoIP（Trust）遮断 ※レビューで判明した漏れの追補（ISS-12） | ⬜ |
| 作成7 | permit ip any any | 戻り通信等は許可 | ⬜ |
| 適用 | E0/2 インバウンドへ適用 | 送信元（DMZ）に最も近い入口で遮断 | ⬜ |

**✔ ミニチェック**: isp→hq-pc ping が hq-edge で落ち、isp→dmz-srv TCP80 が通り ICMP が落ち、dmz-srv→hq-pc が落ちること（[正常性障害試験仕様書](../05_試験/正常性障害試験仕様書.html) ST-N系）。

## Phase 9. PBR — route-map によるポリシールーティング（Mission 2）

**この順の理由**: ACLで通信の可否が固まった後、経路の「選び方」を曲げる。マッチ用ACL（ポリシーオブジェクト）→ route-map → IF適用の順。

| ノード | 順序 | 設定内容 | 投入 |
|---|---|---|---|
| hq-dist | 1. マッチACL作成 | `VOIP-SRC`: permit ip 10.27.50.0 0.0.0.255 any | ⬜ |
| hq-dist | 2. route-map作成 | `PBR_VOICE permit 10`: match = VOIP-SRC / set ip next-hop **10.27.1.1**（hq-core） | ⬜ |
| hq-dist | 3. IF適用 | E0/3（to hq-voip）インバウンドへポリシー適用 | ⬜ |

> 注意: パラメータシート.md には route-map 内での `set dscp ef` 併用が記載されているが、PBR用 route-map の DSCP 設定は
> 未サポートの IOS が多い（[課題管理表](../06_プロジェクト管理/課題管理表.html) 参照）。**マーキングは Phase 10 の MQC 方式を正とする。**

**✔ ミニチェック**: hq-voip から traceroute → 10.27.1.1 経由 / hq-dist で route-map の match カウント増加。

## Phase 10. QoS マーキング — 入口（Mission 3）

**この順の理由**: **キューイング（Phase 11）より先にマーキング**。EFが付かない限り出口のLLQは空振りする。class-map → policy-map → IF適用（input）の順。

| ノード | 順序 | 設定内容 | 投入 |
|---|---|---|---|
| hq-dist | 1. class-map作成 | `VOICE-SRC`: match = ACL `VOIP-SRC`（Phase 9 で作成済みを再利用） | ⬜ |
| hq-dist | 2. policy-map作成 | `MARKING`: class VOICE-SRC に DSCP **EF (46 / 0x2E)** をセット | ⬜ |
| hq-dist | 3. IF適用 | E0/3 インバウンドへ service-policy **input** 適用 | ⬜ |
| （任意） | 業務データ | 社内PCの特定業務トラフィックに **AF21 (18)**、その他は BE (0) | ⬜ |

**✔ ミニチェック**: hq-voip から ping 継続 → 経路上で tcpdump し TOS/DSCP = **0x2E** を確認。

## Phase 11. LLQ / 帯域制御 — 出口（Mission 4）

**この順の理由**: 入口でEFが付いた状態で、WAN出口の輻輳制御を入れる。class-map → policy-map → IF適用（output）の順。

| ノード | 順序 | 設定内容 | 投入 |
|---|---|---|---|
| hq-edge | 1. class-map作成 | `VOICE`（match-any）: match dscp **ef** | ⬜ |
| hq-edge | 2. policy-map作成 | `WAN-EDGE`: class VOICE に priority **2000 kbps**（LLQ絶対保証）/ class-default に fair-queue | ⬜ |
| hq-edge | 3. IF適用 | E0/2（to isp）アウトバウンドへ service-policy **output** 適用 | ⬜ |
| （任意） | shaping | WAN帯域上限（例: 10Mbps）で shaping し、輻輳を意図的に作って効果を観測 | ⬜ |

**✔ ミニチェック**: hq-edge で `show policy-map interface`（E0/2）→ VOICEクラスのパケットカウント増加。iperf 負荷時に EF が守られ class-default でドロップが出ること。
