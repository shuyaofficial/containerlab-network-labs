1. 本書の目的と対象#

本書は、Theme 27ラボ環境の引き渡し後、日常運用および一次切り分けを行うための手順書である。設計思想・パラメータの詳細は01〜03の各設計書、試験の合否基準は05_試験の試験仕様書（単体・結合・正常性障害）、既知の課題は同フォルダの課題管理表を参照する。

対象読者は本ラボの運用担当者（受講者本人、および将来の引き継ぎ先）。全体スケジュールはWBS・スケジュールを参照。

2. 環境の起動と停止#

すべての操作はOrbStack Linux VM clab 上で実行する。

実行場所への接続 
ssh clab@orb
cd /Users/shuya/Documents/claude/Mac仮想環境構築/27_qos_acl_policy_control/04_構築

起動（deploy）

環境デプロイ 
./deploy.sh deploy
i
IOL起動待ちCisco IOL（IOS on Linux）の起動には約2分かかる。直後にinspectしてもrunningにならない場合は少し待って再確認する。

稼働確認（inspect）

ノード稼働状態の確認 
./deploy.sh inspect

全ノード（Cisco IOLおよびLinuxコンテナ、計11台）が running であることを確認する。

停止・破棄（destroy）

環境の破棄 
./deploy.sh destroy
!
設定消失に注意destroyを実行すると保存していないIOS設定はすべて消える。事前に write memory で保存すること（詳細は6章）。

3. 機器へのアクセス#

Cisco IOLへの接続

IOL接続（例: hq-edge） 
sudo docker attach --sig-proxy=false clab-qos-acl-policy-lab-hq-edge

画面がすぐ表示されない場合は、接続後にEnterキーを1〜2回押す。

✕
離脱はCtrl-p→Ctrl-q、Ctrl-cは禁止コンテナを止めずに離脱する場合は必ず Ctrl-p を押し、続けて Ctrl-q を押す。Ctrl-cはIOSプロセス自体を終了させてしまうため、離脱には絶対に使用しない。

Linux端末への接続

Linuxコンテナ接続（例: hq-pc） 
sudo docker exec -it clab-qos-acl-policy-lab-hq-pc bash

Linuxシェルからは exit で抜ける（コンテナは終了しない）。

4. 日常監視ポイント#

以下のコマンドを定期的に実行し、ポリシー制御が意図通り機能しているかを確認する。

OSPF隣接（hq-core／hq-dist／hq-dmz／br-core）

OSPF隣接状態
show ip ospf neighbor

何を見るか: 各エリア（HQ=Area 0／Branch=Area 1）の隣接関係確立状況。正常値: 全隣接がFULL（DR/BDR構成ではFULL/DR・FULL/BDRを含む）。

BGPピア（hq-edge／br-edge／isp）

BGPサマリ
show ip bgp summary

何を見るか: 対向AS（hq-edge=65000／br-edge=65001／isp=65100）とのピア状態。正常値: StateがEstablished相当（数値表示）で、PfxRcdが期待経路数（0ではない）。

ACLヒットカウント

ACLカウンタ確認
show access-lists

何を見るか: OUTSIDE-IN／DMZ-OUT各エントリのヒットカウント。正常値: permit/denyの想定行でカウンタが増加し、想定外のdeny行が急増していないこと。

PBRマッチカウント（hq-dist）

route-map確認
show route-map

何を見るか: PBR_VOICEのmatch/setカウント。正常値: hq-voip（10.27.50.0/24）からの通信発生時にmatchesが増加すること。

QoS/LLQ（hq-edge WAN側）

policy-map確認
show policy-map interface Ethernet0/2

何を見るか: VOICEクラス（LLQ）のパケット送出・ドロップカウント。正常値: VOICEクラスにパケット送出があり、class-defaultで過度なドロップ（Tail Drop急増）が発生していないこと。

5. 障害時の一次切り分け#

症状: コンテナが落ちている

  1. 稼働状態を確認する
    コンテナ状態確認
    sudo docker ps --format 'table {{.Names}}\t{{.Status}}' | grep qos-acl-policy-lab
    Exited等になっているコンテナがないか確認する。
  2. 起動ログを確認する
    ログ確認（例: hq-core）
    sudo docker logs --tail 50 clab-qos-acl-policy-lab-hq-core
    クラッシュ理由やエラーメッセージを確認する。
  3. 復旧させる 原因が特定できない場合は、ラボの再作成（6章）を検討する。再作成前に必要な設定を控えておくこと。

症状: 疎通不可（ping/curlが通らない）

L1（インターフェース状態）→ L3（経路）→ ポリシー（ACLカウンタ）の順に切り分ける。

  1. L1: インターフェース状態 show interfaces で対象インターフェースがup/upであることを確認する。
  2. L3: 経路 show ip route で対向への経路（OSPF/BGP双方）が存在するか確認する。
  3. ポリシー: ACLカウンタ show access-lists で該当ACLのdenyカウンタが増加していないか確認する。
i
連絡先切り分けの結果（症状・実施したコマンドと結果）は速やかにメンターへ連絡する。原因不明の場合も含め、途中経過を共有すること。

6. 定期メンテナンス#

設定保存

設定変更後は各機器で以下を実行し、コンテナ再起動時にも設定を保持する運用とする。

設定保存
write memory
!
destroyでは保存されないwrite memoryはコンテナのファイルシステムには保存されるが、containerlab destroy実行時にはコンテナごと削除されるため設定は失われる（課題管理表 ISS-04参照）。

ラボ再作成手順

再作成
cd 04_構築
./deploy.sh destroy
./deploy.sh deploy
✕
設定消失に注意保存していない設定はすべて消える。事前にwrite memoryを実行し、必要なら実施済みコンフィグをテキストファイルに控えておくこと。config.txtは意図的に非同梱のため、再投入は手動で行う。

7. FAQ#

Q. 接続してもすぐに画面が表示されない

A. 接続後にEnterキーを1〜2回押す。それでも表示されない場合はdocker psで稼働状態を確認する。

Q. docker attachできない・エラーになる

A. sudo docker ps --format 'table {{.Names}}\t{{.Status}}' | grep qos-acl-policy-labでコンテナがrunningか確認する。runningでなければdocker logs --tail 50 <container>で起動ログを確認する。

Q. 誤ってCtrl-cで抜けてしまった

A. IOSプロセス自体が終了している可能性が高い。docker psでExitedになっていないか確認し、docker logsで状況を確認したうえで、必要ならラボの再作成（6章）を検討する。

Q. destroy後に設定が消えた

A. 仕様通りの挙動（課題管理表 ISS-04参照）。config.txtは非同梱のため、全設定を手動で再投入する方針としている。ただしwrite memory済みの設定は、--cleanupを付けないcontainerlab destroy→deployならnvram経由で復元される。

Q. Mac/OrbStack再起動後、ルータ同士が全く通信できない（OSPF隣接が張れない等）

A. 既知事象（課題管理表 ISS-09〜11）。各ルータでwrite memory後、--cleanupなしのcontainerlab destroy → deploy で復旧する（2026-07-10のentrypoint修正以降、iouyapはIOLブート+60秒後に自動起動・自己修復するため手動投入は不要）。個別コンテナのdocker restartは厳禁（データリンクvethが対向側もろとも消失し悪化する）。詳細手順と切り分け方法は 04_構築/トラブルシューティング_IOL受信不能_2026-07-10.md を参照。判定はブート後5分待ってから行うこと。
