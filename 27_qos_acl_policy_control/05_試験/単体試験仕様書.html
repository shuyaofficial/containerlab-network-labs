1. 試験方針#

1.1 目的

Theme 27（QoS/ACL/route-map応用）ラボを構成する各ノードが、単体で正しく構築されていることを検証する。プロトコル隣接やエンドツーエンド疎通など複数ノードにまたがる確認は結合試験仕様書で、ポリシー動作や障害時挙動は正常性障害試験仕様書で扱う。

1.2 範囲

  • Cisco IOL 7台（hq-core / hq-dist / hq-edge / hq-dmz / br-core / br-edge / isp）のコンテナ起動状態・起動ログ・hostname・インターフェースIP/状態・Loopback・設定保存
  • Linux端末 4台（hq-pc / hq-voip / br-pc / dmz-srv）のeth1アドレス設定・デフォルトルート

1.3 前提条件

  • 04_構築/deploy.sh deploy が完了し、全11ノードが投入済みであること
  • 作業は OrbStack Linux VM clab（ssh clab@orb）上で実施すること
  • IOL機器は起動に約2分を要するため、投入直後は起動完了を待ってから試験を開始すること
  • 各機器へのACL/QoS/PBR/BGP設定投入は本試験の対象外（結合試験仕様書・正常性障害試験仕様書実施前に完了していればよい）

3. 試験項目#

IOL機器はCLI確認に sudo docker attach --sig-proxy=false clab-qos-acl-policy-lab-<name> でログインする（離脱は Ctrl-p → Ctrl-q）。Linux端末は sudo docker exec -it clab-qos-acl-policy-lab-<name> bash。判定は実施後に ● 合格 / ▲ 条件付き / ✕ 不合格 のいずれかへ更新する。

4. 合格基準#

  • 本書の全25項目が 合格 であること。
  • 不合格 が1件でもある場合、該当ノードの設定を是正し再試験を実施する。是正内容は 04_構築/構築ログ_テンプレート.md に記録する。
  • Linux端末のIP/デフォルトルートは campus.clab.yml の exec ブロックにより自動投入されるため、不合格の場合はコンテナ再作成（./deploy.sh destroy && ./deploy.sh deploy）で切り分ける。
  • 本試験に合格したのち、結合試験仕様書に進む。
