# トラブルシューティング: VM再起動後にOSPF隣接が全リンクで確立しない（IOL受信不能）

## 日付: 2026-07-10
## 事象

- Mac/OrbStack再起動（11:18 JST）後の再デプロイ以降、**全IOLでOSPF隣接がゼロ**
- `show ip ospf neighbor` 空 / LSDBには過去の相手LSAが残留（=以前は隣接が張れていた証拠）
- ユーザー設定は正しい（passive・エリア・タイマー・P2P全て一致）ことを確認済み

## 切り分けの経過（レイヤを下りながら確定した事実）

1. **設定層**: 両ルータのOSPF設定は正常 → 設定起因ではない
2. **配線層**: `ip netns exec <container> tcpdump -ni ethX proto 89` で**双方向のHelloが線上を流れている**ことを確認 → veth/リンク正常
3. **iouyap層**: strace で iouyap が受信フレームを IOL のソケット（/tmp/netio0/&lt;id&gt;）へ **sendto 成功**していることを確認 → iouyap正常
4. **IOL層**: strace で IOL(qemu-i386) が **recvfrom でHelloを受信している**のに近隣リストに反映せず → IOS内部で無視されている状態
5. **IOS層の裏付け**: 障害当時のbr-coreで `show ip ospf traffic` を採取したところ **RX Hello = 0** — OSプロセスには届いているがIOSのOSPFプロセスは受信を一切カウントしていない（IOL内部の破棄を確証）
6. Hello長で状態判定できる: **length 60 = Neighbor List空（相手未認識）/ length 64以上 = 相手認識済み**

## 誤った対処（やってはいけない）

- ❌ **`docker restart <IOLコンテナ>` は厳禁**。containerlabのデータリンク（veth）はデプロイ時にのみ作られるため、コンテナ再起動で**そのノードの全データIF（eth1〜）が消失**し、対向ノード側のIFも道連れで消える（vethは対で削除される）。実際にbr-edgeで実施してeth1/eth2を喪失し、事態を悪化させた
- ❌ iouyapの再投入だけでは復旧しなかった（本事象ではIOL側の受信処理が復帰しない）

## 復旧手順（確定・実績あり）

```bash
# 0) 各ルータで write memory（nvramはホスト側にバインドされており再デプロイ後も残る）
# 1) nvramバックアップ（保険）
sudo cp -a 04_構築/clab-qos-acl-policy-lab /tmp/clab-nvram-backup-$(date +%H%M)
# 2) --cleanup を付けずに destroy（labディレクトリ=nvram温存。deploy.sh destroyは--cleanup付きなので使わない）
sudo containerlab destroy -t campus.clab.yml
# 3) 再デプロイ（既存nvramが再バインドされ、保存済みコンフィグが復元される）
sudo containerlab deploy -t campus.clab.yml
# 4) IOLブート完了後（60秒〜）、iouyapを入れ直す（既存をkillしてから）
#    ノード内にps/pgrep/killバイナリは無い。/proc走査+shビルトインkillを使う
for n in hq-core hq-dist hq-edge hq-dmz br-core br-edge isp; do
  c=clab-qos-acl-policy-lab-$n
  pids=$(sudo docker exec $c sh -c 'for p in /proc/[0-9]*; do grep -q iouyap $p/comm 2>/dev/null && echo ${p#/proc/}; done')
  for p in $pids; do sudo docker exec $c sh -c "kill $p"; done
  sudo docker exec -d -w /iol $c /usr/bin/iouyap 513
done
# 5) 検証（コンソール不要）: Hello長が64以上 = 隣接確立
sudo ip netns exec clab-qos-acl-policy-lab-br-core tcpdump -c 2 -vvni eth1 proto 89
```

- **判定は5分待ってから**。7台同時ブート（ARM上のqemu-i386）直後はCPU飽和でOSPFの反応が遅く、再投入直後の1〜2分は近隣リスト空のままのことがある（今回も約5分後に全リンク確立）

## 根本原因（確定と推定）

- 確定: 復旧には**containerlabによる完全再デプロイ**が必要だった（コンテナ・veth・iouyap・IOLの整合を作り直す）
- 推定: VM起動直後の`deploy.sh deploy`では、7台同時ブートの高負荷下で iouyap（sleep 5後に起動）とIOLの初期化順序が不整合となり、IOLが受信処理を行わない状態で安定してしまう

## 恒久対策（2026-07-10 実装済み — 以降は手動iouyap投入不要）

同日中に再発（再デプロイ時のiouyap投入漏れ）したため、環境側を構造的に修正した:

1. **entrypoint_attach.sh に iouyap 自己修復起動を実装**: IOLブート+60秒後に自動起動し、終了しても10秒間隔で再起動する。デプロイ・コンテナ再起動のどちらでも配線が自動復旧するため、上記復旧手順の手順4（iouyap手動投入）は**次回デプロイ以降不要**
2. **campus.clab.yml の端末ルートを `ip route add` → `ip route replace` に修正**（→ ISS-10）: dockerの管理側デフォルトルートと衝突して初回から毎回失敗していた（デプロイログの `RTNETLINK answers: File exists` がその痕跡）。端末からの疎通試験が通らない隠れ要因だった
3. **deploy.sh の自動 iouyap 起動（+5秒）を廃止**: 早すぎる投入タイミングを排除し、entrypoint（+60秒・IOLブート後）に一本化。`./deploy.sh iouyap` は手動リカバリ用に残置

## 再発防止・教訓

1. **Mac/OrbStack再起動後は必ず上記の復旧手順（clab destroy→deploy→iouyap再投入）を実施**。docker restart は使わない
2. nvram はホスト側 `04_構築/clab-qos-acl-policy-lab/<node>/nvram_*` にあり、`--cleanup` さえ避ければ再デプロイでコンフィグは失われない（作業前に `write memory` を習慣化）
3. 切り分けは「設定→配線(tcpdump)→iouyap(strace sendto)→IOL(strace recvfrom)」の順でレイヤを下る。**Hello長（60=未認識/64+=認識）**が最速の健全性指標
4. IOLノードのID割当: containerlabが `IOL_PID` 環境変数で注入（br-core=1, br-edge=2, hq-core=5等）。iouyapは常に513。nvramファイル番号はこのIDに対応
