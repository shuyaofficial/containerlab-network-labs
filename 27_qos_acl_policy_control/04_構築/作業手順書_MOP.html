作業手順書（Method of Procedure）

v1.0 承認済み 最終更新 2026-07-08 ソース: deploy.sh, 00_ログイン/ログインコマンド.md, README_Lab_Challenge.md

1. 作業概要#

目的Theme 27ラボ環境を構築し、ACL・PBR・QoS・BGP経路フィルタリングのポリシー設定をMission 1〜6の順に投入する
所要時間目安環境構築 約10分（IOL起動待ち2分含む）＋ 設定投入 各ミッション30〜60分 × 6 ＝ 半日〜1日程度
実施環境OrbStack Linux VM clab（ssh clab@orb）上のDocker + containerlabラボ
対象トポロジ04_構築/campus.clab.yml（7台のcisco_iol + 4台のLinuxエンドポイント）

2. 事前確認#

  1. 作業ディレクトリへ移動
    clab VM上
    ssh clab@orb
cd /Users/shuya/Documents/claude/Mac仮想環境構築/27_qos_acl_policy_control
  2. Dockerデーモンの稼働確認
    Docker稼働確認
    sudo docker ps

    コマンドがエラーなく応答すればDockerは稼働している。

  3. 既存ラボの有無を確認
    04_構築/
    cd 04_構築
./deploy.sh inspect

    同名ラボ（qos-acl-policy-lab）が既に稼働中の場合は、6章の切り戻し手順を確認してから作業を進める。

3. 環境構築#

  1. デプロイ実行
    04_構築/
    ./deploy.sh deploy

    containerlab deploy の実行後、内部で全IOLコンテナへ iouyap（仮想イーサネット多重化プロセス）を自動起動する。

  2. IOL起動待ち（約2分）

    IOLはブートに時間がかかるため、直後は docker attach しても画面が出ないことがある。焦らず約2分待つ。

  3. 全ノードrunning確認
    04_構築/
    ./deploy.sh inspect

    7台のIOLノードと4台のLinuxノード、計11ノード全てが running であることを確認する。

  4. （必要時）iouyapの再起動

    コンテナ再起動直後などで仮想イーサネットが疎通しない場合は、以下で iouyap のみを再実行できる。

    04_構築/
    ./deploy.sh iouyap

4. 設定投入#

詳細な設定値はパラメータシート、動作原理は詳細設計書、完成形の一例はコンフィグ解答編を参照。すべて手動でCLI投入する（clab.ymlへの設定記述は禁止）。

Mission 1: ベースライン構築＋ゾーン間ACL

  1. 対象: 全7機器（OSPF Area 0 / Area 1、BGP AS65000/65001/65100）基本のルーティングを確立する。確認: show ip ospf neighbor / show ip bgp summary
  2. 対象: hq-edge（OUTSIDE-IN）・hq-dmz（DMZ-OUT）ゾーン間ACLを作成しインバウンドへ適用する。確認: show access-lists

Mission 2: route-mapによるPBR

  1. 対象: hq-distACL VOIP-SRC → route-map PBR_VOICE（match / set ip next-hop）を作成し、Ethernet0/3インバウンドへ ip policy route-map を適用する。確認: show route-map

Mission 3: DiffServ QoSの基礎（DSCPマーキング）

  1. 対象: hq-distMission 2のroute-mapへ set dscp ef を追加し、hq-voip発トラフィックをマーキングする。確認: エンドポイントでの tcpdump によるTOS/DSCPフィールド確認

Mission 4: LLQ + 帯域制御

  1. 対象: hq-edge class-map VOICE（match dscp ef）→ policy-map WAN-EDGE（priority 2000 / fair-queue）→ Ethernet0/2アウトバウンドへ service-policy を適用する。確認: show policy-map interface Ethernet0/2

Mission 5: prefix-listによる経路広告制御

  1. 対象: hq-edge・br-edge各拠点でprefix-list（HQ_NET / BRANCH_NET）を定義し、neighbor ... prefix-list ... out を適用する。確認: show ip prefix-list
  2. 対象: isp各ピアへ neighbor ... default-originate を設定する。対象: hq-edge受信したデフォルトルートをOSPFへdefault-information originateで再配布する。確認: show ip bgp（isp上で受信経路を確認）

Mission 6: 総合シナリオ試験

  1. 対象: 全機器・全エンドポイントリンク障害（インターフェースshutdown等）を意図的に発生させ、迂回後もACL・PBR・QoSのポリシーが継続して適用されることを確認する。詳細は5章および正常性障害試験仕様書を参照。

5. 事後確認#

設定投入後は、Mission 1〜6の確認項目に沿って、ゾーン間ACL・PBR・DSCPマーキング・LLQ優先制御・経路広告制御・総合シナリオの一連の試験を実施する。個々の機器単位の確認項目は単体試験仕様書、拠点間・エンドツーエンドでの確認項目は結合試験仕様書、ポリシー動作と障害時動作の確認項目は正常性障害試験仕様書に定義されている。

i
試験文書の構成本ラボの試験文書は「単体試験仕様書（機器単体の設定・状態確認）」「結合試験仕様書（隣接疎通・ネイバー確立・エンドツーエンド確認）」「正常性障害試験仕様書（ポリシー動作と障害時動作の確認）」の3種で構成する。いずれも05_試験/試験計画書.mdを基に作成されている。

6. 切り戻し手順#

状態手順
設定を保存していない場合各機器で reload するだけで、NVRAMに未保存の設定は消え初期状態（config.txtなしの空コンフィグ）に戻る
write memory済みの場合reloadだけでは設定が残ってしまうため、./deploy.sh destroy → ./deploy.sh deploy でラボ全体をクリーンに再構築する
04_構築/
./deploy.sh destroy
./deploy.sh deploy
!
write memory の注意write memory（copy running-config startup-config）を一度実行すると、以後の reload では設定が保持されてしまう。「まっさらな状態からやり直したい」場合は必ず destroy → deploy を使うこと。

7. 注意事項#

✕
docker attachからの離脱はCtrl-p → Ctrl-q、Ctrl-cは禁止sudo docker attach --sig-proxy=false clab-qos-acl-policy-lab-<name> でログイン後、コンテナを止めずに離脱するには Ctrl-p を押してから続けて Ctrl-q を押す。Ctrl-c を使うとIOLプロセス（PID1）にシグナルが伝わりコンテナが停止する。
  • Ethernet0/0（mgmt、172.27.27.0/24）はcontainerlabが自動設定するため、受講者が手動設定する必要はない。
  • campus.clab.yml への設定記述は禁止。全ポリシー設定はCLIから手動投入する（Linuxエンドポイント4台のIP設定のみexecで自動化済み）。
  • 管理用パスワード・IOURCライセンス値は公開リポジトリのためサニタイズ済み。各自のライセンス値へ差し替えて使用する。
