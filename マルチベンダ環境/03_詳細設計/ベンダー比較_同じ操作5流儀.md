# ベンダー比較_同じ操作5流儀 — マルチベンダ環境

「同じ目的の操作」を5ベンダー（FRR／MikroTik RouterOS／Cisco IOS(IOL)／Nokia SR Linux／OpenWrt）の
流儀で対比する。コマンドは各ベンダーの一次資料・実測に基づく設計値であり、`04_構築`完了後の
実投入結果ではない（実投入ログは04_構築側の担当）。

| ベンダー | イメージ/バージョン | 本ラボのノード | 認証・アクセス |
|---|---|---|---|
| FRRouting | 10.2.1 | isp-a, isp-b | `docker exec` + `vtysh` |
| MikroTik RouterOS | CHR 7.21.4 | hq-edge | SSH（admin/admin）+ REST（https、自己署名証明書） |
| Cisco IOS（IOL） | 15.7.3M2 / L2-advipservices-2017 | hq-core, hq-sw | `docker attach`（SSH不可） |

## ⑥ 設定変更（例: ループバックアドレス追加）とロールバック

| ベンダー | 変更 | ロールバック |
|---|---|---|
| RouterOS | `/ip/address/add address=192.0.2.1/32 interface=lo` | Safe Mode（`Ctrl+X`または`/system safe-mode`）で切断時に自動破棄。付録2で実証済み: `PUT`→`GET`再読取で`.id`確認→`DELETE`→`/export`前後diffがタイムスタンプ行のみ（完全ロールバック） |
| Cisco IOS（IOL） | `interface loopback0` / `ip address 192.0.2.1 255.255.255.255` | アトミックな取消機構なし。`no`系コマンドで手動取消、または事前に`archive config`済みなら`configure replace` |
| SR Linux | `enter candidate` → `set / interface system0 subinterface 0 ipv4 address 192.0.2.1/32` → `commit confirmed 120` | confirmedタイマー（120秒）内に`commit save`しなければ**自動ロールバック**（他ベンダーに無いタイマー式の安全機構） |
| FRR | `vtysh -c "configure terminal" -c "interface lo" -c "ip address 192.0.2.1/32"` | アトミックな取消機構なし。`no ip address 192.0.2.1/32`で手動取消 |
| OpenWrt | `ip addr add 192.0.2.1/32 dev lo` | `ip addr del 192.0.2.1/32 dev lo`で手動取消（imperativeのため取消も手動）。本来のOpenWrt流儀なら`uci revert network`だが、本コンテナはnetifd不動作のため使えない |

## ⑧ APIでの取得

| ベンダー | 例 | 実測状況 |
|---|---|---|
| RouterOS | `curl -sk -u admin:admin https://172.20.50.11/rest/system/resource` | **実証済み**（付録2、chr7214でFULL判定） |
| SR Linux | `curl -sk -u admin:'NokiaSrl1!' -X POST https://172.20.50.13/jsonrpc -H 'Content-Type: application/json' -d '{"jsonrpc":"2.0","id":1,"method":"get","params":{"commands":[{"path":"/interface","datastore":"state"}]}}'` | JSON-RPC疎通・構造化JSON応答は実証済み（HTTP/HTTPS両方で応答）。上記payload自体はSR Linux公開仕様の標準形式 |
| Cisco IOS（IOL） | — | **console手順のみ**（SSH不可。`docker attach`後に`show`コマンドを人手/スクリプトで採取） |
| FRR | `docker exec clab-mvlab-isp-a vtysh -c "show ip bgp summary json"` | HTTP APIは無し。`vtysh -c`のJSON出力オプションをプログラム的取得の代替とする（**console手順**） |
| OpenWrt | `ubus call network.interface dump`（本来のOpenWrt API） | netifd不動作のためネットワーク関連ubus呼び出しは機能しない。VM版OpenWrtへの切替で有効化予定（README拡張案） |

## まとめ（本ラボで見えた設計上の示唆）

- **API-firstの実装度**: RouterOS（REST）とSR Linux（JSON-RPC）は実際に可逆変更まで実証できたが、
  IOL（console限定）とFRR（vtysh限定）はプログラム的取得に一段の工夫（JSON出力オプション等）が要る。
- **ロールバックの思想差**: RouterOSは切断ベースのSafe Mode、SR Linuxはタイマーベースのconfirmed
  commit、Cisco IOS/FRRは明示的な取消機構を持たず`no`系コマンドでの手動対応が基本、OpenWrt(本環境)
  はimperativeゆえロールバックという概念自体が薄い。
- **既定の広報/受理ポリシー**: SR Linuxはexport/importポリシーを明示しない限り経路を広報・受理
  しない（安全側の既定）。Cisco IOS/RouterOS/FRRは`network`文や`redistribute`を書けば広報される
  素直な挙動で、意図しない広報を防ぐには別途フィルタが要る。
