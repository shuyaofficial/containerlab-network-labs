# 試験計画書: マルチベンダ環境

対象: `multivendor.clab.yml`（`mvlab`）。IP/AS/Router IDは [../02_基本設計/IPアドレス管理表.md](../02_基本設計/IPアドレス管理表.md) を正とする。
自動確認スクリプトは [verify.sh](verify.sh)（**OrbStack VM上で実行**、TSVで OK/NG/INFO を出力）。

## 試験環境・前提

- `ssh clab@orb` でVMにログインし、VM上で `sudo docker exec` / `curl` / `sshpass` を使用する。
- 認証: RouterOS `admin/admin`（devices.yaml）、SR Linux `admin/NokiaSrl1!`。
- IOL 2台（hq-core/hq-sw）はSSH不可・console運用のため、OSPF隣接は `docker attach` での目視、または他ノード側からの間接確認とする。
- 事前に `04_構築/deploy.sh deploy` 完了、かつ RouterOS/BGP/OSPF が収束（デプロイ後数分）していること。

## 3. 冗長試験（デュアルホームeBGPの切替）

hq-edge は isp-a / isp-b の2本でインターネット側へデュアルホームしている。isp-a向けリンクを落とし、
経路が isp-b 側へ切り替わって疎通が維持されることを確認する。

| No | 目的 | 手順 | 期待値 | 結果 |
|---|---|---|---|---|
| 3-1 | 障害前の基準 | `docker exec clab-mvlab-hq-pc1 ping -c2 198.51.100.1` | 到達 | |
| 3-2 | isp-a向けリンク断 | `sshpass -p admin ssh admin@172.20.50.11 "/interface disable ether2"` | ether2 down、to-isp-a BGPセッション断 | |
| 3-3 | 切替後の疎通維持 | `docker exec clab-mvlab-hq-pc1 ping -c4 198.51.100.1` | 到達（isp-b→isp-a 経由へ迂回） | |
| 3-4 | 経路の迂回確認 | `sshpass -p admin ssh admin@172.20.50.11 "/routing/bgp/session print"` | to-isp-b のみ表示（to-isp-a は消える） | |
| 3-5 | 復旧 | `sshpass -p admin ssh admin@172.20.50.11 "/interface enable ether2"` | ether2 up、to-isp-a BGP再確立、両セッションestablished | |

> 3-3 の到達根拠: isp-a の 198.51.100.1/32 は isp-b が eBGP で再広報しているため、hq-edge は isp-b 経由でも
> 198.51.100.1 に到達できる。ether2 断でも 198.51.100.1 への疎通は維持される。

## 判定基準

- **疎通試験（1章）・経路試験（2章）が全てOK** で相互接続の合格とする（最重要）。
- 冗長試験（3章）は切替と復旧の両方がOKであること。
- API試験（4章）はベンダー運用比較の付随確認。書式差により要調整の可能性があるため、
  NG時は `04_構築/構築手順書.md` §6「要実機確認ポイント」を参照して切り分ける。
- 自動一括確認は [verify.sh](verify.sh) の `SUMMARY FAIL=0` を合格の目安とする（OSPFのIOL目視項目はINFO扱い）。
