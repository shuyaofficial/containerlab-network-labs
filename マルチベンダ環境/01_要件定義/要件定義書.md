# 要件定義書: マルチベンダ環境

## 1. 背景・目的

実務のネットワークは1社であっても単一ベンダーで統一されているとは限らない。ISP接続・WANエッジ・
キャンパス・DC・ブランチCPEといったゾーンごとに、コスト・機能・運用性の観点で異なるベンダーが
選定されることが多い。本テーマは「1社の企業ネットワークを、ゾーンごとに最適ベンダーで構成する」
という前提をラボ化し、**同じ目的の操作（経路確認・ピア状態確認・設定投入・API取得）がベンダーに
よってどれだけ流儀・コマンド体系・自動化しやすさが異なるか**を実機（コンテナ実装）で体感すること
を目的とする。

## 2. 対象範囲

| 区分 | 内容 |
|---|---|
| 対象ゾーン | Untrust（ISP2社）／WANエッジ（HQデュアルホーム）／Trust（HQキャンパス）／DC（リーフ×2）／ブランチ（CPE+NAT） |
| 対象プロトコル | eBGP（AS間）、OSPF（HQキャンパス内、単一エリア0）、VLAN/SVI、静的経路+NAT（ブランチ） |
| 対象ベンダー | FRRouting 10.2.1／MikroTik RouterOS CHR 7.21.4／Cisco IOL 15.7.3M2 + L2-advipservices-2017／Nokia SR Linux v26.3.3／OpenWrt 24.10.7（rootfs） |
| 対象操作比較 | インターフェース一覧／ルーティングテーブル／BGPピア状態／OSPF隣接／設定表示・エクスポート／設定変更とロールバック／保存／APIでの取得 の8操作 |
| 扱わない範囲 | NX-OS等の追加ノード拡張（README拡張案に記載のみ）、VM型OpenWrtへの切替、実際のトラフィック負荷試験、SNMP/監視基盤の統合 |

## 3. 機能要件

- ISP 2社（isp-a／isp-b）間をeBGPでピアさせ、双方の自ASループバックを交換できること。
- HQ WANエッジ（hq-edge）が isp-a・isp-b の両方とeBGPデュアルホーム構成を組み、片系障害時にもう一方の
  デフォルトルートで継続到達できること。
- HQ WANエッジは自社アドレス空間（10.50.0.0/16）を集約広報し、未割当領域へのブラックホール経路を
  あわせて持つこと。
- HQキャンパス（hq-core／hq-sw）はOSPF area0で構成し、hq-sw上のSVI（VLAN10／VLAN20）で収容した
  端末セグメントの経路をOSPFで広報できること。
- hq-coreはOSPFとDCリーフ向けeBGPの再配信境界（ASBR相当）として機能し、DC側の経路をOSPFへ、
  デフォルトルートをDCリーフへBGP default-originateで配布できること。
- DCリーフ（dc-leaf1／dc-leaf2）は自LANサブネット（/24）をeBGPで広報し、hq-coreからのデフォルト
  ルートを受け取れること。SR Linuxの既定挙動に合わせ、export/importポリシーを明示すること。
- ブランチCPE（br-edge）はISPへの静的デフォルトルートとNAT（masquerade）でLAN側端末を外部到達
  させること（BGP／OSPFには参加しない）。
- 5ベンダーそれぞれについて、インターフェース一覧・ルーティングテーブル・ピア状態・設定表示・
  設定変更とロールバック・保存・APIでの取得、という同一目的の操作を横断比較できるドキュメントを
  整備すること。

## 4. 非機能要件

- IOL・RouterOS CHRの起動に約1.5〜2分かかる（x86 QEMUエミュレーション経由）ことを前提に、
  デプロイ完了の目安時間をREADMEに明記すること。
- SR Linux・FRR・OpenWrtはarm64ネイティブで数秒〜十数秒の起動を前提とする。
- RouterOS・SR LinuxはAPI（REST／JSON-RPC）経由での状態取得・可逆変更が実証可能な状態を保つこと
  （認証情報・エンドポイントは環境検証ドキュメントの実測値に一致させる）。
- IOLはSSH不可という制約を前提に、console（`docker attach`）運用手順を明文化すること。

## 5. ゾーン設計方針

『ネットワーク・デザインパターン』のゾーン区分を次のように適用する。

| ゾーン | 本ラボでの対応 |
|---|---|
| Untrust | isp-a／isp-b（インターネット上のサービス役としてループバックを保持） |
| WANエッジ（Untrust/Trust境界） | hq-edge（デュアルホームeBGP冗長、Untrustゾーンのデザインパターンに準拠） |
| Trust | hq-core／hq-sw／hq-pc1／hq-pc2（HQキャンパス） |
| DC（Trust内の別セグメント） | dc-leaf1／dc-leaf2／srv-web／srv-db |
| ブランチ（簡易WANゾーン） | br-edge／br-pc（NAT配下、BGP/OSPF不参加の最小構成） |

詳細な設計根拠（書籍引用）は [02_基本設計/基本設計書.md](../02_基本設計/基本設計書.md) を参照。

## 6. 制約

- arm64（Apple Silicon + OrbStack）+ KVMなし環境。x86 VM型イメージ（RouterOS CHR）は
  QEMU TCG完全エミュレーションとなり、起動に`QEMU_CPU=max`環境変数が必須。
- Cisco IOL（hq-core／hq-sw）はGNS3世代イメージのためSSH不可。`docker attach --sig-proxy=false`の
  コンソール運用のみで、離脱は`Ctrl-p`→`Ctrl-q`。
- OpenWrt（`openwrt/rootfs`）コンテナはnetifdが動作しないため、`uci`による設定変更は実際のネット
  ワークスタックに反映されない。本ラボでは`uci show`等の**閲覧デモに限定**し、実際のIP付与・NAT
  設定は`ip`／`nft`コマンドを直接投入する方式とする。将来VM版OpenWrtへ切り替えれば本来の
  `uci commit`/`revert`運用を再現できる（README拡張案参照）。
- IOL 2台構成（hq-core／hq-sw）は当初router-on-a-stick（hq-core側dot1Qサブインターフェース＋
  hq-sw側トランク）で設計したが、スモークテストで不安定と判明したため、SVI方式
  （hq-sw側にVLAN10/20のSVIを置き、hq-core⇔hq-sw間はルーテッドリンク）へ変更した
  （23番ラボ collapsed core の実績方式）。詳細は基本設計書の設計判断の記録を参照。
- IOL・RouterOS CHRの起動完了までデプロイ全体で約2〜3分を要する。

## 7. 受け入れ基準

- 全13ノードが`deploy.sh inspect`で`running`になること。
- hq-edgeがisp-a・isp-b双方とeBGPを確立し、`show`/`print`系コマンドでEstablishedを確認できること。
- hq-core⇔hq-sw間のOSPF隣接がFull（またはRouterOSの2-Way/Full相当）になり、hq-sw配下の
  VLAN10/VLAN20サブネットがhq-edge側の経路テーブルに現れること。
- dc-leaf1・dc-leaf2それぞれの自LANサブネットがhq-core経由でHQ側から到達できること。
- br-pcからインターネット役（isp-a／isp-bのループバック等）へNAT経由で疎通できること。
- 5ベンダー×8操作の比較表（[03_詳細設計/ベンダー比較_同じ操作5流儀.md](../03_詳細設計/ベンダー比較_同じ操作5流儀.md)）が
  実測または一次資料（RouterOS REST／SR Linux JSON-RPCは実証済み、IOLはconsole手順）に基づいて
  埋まっていること。
