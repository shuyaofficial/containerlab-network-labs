07. 障害トリアージAI

:8607 補助 最終更新 2026-07-17

症状と証拠（show出力・ログの断片）を貼るだけで、OSI下位層優先で原因仮説をランク付けし、次に叩くべき読み取り専用コマンドまで添えて返すAIアプリ。決定論的なツールでは担えない“推論”の部分を補う相棒。

export ANTHROPIC_API_KEY=sk-ant-...
cd /Users/shuya/Documents/claude/ネットワークツール/07_triage && bash start.sh

このラボでの位置づけ#

補助 機器には接続せず、貼り付けたテキストのみを扱うため6ベンダーすべてに対応できる。ただしANTHROPIC_API_KEYが必須で、貼り付けた内容はAnthropic APIへ送信される（外部送信）。

セットアップ（登録する値）#

機器接続の登録は不要。環境変数ANTHROPIC_API_KEYを起動前に設定するだけでよい。証拠として貼り付ける出力の採取元は06_terminal（SSH機器）やdocker attach（IOL）。

使い方ステップ#

シナリオ例: hq-edge（RouterOS）とhq-core（IOL）の間でOSPFネイバーがFULLにならない。

  1. ANTHROPIC_API_KEYを設定して起動し、http://127.0.0.1:8607 を開く。
  2. 「症状」欄に「hq-edge(RouterOS)とhq-core(IOL)間のOSPFネイバーがFULLにならない」のように記入する。
  3. 06_terminalでhq-edgeに接続し /routing/ospf/neighbor print を実行、ssh clab@orb → docker attach --sig-proxy=false clab-mvlab-hq-core でhq-coreに接続し show ip ospf neighbor を実行する。両方の出力をコピーする。
  4. 07_triageの「証拠」欄に両方の出力を貼り付け、「補足」欄に機器・ベンダー（RouterOS 7.21.4 / IOL 15.7.3M2）を記入する。
  5. 解析をクリック（またはCmd/Ctrl+Enter）。思考の要約がストリーミングされたのち、確信度・レイヤー・根拠つきの仮説がランク付けされて表示される。
  6. 提示された確認コマンド（読み取り専用）をクリックしてコピーし、実機（hq-edgeまたはhq-core）で実行する。
  7. 結果を「確認コマンドの結果を貼って続ける」欄に貼り付け、続けて解析で仮説を更新する（相棒ループ、何度でも継続可）。
  8. 原因が特定できたらこのセッションを保存で記録に残す。設定変更の反映は05_configで版管理する。

制約と注意#

APIキー必須・外部送信: ANTHROPIC_API_KEYが未設定だとモデルバッジが赤くなり解析できない。貼り付けた症状・証拠・補足はAnthropic APIに送信されるため、パスワードなどの機密情報は含めないこと。
提示コマンドは読み取り専用: 確認コマンドは設定変更を伴わないコマンドのみが提示される仕様。実行結果の解釈は最終的に人間が行う。
