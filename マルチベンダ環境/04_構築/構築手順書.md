# 構築手順書: マルチベンダ環境

対象トポロジ: `multivendor.clab.yml`（ラボ名 `mvlab`、mgmt `172.20.50.0/24`、コンテナ名prefix `clab-mvlab-`）。
IP/AS/Router IDは [../02_基本設計/IPアドレス管理表.md](../02_基本設計/IPアドレス管理表.md) を正とする。

## 0. 構成サマリ（5ベンダー13ノード）

| ノード | kind / image | 役割 | config投入方式 |
|---|---|---|---|
| isp-a / isp-b | linux / `quay.io/frrouting/frr:10.2.1` | ISP（eBGP） | binds（daemons + frr.conf） |
| hq-edge | mikrotik_ros / `vrnetlab/mikrotik_routeros:7.21.4` | WANエッジ（デュアルホームeBGP+OSPF） | `deploy.sh config`（ブート後sshで1行投入） |
| hq-core | cisco_iol / `vrnetlab/cisco_iol:15.7.3M2` | キャンパスコア（OSPF/eBGP境界） | startup-config（`.cfg`→config.txt） |
| hq-sw | cisco_iol type:l2 / `L2-advipservices-2017` | SVIゲートウェイ（VLAN10/20） | startup-config（`.cfg`→config.txt） |
| dc-leaf1 / dc-leaf2 | nokia_srlinux / `ghcr.io/nokia/srlinux:latest` | DCリーフ（eBGP） | `deploy.sh config`（ブート後sr_cliで一括commit） |
| br-edge | linux / `openwrt/rootfs:armsr-armv8-24.10.7` | ブランチCPE（NAT） | `exec`直叩き（netifd不動作のため） |
| hq-pc1/hq-pc2/srv-web/srv-db/br-pc | linux / `wbitt/network-multitool:latest` | 端末 | `exec`（IP+default経路） |

> **2026-07-17実機テストで判明**: RouterOS(hq-edge)とSR Linux(dc-leaf1/2)は
> `startup-config`（RouterOSはFTP自動投入、SR Linuxはclabバインド）が
> 不達・構文エラーで全滅した（IOL/FRRのbind/startup-configは正常動作）。
> このため両ベンダーのみ「ブート後にコンテナ外からconfigを後投入」する方式へ
> 変更した（`deploy.sh deploy` の最後に自動実行、単独では `deploy.sh config`）。

## 1. デプロイ前チェック

```bash
free -h
```

- IOL 2台（hq-core / hq-sw）＝約170〜220MiB/台、RouterOS CHR（hq-edge）＝x86 QEMUエミュレーションで数百MiB〜、
  SR Linux 2台・FRR 2台・OpenWrt・端末5台はarm64ネイティブで軽量。OrbStack VM 15GBなら十分な余裕がある。
- **必ず `./deploy.sh deploy` 経由でデプロイする**（`containerlab deploy` を直接叩くと初回は `/opt/clab/.iourc` が
  無くIOLのbindが失敗する）。

## 2. デプロイ

```bash
ssh clab@orb
cd /Users/shuya/Documents/claude/Mac仮想環境構築/マルチベンダ環境/04_構築
./deploy.sh deploy
```

- `deploy.sh deploy` は、デプロイ前にVM内のIOLイメージ（`15.7.3M2` / `L2-advipservices-2017`）から
  ライセンス値を抽出し `/opt/clab/.iourc`（mode 600）へ書き出す。`multivendor.clab.yml` が全IOLノードへ
  `/opt/clab/.iourc:/iol/.iourc:ro` をbindし、`entrypoint_attach.sh` の「有効値なら尊重」ロジックで使われる。
- ライセンス値はVM内にのみ存在させる。**gitリポジトリ・Mac側へコピーしないこと**。

## 3. config投入方式（ベンダー別）

- **FRR（isp-a/isp-b）**: `binds` で `configs/isp-X.daemons → /etc/frr/daemons`、
  `configs/isp-X.frr.conf → /etc/frr/frr.conf`。IPは frr.conf の `interface` 節で付与（zebra が設定）。
  daemons は bgpd のみ有効（clab公式 frr01 と同型）。frr.conf に `no bgp ebgp-requires-policy` を含め、
  FRR 9+ のRFC8212既定（eBGPはポリシー必須）を無効化している。**実機で正常動作を確認済み（変更不要）**。
- **RouterOS（hq-edge）**: `startup-config`（FTP自動投入）は不採用。`deploy.sh` が
  hq-edgeのブート完了（`/system/resource print` が通る）をポーリングで待ってから、
  `configs/hq-edge.rsc` を **1行ずつssh実行**する（`#`行・空行はスキップ）。
  実機で判明した必須事項:
  - **インラインコメント厳禁**（`#`は行頭のみ）。1行ずつssh実行する方式では行末コメントが
    その行の構文エラーになる。
  - `/routing/bgp/instance` は **`default`という既定インスタンスが存在しない**ため
    `set default ...` は失敗する。`add name=mv as=65010 router-id=...` で**専用インスタンスを新規作成**し、
    `/routing/bgp/connection` の `instance=mv` で参照する。
  - `/routing/bgp/connection` は **`instance=` と `local.role=ebgp` が必須**（無いと
    `missing value(s) of argument(s) instance local.role` で失敗）。`connection` に `router-id=` という
    プロパティは無い（Router IDは `instance` 側で指定する）。
  - 証明書生成/`www-ssl`（https）は非採用。`/ip service set www disabled=no`（http）のみで
    REST APIは動作する（証明書署名の非同期タイミングが不安定だったため）。
  - v7書式は `/routing/ospf/instance`+`area`+`interface-template`、`/routing/bgp/instance`+`connection`、
    networkは `address-list`(`bgp-nets`)+`blackhole`経路+`output.network`。
  - clab eth割当は **eth1=ether2 / eth2=ether3 / eth3=ether4**。
- **Cisco IOL（hq-core/hq-sw）**: `startup-config: configs/hq-X.cfg`（clab が config.txt にbind）＋
  `binds`（entrypoint_attach.sh、/opt/clab/.iourc）。**IOLはSSH不可・console運用のため aaa/ssh 設定は入れない**。
  hq-sw は L2イメージでの確実動作のため、hq-core向けアップリンクを専用トランジットVLAN99+SVIでL3化している。
  **実機で正常動作を確認済み（変更不要）**。
- **SR Linux（dc-leaf1/2）**: `startup-config`（clabバインド）は**コンテナに届かず未適用**だったため不採用。
  `deploy.sh` が `configs/dc-leafX.cli` から `#`行・空行を除いた `set` 行を集め、
  `"enter candidate" → set行群 → "commit stay" → "quit"` を **`docker exec -i <container> sr_cli` へ
  stdinパイプで渡し一括commit**する（引数渡しではなくstdinパイプが必須。実機で
  `All changes have been committed` を確認済み）。
  **RFC8212対策として export（自prefix accept）/ import（accept-all）ポリシーを必ず含める**。subinterfaceに `ip-mtu 1500`。
- **OpenWrt（br-edge）**: netifd不動作のため `exec` 直叩き。IP付与→default経路→`ip_forward`→
  nft によるmasquerade SNAT。nftのbase chain宣言（波括弧）は `sh -c '...'` でクォートして1コマンドとして渡す。
  **実機確認**: `ip route add default` はclabが先に入れているmgmt側default(172.20.50.1)と競合して失敗し
  WAN向け経路が入らなかったため、端末(multitool)と同じ **`ip route replace default via <GW>`** に統一した。
  nft 3行（table/chain/masquerade）はそのまま追加できることを確認済み。
- **端末（multitool）**: `exec` で eth1 にIP付与＋`ip route replace default via <GW> dev eth1`。
  default はLAN側へ張り替えるが、mgmt(172.20.50.0/24)は接続経路として残るため `docker exec` は常時可能。

> 全ベンダー間P2Pリンクに `mtu: 1500` を明示（clab既定9500によるOSPF/BGPのMTUミスマッチ回避）。端末リンクも1500で統一。

## 4. 起動確認

```bash
./deploy.sh status      # 13コンテナのStatus一覧
./deploy.sh inspect     # containerlab inspect
```

- 13ノード全てが `running` であること。
- IOL 2台はブートログに `License Error` が無いことを確認する。
  ```bash
  sudo docker logs --tail 50 clab-mvlab-hq-core
  ```
- `entrypoint_attach.sh` がIOLブート完了後（+約60秒）に `iouyap` を自動起動する。自動起動が働かない場合のみ
  手動リカバリ `./deploy.sh iouyap`（対象は hq-core / hq-sw の2台。hq-edge=MikroTik と br-edge=OpenWrt は除外）。
- RouterOS CHR（hq-edge）は起動に約2分。`deploy.sh deploy` はこの起動待ちも兼ねて
  `deploy.sh config`（hq-edge / dc-leaf1 / dc-leaf2 へのconfig後投入）を自動実行する。
  - hq-edgeが120秒以内に応答しない等で自動投入が失敗した場合は、単独で再実行できる。
    ```bash
    ./deploy.sh config
    ```
  - BGP/OSPFの確立まで、config投入後さらに数十秒〜数分待つ。

## 6. 実機テストで確定した事項（2026-07-17）

以前は「要実機確認」としていた箇所を、実機テストで下記のとおり解決・確定した。

- **hq-edge.rsc**: `startup-config`（FTP自動投入）を廃止し `deploy.sh config` による
  ブート後ssh投入に変更（§3参照）。`/routing/bgp/instance` は専用インスタンス`mv`を新規作成する方式に、
  `/routing/bgp/connection` は `instance=mv` + `local.role=ebgp` 必須指定に確定。証明書/`www-ssl`は削除し、
  `www`(http)のみで運用する。この構文で isp-a/isp-b 双方とのBGPセッションestablishedを確認済み。
- **dc-leafX.cli**: `startup-config`（clabバインド）を廃止し `deploy.sh config` による
  ブート後sr_cli投入（stdinパイプ、candidateモード一括commit）に変更（§3参照）。
  BGP階層（`group`/`neighbor`/`afi-safi`/`export-policy`/`import-policy`）・`system json-rpc-server` は
  本ファイルの構文のまま commit 成功を確認済み。
- **br-edge（clab.yml exec）**: `nft` 3行（table/chain/masquerade）はそのまま追加可能なことを確認済み。
  唯一の問題はdefault経路の投入方式で、`ip route add default` がclab既定のmgmt側defaultと競合して
  失敗していたため `ip route replace default via 10.50.255.13` に修正（§3参照）。

## 7. destroy時の注意

```bash
./deploy.sh destroy
```

- 内部で `--cleanup` 付き `containerlab destroy` を実行するため、**IOLのnvram（保存済みコンフィグ）ごと削除される**。
- 設定を残したまま一旦壊す場合は `--cleanup` なしで直接実行する。
  ```bash
  sudo containerlab destroy -t multivendor.clab.yml
  ```

## 9. 実機構築で判明した課題と対策（2026-07-17）

実機デプロイで発見・解決した技術課題を、将来の再現・トラブルシュートのために一覧化する
（§3・§6の記述の要点を課題→対策の形にまとめ直したもの。詳細な構文は[configs/](configs/)配下の
各ファイルを参照）。

| No | 課題 | 事象 | 対策 |
|---|---|---|---|
| 1 | RouterOS v7.21 BGP構文 | `/routing/bgp/instance set default ...` は失敗する（`default`という既定インスタンスが存在しない）。`/routing/bgp/connection` は `instance=` と `local.role=ebgp` が無いと `missing value(s) of argument(s) instance local.role` で失敗する | `/routing/bgp/instance add name=mv as=<AS> router-id=<ID>` で専用インスタンスを先に作成し、`connection` 側で `instance=mv` と `local.role=ebgp` を明示指定する（[configs/hq-edge.rsc](configs/hq-edge.rsc)） |
| 2 | RouterOS `.rsc` 投入方式 | `deploy.sh` が1行=1sshセッションで投入するため、RouterOS流の「パス行→bare `add`」のブロック記法は2行目がrootコンテキストで実行され失敗する | 各行を「パス+コマンド」で完結させる（例: `/ip address add address=... interface=...`）。インラインコメント（行末`#`）も同じ理由で構文エラーになるため使用しない |
| 3 | `deploy.sh` の `ssh -n` 必須 | `push_routeros` の `while` ループ内で `ssh` に `-n` を付けないと、ssh が標準入力（`.rsc`ファイル自体）を食い潰し2行目以降が消える（addr=2件のみ投入されBGP設定が丸ごと欠落する形で再現） | ループ内sshは必ず `ssh -n ...` にする（[deploy.sh](deploy.sh) `push_routeros()`） |
| 4 | SR Linux configの配信経路 | clabの `startup-config`（バインド）がコンテナに届かず未適用のまま起動する | `configs/dc-leafN.cli` の `set` 行を集め、`enter candidate` → `set` 行群 → `commit stay` → `quit` を `docker exec -i <container> sr_cli` へ**stdinパイプ**で渡し一括commitする（引数渡しは不可）。routing-policyのprefixマッチは `match prefix prefix-set <name>`（v26.3.3構文）。BGPのAS/router-id/admin-state等は同一candidateで一括commitが必須（[deploy.sh](deploy.sh) `push_srl()`、[configs/dc-leaf1.cli](configs/dc-leaf1.cli)） |
| 5 | SR Linux subinterfaceのMTU競合 | clabが `link mtu:1500` をポートMTU（port-mtu）にも適用するため、subinterfaceの `ip-mtu 1500` と衝突し `ip-mtu-too-large` でサーバ向けインターフェースがdownする（HQ↔DC不通の主因） | ポート自体のMTUは `set / interface ethernet-1/X mtu 9000` へ引き上げ、`ip-mtu 1500` はsubinterface側にのみ残す（[configs/dc-leaf1.cli](configs/dc-leaf1.cli)） |
| 6 | IOLの初期設定ダイアログ | GNS3世代のCisco IOL（hq-core/hq-sw）はブート時に `Would you like to enter the initial configuration dialog? [yes/no]:` で停止し、応答するまで `startup-config`（config.txt）をロードせずOSPF/BGPも起動しない | `deploy.sh` の `answer_iol_dialog()` が `expect` で `docker attach` 経由で `no` を自動応答する。「no」応答の瞬間に全隣接がFULL/Upになることを実機確認済み |
| 7 | OpenWrt NATのnftルール投入 | `nft` のbase chain宣言（波括弧を含む）をclab `exec` へ直接渡すと `unexpected }` で壊れる | ルールセットを `configs/br-edge-nat.nft` としてファイル化し、`binds` でコンテナへ読み取り専用マウントした上で `nft -f /etc/br-edge-nat.nft` で読み込む（[multivendor.clab.yml](multivendor.clab.yml) br-edge） |

> 上記の課題1・2・4・5・6・7は初回実装時点で不採用/未確定だった構成方式に対応する。§3・§6の各ベンダー節は
> 対策反映後の最終手順を示す。
