4. 冗長性・拡張性#

項番設計項目決定方針対応要件ID
4.1WAN冗長化hq-edgeはisp-a・isp-bの2社に対してeBGPデュアルホームを組む。片方の回線・BGPセッションが落ちても、残るISPからのデフォルトルートで継続到達できる。REQ-MV02
4.2キャンパス内冗長本ラボの対象外。hq-core・hq-swは各1台構成（HSRP/STP二重化は組まない）。ベンダー間の操作比較に主眼を置くため将来課題とする。REQ-MV01
4.3拡張性DCリーフはeBGPで自LANを広報するだけのシンプル構成のため、dc-leaf3以降の追加はhq-core側にeBGPセッションを1本追加するだけで対応できる。REQ-MV03

デュアルホーム切替の状態遷移は『ネットワーク・デザインパターン』p095「メイン回線がダウンした場合…ISP#2から受け取ったデフォルトルートをベストパスにして、ルーティングテーブルに載せます」の記述に沿う。isp-a向けリンク断での切替は試験結果報告書の冗長試験で検証済み。

6. 設計判断（方式比較）#

実機のスモークテストで得た知見から、当初設計を変更した2つの重要な判断を記録する。

6.1 VLANゲートウェイ方式（router-on-a-stick からの変更） REQ-MV04

比較観点案A: hq-sw SVI方式採用案B: router-on-a-stick（dot1QサブIF）
安定性hq-sw側にVLAN10/20のSVIを置き、hq-core⇔hq-sw間はルーテッドリンク。IOL＋containerlab veth環境で安定動作を確認hq-core側でdot1Qサブインターフェース終端＋hq-sw側L2トランク。スモークテストで不安定
実績23番ラボ23_campus_collapsed_coreの実績方式を踏襲本環境ではdot1Qサブインターフェース側の相性問題が疑われる
ネイティブVLAN整合ルーテッドリンクのためタグ整合の依存が少ないタグVLAN/ネイティブVLANの整合が接続機器間で必須（不一致で通信不可）
採用理由router-on-a-stick方式はスモークテストで不安定と判明したため、23番collapsed coreで実績のあるSVI方式（hq-sw側SVI＋hq-core⇔hq-sw間ルーテッドリンク）を採用した。原因の切り分けは構築ログ側の課題とし、本書ではSVI方式を確定値とする。

6.2 config投入方式 REQ-MV07

比較観点案A: clab startup-config案B: ブート後にssh／sr_cliで後投入採用
RouterOS (hq-edge)FTP自動投入がエラー行でimportを丸ごと中断し、IP設定すら入らなかったブート完了をポーリング待ちし、hq-edge.rscを1行ずつssh実行
SR Linux (dc-leaf1/2)clabバインドがコンテナに届かず未適用だったdc-leafX.cliのset行をcandidateモードで一括commit（stdinパイプ）
IOL / FRRstartup-config／binds は正常動作（変更不要）IOL/FRRは案Aのまま。後投入は不要
採用理由RouterOS・SR Linuxのstartup-configは実機でFTP中断／未達により全滅したため、両ベンダーのみ「ブート後にコンテナ外からconfigを後投入」する方式（deploy.sh config）へ変更した。確実にconfigが届く後投入方式を採用する。IOL/FRRは従来どおりstartup-config/bindsを継続。
