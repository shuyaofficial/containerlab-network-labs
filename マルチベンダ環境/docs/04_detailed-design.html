RouterOShq-edge#

MikroTik RouterOS CHR 7.21.4、AS65010。isp-a／isp-bとeBGPデュアルホーム、hq-coreとOSPF area0。BGP学習defaultをOSPFへ配布する。 REQ-MV02REQ-MV06

項目仕様
インターフェースether2=10.50.255.6/30(isp-a) / ether3=10.50.255.10/30(isp-b) / ether4=10.50.255.17/30(hq-core)
BGP AS / Router ID65010 / 10.50.255.101（instance mv）
eBGP peerisp-a=10.50.255.5(AS65001) / isp-b=10.50.255.9(AS65002)。両ISP対等、通常のベストパス選択に委ねる
広報10.50.0.0/16（address-list bgp-nets + blackhole経路）
OSPFarea0、ether4のみ参加。originate-default=alwaysでdefaultをHQ内へ配布
API/ip service set www disabled=no（REST http有効化）
投入時の注意connectionはinstance=とlocal.role=ebgpが必須。1行=1 sshセッションで投入、インラインコメント厳禁
configs/hq-edge.rsc（抜粋）
# インターフェースIP（clab eth割当: eth1=ether2 / eth2=ether3 / eth3=ether4）
/ip address add address=10.50.255.6/30 interface=ether2 comment=to-isp-a
/ip address add address=10.50.255.10/30 interface=ether3 comment=to-isp-b
/ip address add address=10.50.255.17/30 interface=ether4 comment=to-hq-core
# OSPF area0（ether4=hq-core向け。default経路をHQ内へ配布）
/routing/ospf/instance add name=v2 version=2 router-id=10.50.255.101 originate-default=always
/routing/ospf/area add name=backbone area-id=0.0.0.0 instance=v2
/routing/ospf/interface-template add interfaces=ether4 area=backbone
# BGP広報用の自社集約（10.50.0.0/16。blackholeで経路存在を担保）
/ip firewall address-list add list=bgp-nets address=10.50.0.0/16
/ip route add dst-address=10.50.0.0/16 blackhole comment=bgp-aggregate
# BGPインスタンス（v7.21はconnectionがinstance必須）＋ eBGPデュアルホーム
/routing/bgp/instance add name=mv as=65010 router-id=10.50.255.101
/routing/bgp/connection add name=to-isp-a instance=mv remote.address=10.50.255.5 remote.as=65001 local.role=ebgp output.network=bgp-nets
/routing/bgp/connection add name=to-isp-b instance=mv remote.address=10.50.255.9 remote.as=65002 local.role=ebgp output.network=bgp-nets
# REST API（API試験用。http有効化のみ）
/ip service set www disabled=no

Cisco IOLhq-core#

Cisco IOL 15.7.3M2、AS65011。OSPF（hq-edge/hq-sw）とeBGP（dc-leaf1/2）の再配信境界。DCからeBGPで学んだ/24をOSPFへ再配信し、DCリーフへはdefaultをdefault-originate。SSH不可・console運用のためaaa/ssh設定は入れない。 REQ-MV03

項目仕様
インターフェースEt0/1=.18(hq-edge) / Et0/2=.21(dc-leaf1) / Et0/3=.25(dc-leaf2) / Et1/0=.29(hq-sw)（すべて/30）
OSPF Router ID10.50.255.102（area0: Et0/1・Et1/0が参加）
再配信redistribute bgp 65011 subnets（DCの/24をOSPFへ）。DC向け/30はOSPFに載せない
BGP AS / Router ID65011 / 10.50.255.102
eBGP peerdc-leaf1=.22(AS65021) / dc-leaf2=.26(AS65022)。両者へdefault-originateのみ
configs/hq-core.cfg（抜粋）
interface Ethernet0/1
 description to-hq-edge
 ip address 10.50.255.18 255.255.255.252
interface Ethernet1/0
 description to-hq-sw
 ip address 10.50.255.29 255.255.255.252
! OSPF: DCから eBGP で学んだ /24 を OSPF へ再配信し HQ内へ伝える
router ospf 1
 router-id 10.50.255.102
 redistribute bgp 65011 subnets
 network 10.50.255.16 0.0.0.3 area 0
 network 10.50.255.28 0.0.0.3 area 0
! eBGP: 両 DC リーフへ default-originate のみ
router bgp 65011
 bgp router-id 10.50.255.102
 neighbor 10.50.255.22 remote-as 65021
 neighbor 10.50.255.26 remote-as 65022
 address-family ipv4
  neighbor 10.50.255.22 activate
  neighbor 10.50.255.22 default-originate
  neighbor 10.50.255.26 activate
  neighbor 10.50.255.26 default-originate
 exit-address-family

Cisco IOLhq-sw（SVIゲートウェイ）#

Cisco IOL L2-advipservices-2017。ip routing有効＋VLAN10/20 SVI＋OSPF area0参加（23番ラボ実績のSVIルーティング方式）。hq-core向けアップリンクも専用トランジットVLAN99+SVIでL3化し、L2イメージでも確実に動作させる。 REQ-MV04

項目仕様
SVI（GW）Vlan10=10.50.10.1/24 / Vlan20=10.50.20.1/24 / Vlan99=10.50.255.30/30(hq-core transit)
アクセスポートEt0/1=VLAN99(hq-core) / Et0/2=VLAN10(hq-pc1) / Et0/3=VLAN20(hq-pc2)
OSPF Router ID10.50.255.105（area0）
OSPF参加Vlan10/Vlan20はpassive-interface（端末へHelloを流さない）、Vlan99はアクティブ

vlan 20

vlan 99

SR Linuxdc-leaf1 / dc-leaf2#

Nokia SR Linux v26.3.3。dc-leaf1=AS65021(srv-web)／dc-leaf2=AS65022(srv-db)。hq-coreとeBGP。SR LinuxはRFC8212既定で無広報・無受理のため、export（自prefix accept）/import（accept-all）ポリシーを必ず明示する。JSON-RPCをAPI試験用に有効化。 REQ-MV03REQ-MV06

項目仕様（dc-leaf1 / dc-leaf2）
インターフェースe1-1=10.50.255.22 or .26 /30(hq-core) / e1-2=10.50.30.1 or 10.50.31.1 /24(GW)
BGP AS / Router ID65021 / .103（leaf1） · 65022 / .104（leaf2）
eBGP peer10.50.255.21 or .25（AS65011 hq-core）
export-policy自LAN（10.50.30.0/24 or 10.50.31.0/24）のみ広報（RFC8212対策で必須）
import-policyhq-coreからのdefault route含む全受理（accept-all）
APIsystem json-rpc-server（mgmt VRF上でhttp有効化）
configs/dc-leaf1.cli（抜粋）
# インターフェース（e1-1=hq-core向け / e1-2=srv-web向けGW）
set / interface ethernet-1/1 subinterface 0 ipv4 address 10.50.255.22/30
set / interface ethernet-1/2 subinterface 0 ipv4 address 10.50.30.1/24
# routing-policy（RFC8212対策: export=自LANのみ / import=全許可）
set / routing-policy prefix-set lan-prefix prefix 10.50.30.0/24 mask-length-range exact
set / routing-policy policy export-lan statement 10 match prefix prefix-set lan-prefix
set / routing-policy policy export-lan statement 10 action policy-result accept
set / routing-policy policy import-all default-action policy-result accept
# BGP（AS65021、対 hq-core eBGP）
set / network-instance default protocols bgp autonomous-system 65021
set / network-instance default protocols bgp router-id 10.50.255.103
set / network-instance default protocols bgp group ebgp-hq peer-as 65011
set / network-instance default protocols bgp group ebgp-hq export-policy [ export-lan ]
set / network-instance default protocols bgp group ebgp-hq import-policy [ import-all ]
set / network-instance default protocols bgp neighbor 10.50.255.21 peer-group ebgp-hq
# JSON-RPC サーバ（API試験用。mgmt VRF上で http 有効化）
set / system json-rpc-server admin-state enable
set / system json-rpc-server network-instance mgmt http admin-state enable

FRRisp-a / isp-b#

FRRouting 10.2.1。isp-a=AS65001／isp-b=AS65002。互いをeBGPでピアし、hq-edgeともeBGP。ループバック（198.51.100.1/2）を「インターネット上のサービス役」として広報する。isp-bはbr-edge向けP2P（10.50.255.12/30）も広報し、ブランチNAT復路の到達性を確保する。 REQ-MV01

項目仕様
Loopbackisp-a=198.51.100.1/32 / isp-b=198.51.100.2/32（インターネット役）
eBGP peer相互（.1⇔.2）＋ hq-edge（isp-a=.6 / isp-b=.10、AS65010）
広報lo/32・自ISPのP2Pセグメント。isp-bは10.50.255.12/30も広報（NAT復路）
RFC8212no bgp ebgp-requires-policyでFRR9+のポリシー必須既定を無効化（ラボ簡素化）
br-edge向けeBGPは張らない（isp-b:eth3 は connected のみ、static+NAT運用）
configs/isp-a.frr.conf（抜粋）
! インターフェース（zebra がIP付与）
interface lo
 ip address 198.51.100.1/32
interface eth2
 description to-hq-edge
 ip address 10.50.255.5/30
! eBGP（AS65001）
router bgp 65001
 bgp router-id 198.51.100.1
 no bgp ebgp-requires-policy
 neighbor 10.50.255.2 remote-as 65002
 neighbor 10.50.255.6 remote-as 65010
 address-family ipv4 unicast
  network 198.51.100.1/32
  network 10.50.255.0/30

  neighbor 10.50.255.6 activate

OpenWrtbr-edge#

OpenWrt 24.10.7（rootfs）。BGP/OSPF不参加。isp-bへの静的デフォルトルート＋masquerade（source-NAT）でブランチLANを外部到達させる。netifd不動作のためip／nftを直接投入する（uciは閲覧デモのみ）。 REQ-MV05

項目仕様
インターフェースeth1(WAN)=10.50.255.14/30(GW .13) / eth2(LAN)=192.168.40.1/24
デフォルトルートip route replace default via 10.50.255.13（addはmgmt側defaultと競合するためreplace）
NAT192.168.40.0/24 → eth1へ masquerade（nft -f で読み込み）
uciの扱い閲覧デモのみ。ip/nft直接投入のためコンテナ再起動で揮発
multivendor.clab.yml exec + configs/br-edge-nat.nft（抜粋）
# exec 直叩き（IP付与 → default経路 → ip_forward → masquerade）
ip address add 10.50.255.14/30 dev eth1
ip address add 192.168.40.1/24 dev eth2
ip route replace default via 10.50.255.13
sysctl -w net.ipv4.ip_forward=1
nft -f /etc/br-edge-nat.nft
# --- br-edge-nat.nft（波括弧を含むためファイルで読み込む） ---
table ip nat {
  chain postrouting {
    type nat hook postrouting priority 100; policy accept;
    oifname "eth1" masquerade
  }
}

共通共通設計 ・ 同じ操作の5流儀#

全機器に共通する設計方針と、5ベンダーで「同じ目的の操作」がどう異なるかの対比（一次資料・実測に基づく）。 REQ-MV06REQ-MV07

項目値
IGPOSPF単一エリア0（hq-edge／hq-core／hq-swの3台のみ参加）
EGPeBGPのみ（iBGP不使用。各ASは1台構成）
Router ID方針ループバック未使用。経路上に現れない識別子専用の値（10.50.255.101〜.105）を明示設定
MTU全P2Pリンクに mtu 1500 を明示（clab既定9500によるOSPF/BGP MTUミスマッチ回避）
管理アクセスRouterOS=SSH+REST / IOL=console（SSH不可） / SR Linux=SSH+JSON-RPC / FRR=docker exec+vtysh / OpenWrt=docker exec+ip/nft

同じ目的の操作を5流儀で対比（03_詳細設計/ベンダー比較_同じ操作5流儀.md 実測ベース）

操作各ベンダーの流儀
ルーティングテーブルRouterOS: /ip route print · IOL: show ip route · SR Linux: show network-instance default route-table · FRR: vtysh -c "show ip route" · OpenWrt: ip route show
BGPピア状態RouterOS: /routing/bgp/session print · IOL: show ip bgp summary · SR Linux: show ... bgp neighbor · FRR: vtysh -c "show ip bgp summary"
設定変更のロールバックRouterOS=Safe Mode（切断で自動破棄） · SR Linux=commit confirmed（タイマー式自動ロールバック） · IOS/FRR=手動(no系) · OpenWrt=imperative（手動）
保存RouterOS: 即時自動 · IOL: copy run start · SR Linux: commit save · FRR: write memory · OpenWrt: ip/nft直投入は揮発
APIでの取得RouterOS=REST（実証済み） · SR Linux=JSON-RPC（実証済み） · IOL=console限定（SSH不可） · FRR=vtysh JSON出力 · OpenWrt=netifd不動作で対象外
前の工程トポロジ図 次の工程試験結果報告書
