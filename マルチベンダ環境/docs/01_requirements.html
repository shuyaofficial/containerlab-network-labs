1. 背景・目的#

実務のネットワークは1社であっても単一ベンダーで統一されているとは限らない。ISP接続・WANエッジ・キャンパス・DC・ブランチCPEといったゾーンごとに、コスト・機能・運用性の観点で異なるベンダーが選定される。本ラボは「1社の企業ネットワークを、ゾーンごとに最適ベンダーで構成する」という前提をラボ化する。

  1. ゾーンごとに実務で採用されやすいベンダーを意図的に割り当て、相互接続する
  2. 同じ目的の操作（経路確認・ピア状態確認・設定投入・API取得）の流儀差を実機で体感する
  3. 外側はeBGP・内側はOSPF・DCは再びeBGPという企業ネットワークの階層化を踏襲する
  4. API-firstな機器（RouterOS REST／SR Linux JSON-RPC）での状態取得・可逆変更を実証する

2. 対象範囲#

対象

  • 5ゾーン: Untrust（ISP2社）／WANエッジ／Trust（HQキャンパス）／DC（リーフ×2）／ブランチ（CPE+NAT）
  • プロトコル: eBGP（AS間）・OSPF（HQ内 area0）・VLAN/SVI・静的経路+NAT（ブランチ）
  • 5ベンダー: FRR 10.2.1／RouterOS CHR 7.21.4／Cisco IOL 15.7.3M2＋L2-advipservices-2017／Nokia SR Linux v26.3.3／OpenWrt 24.10.7
  • 8操作の横断比較（IF一覧／経路表／BGP／OSPF／設定表示／変更・ロールバック／保存／API）

対象外

  • ACL/ファイアウォールルール（27番テーマが別途担当）
  • 実トラフィック負荷試験・SNMP/監視基盤の統合
  • キャンパス内の機器冗長（HSRP/STP二重化。将来課題）
  • NX-OSノード拡張・VM版OpenWrtへの切替（README拡張案のみ）

6. 制約・前提#

  • arm64（Apple Silicon + OrbStack）+ KVMなし環境。x86 VM型イメージ（RouterOS CHR）はQEMU TCG完全エミュレーションとなり、起動にQEMU_CPU=maxが必須。
  • Cisco IOL（hq-core／hq-sw）はGNS3世代イメージのためSSH不可。docker attach --sig-proxy=falseのコンソール運用のみで、離脱はCtrl-p→Ctrl-q。
  • OpenWrt（rootfs）コンテナはnetifdが動作しないため、uciによる設定変更はネットワークスタックに反映されない。uci show等の閲覧デモに限定し、IP付与・NATはip／nftを直接投入する。
  • IOL 2台構成は当初router-on-a-stick（dot1QサブIF＋トランク）で設計したが、スモークテストで不安定と判明したため、SVI方式（23番ラボ collapsed core の実績方式）へ変更した。
  • RouterOS／SR Linuxのstartup-config（FTP自動投入／clabバインド）は実機で不達・構文エラーとなったため不採用とし、ブート後にssh／sr_cliで後投入する方式へ変更した。
