症状全容マトリクス
----------------------------------------------------------------------

- 提示する初期観測: 「メイン回線(hq-edge1のISP側)は計画メンテで停止中（これは正常。触ってはならない）。設計上はサブ回線(hq-edge2)へ自動切替のはずが、本社PCからインターネット(8.8.8.8)が全断」
- 全容: HQ-PC→8.8.8.8 失敗 / HQ-PC→10.0.0.x（社内バックボーン）正常 / core経路表: defaultはedge2向けのみ（切替自体は成功している） / edge2 BGP Established・default受信済み / edge2 `show ip nat translations` 空・`show ip nat statistics` のinside interfaces欄が空
- VPN/支社について: hq-edge1のTunnel0はsource=e0/1のため、e0/1 shutで**落ちる**（=HQ-PC→支社10.2.40.0/24も不通）。これはメンテ（e0/1 shut）の当然の帰結であり本シナリオの隠し障害ではない。焦点はHQ→インターネットのみ。初期観測では支社/VPNに一切触れず、焦点をぼかさないようにしてある。

【GM運用注意】ブリーフィングにVPNの状態は書いていない。Tunnel0はe0/1 shutで落ちるのが正しい挙動。プレイ中にユーザーが支社/VPN不通に気づいたら「メンテ（e0/1停止）の既知影響であり、本件の調査対象外」とGMが回答すること。ここに誘導されてNAT以外を根本原因と宣言した場合は不一致とする。

----------------------------------------------------------------------
模範切り分けフロー
----------------------------------------------------------------------

1. 切替の成否をライブで確認: core1 `show ip route 0.0.0.0`（edge2向けdefaultのみ存在→切替は成功しているという事実確定。「切り替わっていない」という申告を疑う＝観測事実と伝聞の区別）
2. 反証: edge2 `show ip bgp summary`（Established）・`show ip route`（0.0.0.0/0 BGP受信）→「BGP/経路は無罪」を証明
3. 順方向実測: HQ-PC→traceroute 8.8.8.8（edge2まで届いてその先で消える）
4. NAT層: edge2 `show ip nat statistics`（inside interfaces欄が空＝insideタグ欠落）・`show ip nat translations`（空）
5. 差分比較: 正常機hq-edge1の `show run | include ip nat|interface` と比較（06-30教訓: 動いている正解形と差分を取る）
6. 根本原因宣言: 「hq-edge2の内側IF群から ip nat inside が外れており、切替後のPATが不発」
