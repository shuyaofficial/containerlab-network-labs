注入内容（平文）
----------------------------------------------------------------------

対象機器: br-edge のみ

br-edge:
conf t
no ip nat inside source list 1 interface Ethernet0/1 overload
ip nat source list 1 interface Ethernet0/1 overload
end

※ `inside` 欠落によりNVI解釈となり、if単位の inside/outside タグが無視されNAT不発。パケットは私設IPのままISPへ→ISPの `0.0.0.0 Null0` で破棄。

切り戻し（平文）
----------------------------------------------------------------------

br-edge:
conf t
no ip nat source list 1 interface Ethernet0/1 overload
ip nat inside source list 1 interface Ethernet0/1 overload
end

----------------------------------------------------------------------
症状全容マトリクス
----------------------------------------------------------------------

- 提示する初期観測: 「支社PC(10.2.40.100)から 8.8.8.8 へのpingが100%失敗。一方、本社営業部(10.1.10.100)への業務通信は正常」
- 全容: BR-PC→8.8.8.8 失敗 / BR-PC→200.0.3.254(ISP直結IF)も失敗（復路がNull0） / BR-PC→10.1.10.100 成功（VPN・NAT非関与） / br-edge自身→8.8.8.8 成功（自己発はNAT不要…IOSの自己発ICMPは送信元200.0.3.1のため成功。ユーザーの有力な切り分け材料） / HQ側インターネットも正常

Mistakes反映(準): テーマ21実史実（05_ipsec_nat_troubleshooting_log.md #1 のNVI罠を再演）。ログ由来であることは本封印ファイルにのみ記載する。

----------------------------------------------------------------------
模範切り分けフロー
----------------------------------------------------------------------

1. 事実採取: BR-PC `ip route`（default via 10.2.40.254 正常）→ br-edge `show ip route`（default→200.0.3.254 正常）→ ルーティング正常の証明
2. 反証: br-edge 自身から `ping 8.8.8.8`（成功）→ 経路・ISPは無罪。PC発だけ死ぬ→「送信元アドレスの差」に着目＝NAT層
3. `show ip nat translations`（空）／`show ip nat statistics`（Hits増えない・Dynamic mappings欄の異常）
4. `show run | include ip nat`（`ip nat source list`＝insideなしのNVI形を発見）
5. 根本原因宣言: 「NAT適用コマンドがNVI形（inside欠落）に変わっており、PATが不発」

----------------------------------------------------------------------
ヒント（3段階・全文）
----------------------------------------------------------------------

1. 方角: 「支社→インターネット方向のみ。本社・VPN・ISP自体は健全」
2. 層: 「ルーティングは正常だと自分で証明せよ。死んでいるのはL3の“変換”機能」
3. 観測ポイント: 「br-edgeで show ip nat translations / show ip nat statistics。running-configのip nat行を一言一句、正常機(hq-edge1)と見比べよ」
