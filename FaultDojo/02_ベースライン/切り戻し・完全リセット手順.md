# 切り戻し・完全リセット手順

プレイ終了後にベースラインへ復帰するための手順です。障害の程度に応じて3段階のレベルを使い分けます。

## 1. Level1: シナリオのrestore.sh実行（通常の切り戻し）

各シナリオフォルダ内の `restore.sh` を、プレイヤー本人が実行します。

```bash
sudo bash restore.sh
```

`inject.sh` の逆操作のみを行うスクリプトです。実行後は、[ベースライン構築手順.md](ベースライン構築手順.md) の「5. ベースライン健全性チェック」を再実行して復旧を確認してください（具体的な復旧確認コマンドの案内はネタバレになるため、restore.sh自体はチェックリストの再実行を促す表示のみを行います）。

## 2. Level2: コンテナ再起動（注入がrunning-configに留まっている場合）

障害注入はrunning-configのみを対象としており、ベースライン構築時に `write memory` 済みのstartup-configには影響しません。そのため、対象コンテナを再起動するとベースライン状態に復帰します。

```bash
docker restart clab-capstone-<node>
```

IOL機器は再起動に**2分以上**かかる場合があります。再起動後は、健全性チェックを再実行して復旧を確認してください。

> 注意: [ベースライン構築手順.md](ベースライン構築手順.md) の「6. 注意事項」の通り、プレイ中に `write memory` をしてしまった場合、この安全網は機能しません。その場合はLevel3を実施してください。

## 3. Level3: ラボの完全破棄・再構築

```bash
sudo containerlab destroy -t capstone.clab.yml --cleanup
```

実行後、[ベースライン構築手順.md](ベースライン構築手順.md) を最初からやり直します。

## 4. 異常系: ラボが正常に起動しない場合

### 4.1 コンテナ状態の確認

```bash
docker ps -a | grep clab-capstone
```

停止済み・異常終了したコンテナがないか確認します。

### 4.2 メモリ逼迫の確認

本ラボは6ノード×1GB程度のメモリを要求します。Mac側のメモリ逼迫がないか確認してください。

### 4.3 残骸コンテナの掃除

Level3実行後もコンテナが残っている場合は、`docker ps -a` の出力を確認したうえで手動での掃除を検討してください。

### 4.4 L2が全域不通の場合

L2到達性がラボ全体で失われている場合、IOL機器のL2転送を担う `iouyap` プロセスが機能していない可能性があります。対象コンテナで以下を実行し、プロセスの有無を確認してください。

```bash
docker top <コンテナ名>
```
