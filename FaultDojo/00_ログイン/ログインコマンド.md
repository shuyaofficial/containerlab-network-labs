# NW Fault Dojo 機器別ログインコマンド

対象ラボはテーマ21キャプストン（`clab-capstone-*`）のトポロジを流用します。実行場所はすべてOrbStackのclab VMです。

## 1. VM接続

```bash
ssh clab@orb
```

以降の操作はすべてこのVM内で行います。

## 2. 管理IPの解決（getip方式）

各機器の管理IPはコンテナ名から動的に解決します（IPドリフトに強い方式。テーマ21 `configure_all.sh` の `getip()` 関数と同じ）。

```bash
docker inspect -f '{{range .NetworkSettings.Networks}}{{.IPAddress}}{{end}}' clab-capstone-<node>
```

`<node>` には `isp` / `hq-edge1` / `hq-edge2` / `hq-core1` / `hq-core2` / `hq-pc-sales` / `br-pc` / `br-edge` のいずれかが入ります。

## 3. 機器別ログイン

### 3.1 SSH機器（isp / hq-edge1 / hq-edge2 / hq-core1 / hq-core2 / br-edge）

管理IPを解決したうえで、以下のコマンドでSSH接続します。認証情報は `admin` / `admin` です。

```bash
sshpass -p admin ssh -o StrictHostKeyChecking=no -o UserKnownHostsFile=/dev/null \
  -o KexAlgorithms=+diffie-hellman-group14-sha1 -o HostKeyAlgorithms=+ssh-rsa \
  -o PubkeyAcceptedAlgorithms=+ssh-rsa admin@<mgmt-ip>
```

`<mgmt-ip>` は「2. 管理IPの解決」で取得したIPに置き換えます。

### 3.2 Linux端末（hq-pc-sales / br-pc）

```bash
docker exec -it clab-capstone-<node> sh
```

`<node>` には `hq-pc-sales` または `br-pc` が入ります。

## 4. 終了方法

いずれの接続も `exit` で抜けます。

## 5. ログインできない場合

### 5.1 コンテナ状態の確認

```bash
docker ps --format 'table {{.Names}}\t{{.Status}}' | grep clab-capstone
```

### 5.2 管理IPが解決できない場合

```bash
docker inspect clab-capstone-<node>
```

出力の `NetworkSettings.Networks` 配下にIPが割り当てられているか確認します。

### 5.3 SSHが応答しない場合（起動待ち）

IOL機器（isp / hq-edge1 / hq-edge2 / hq-core1 / hq-core2）はx86エミュレーションのため起動に時間がかかります。**起動中はノードあたり2分以上SSHを待つ**想定で、コンテナ起動直後の接続失敗は異常ではありません。しばらく待ってから再試行してください。

## 6. ステータス

本ログイン方式は、テーマ21の `configure_all.sh` で実証済みの方式（getip解決 + sshpassによるSSH投入）をそのまま踏襲したものです。本道場での実ログインは、初回検証プレイで確認します。
