# 構築ログ：テーマ17 ネットワーク自動化基盤の構築（Ansible）

## 1. 構築概要
- **目的**: ネットワーク機器の設定を人間が手作業で行う従来の手法（CLI）から脱却し、インフラをコード化（Infrastructure as Code: IaC）してプログラムから一括設定・管理する手法を学ぶ。
- **使用技術**: Ansible (Playbook), YAML, Cisco IOS
- **構成**:
  - `Controller`: Ubuntuコンテナ（自動化指令サーバー / Ansible, Python環境）
  - `R1`, `R2`: Cisco IOLルーター

## 2. 実施内容
1. **Ansible環境の構築と疎通確認**
   - Containerlabの `exec` 機能を活用し、Ubuntuコンテナ起動時にAnsibleと関連モジュールを自動インストールする環境を構築。
   - `ansible_inventory.ini`（名簿）と `ansible.cfg`（設定ファイル）を作成し、Ansibleが各ルーターにSSH接続できる環境を整備した。
   - `ansible -m ping all` を実行し、全ルーターへの一斉ログインテストをパスした。

2. **PlaybookによるOSPF全自動構築**
   - R1とR2の初期IP設定およびOSPF（Area 0）の構築手順を記述した `ospf_setup.yml`（Playbook）を作成。
   - `ansible-playbook` コマンドを1回叩くだけで、両ルーターに同時に設定が流し込まれる挙動を確認。

3. **【検証】自動化結果の確認**
   - R1に手動でログインし、`show ip ospf neighbor` と `show run` を実行。
   - 人間が一度も `conf t` などの設定コマンドを入力していないにも関わらず、R2とのネイバーが「FULL」状態で確立し、正常にルーティングが動作していることを証明した。

## 3. トラブルシューティングと学び
- **Ansibleの警告（WARNING）とエラー**:
  - 古いプラグイン指定（`stdout_callback = yaml`）によるエラーや、非推奨機能の警告メッセージに直面した。エラー文を正しく読み解き、`ansible.cfg` を修正して警告を抑止（`deprecation_warnings = False`）しつつ、最新のAnsible Coreに適した形へ修正する対応を行った。
- **IaC（Infrastructure as Code）の威力**:
  - 数十行のYAMLコードを実行するだけで複雑なネットワークが数秒で完成するプロセスを体験し、「宣言的（Declarative）なインフラ管理」の圧倒的なスピードと確実性を実感した。
