# テーマ microseg_nftables マイクロセグメンテーション（nftables/IOL VLAN・ACL）試験結果（2026-07-05）

NW-ZT トラック N4。マイクロセグの本質「**同一セグメント（east-west）の通信を default-deny にし、許可した端末・パスだけを通して横移動を遮断する**」を、Cisco IOL の VLAN/inter-VLAN ACL（層1）とホスト nftables（層2）の**二層構成**で実機実証した。

## 試験環境（clab, arm64）

| 要素 | 値 |
|---|---|
| 仮想化基盤 | OrbStack VM（arm64）、containerlab 0.77、docker 29.1.3 |
| スイッチ | `sw1`（cisco_iol L2-advipservices-2017, L3 SVI + ip routing） |
| 端末イメージ | `microseg-endpoint:local`（ubuntu:24.04 + nftables/iproute2/iputils-ping/curl/ncat/python3） |
| トポロジ | pc10a(172.50.10.11)・pc10b(172.50.10.12) = **同一 VLAN10** / srv20(172.50.20.11) = VLAN20（http:80 + tcp:22 疑似サービス） |
| コンテナ名接頭辞 | `clab-microseg-`（既存稼働中 `clab-dynamic-routing-lab-*` とは完全分離） |

- **east-west の定義**: pc10a と pc10b は同一 VLAN10・同一サブネット（172.50.10.0/24）の横並び。この2台間の通信が「同一セグメント内の east-west」。srv20 は別 VLAN20 で、VLAN 間通信は sw1 のルーティング（inter-VLAN）を経由する。

## 実測エビデンス（要点）

### G2 — inter-VLAN ACL（層1）

```text
--- pc10a -> srv20:80 (期待: 到達/200) ---
HTTP=200

--- pc10a -> srv20:22 (期待: 遮断) ---
bash: connect: No route to host
BLOCKED
```

ACL マッチカウンタ（`show ip access-lists SEG10-OUT`、疎通試験後）:

```text
Extended IP access list SEG10-OUT
    10 permit tcp 172.50.10.0 0.0.0.255 host 172.50.20.11 eq www (6 matches)
    20 permit icmp 172.50.10.0 0.0.0.255 172.50.20.0 0.0.0.255
    30 deny tcp 172.50.10.0 0.0.0.255 host 172.50.20.11 eq 22 (1 match)
    40 permit ip any any (6 matches)
```

- tcp/80（www）に **6 matches**＝curl の到達が ACL の permit 行を実際に通過したことの実測。
- tcp/22 の deny 行に **1 match**＝22 への到達試行が確実に ACL で拒否されたことの実測（ping 疎通確認等で発生した icmp は別カウンタ line 20 でカウントされておりゼロ、curl/tcp 接続試行のみが deny にヒットした)。

SVI 状態（設定投入直後の一時的な down 表示を経て、再確認時は up/up）:

```text
Interface              IP-Address      OK? Method Status                Protocol
Vlan10                 172.50.10.1     YES manual up                    up
Vlan20                 172.50.20.1     YES manual up                    up
```

### G3 — intra-VLAN10 host nftables（層2・最重要ゲート）

**適用前**（pc10a → pc10b、同一 VLAN10 内）:

```text
PING 172.50.10.12 (172.50.10.12) 56(84) bytes of data.
64 bytes from 172.50.10.12: icmp_seq=1 ttl=64 time=6.86 ms
64 bytes from 172.50.10.12: icmp_seq=2 ttl=64 time=1.83 ms
64 bytes from 172.50.10.12: icmp_seq=3 ttl=64 time=1.50 ms
--- 172.50.10.12 ping statistics ---
3 packets transmitted, 3 received, 0% packet loss
```

pc10b に nftables 投入（`nft add rule inet filter input ip saddr 172.50.10.11 counter drop`）:

```text
table inet filter {
	chain input {
		type filter hook input priority filter; policy accept;
		ip saddr 172.50.10.11 counter packets 0 bytes 0 drop
	}
}
```

**適用後**（同じ pc10a → pc10b）:

```text
PING 172.50.10.12 (172.50.10.12) 56(84) bytes of data.
--- 172.50.10.12 ping statistics ---
3 packets transmitted, 0 received, 100% packet loss
```

nftables カウンタ（適用後・遮断試験直後）:

```text
table inet filter {
	chain input {
		type filter hook input priority filter; policy accept;
		ip saddr 172.50.10.11 counter packets 3 bytes 252 drop
	}
}
```

- **適用前後で pc10a→pc10b の到達性が 100%→0% に反転**、かつ nft の drop カウンタが `0→3 packets` に増加。同一 VLAN 内の端末間通信をホスト nftables が実際に遮断したことを、対照とパケットカウンタの両方で実証。

### G4 — 二層の役割分担まとめ

| 通信 | 制御点 | 適用前 | 適用後/対象外 | 結果 |
|---|---|---|---|---|
| pc10a → srv20:80（VLAN間） | 層1: IOL inter-VLAN ACL（SVI Vlan10 の `ip access-group SEG10-OUT in`） | — | ACL: permit tcp eq 80 | 到達（200） |
| pc10a → srv20:22（VLAN間） | 層1: IOL inter-VLAN ACL | — | ACL: deny tcp eq 22 | 遮断 |
| pc10a → pc10b（VLAN内・同一セグメント） | **層1は対象外**（同一VLANのためスイッチのルーティング/ACLを経由しない） | 到達（3/3, 0%損失） | 層2: pc10b の nftables drop | **遮断（0/3, 100%損失）** |

- **VLAN/ACL がセグメント間（inter-VLAN）にしか効かないことの証明**: pc10a→pc10b は同一 VLAN10・同一サブネットのため sw1 の L3 ルーティングを経由しない。実際、SEG10-OUT ACL は Vlan10 SVI 通過トラフィックにしか適用されないため、pc10a→pc10b の直接通信には一切関与しない（ACL 側にこの2台間を制御するルールを書いても効果がない）。
- **同一セグメント内の横移動を止めるにはホスト側 nftables が必須**であることを、pc10b 側の nftables 適用前後の対照（100%到達→100%遮断）で直接示した。

## 総括

**マイクロセグメンテーションの核心「VLAN/ACL はセグメント間、nftables はセグメント内」の役割分担を、Cisco IOL + Linux ホストの実機構成で二層とも実証完了。** 層1（inter-VLAN ACL）は VLAN10→VLAN20 の到達を tcp/80 のみに絞り込み、tcp/22 を確実に拒否した（ACL マッチカウンタで実測）。層2（ホスト nftables）は同一 VLAN10 内の pc10a→pc10b 通信を、適用前(3/3到達)→適用後(0/3到達)の対照とパケットカウンタ(0→3 drop)で確実に遮断したことを示した。

「VLAN に入れた＝業務VLAN内は自由に到達できる」状態から、「業務VLAN内でも許可されていない端末間は遮断される」状態への変化を、同一トポロジ・同一端末ペアに対するビフォー/アフター比較で提示できたことが最大の成果。詰まりどころ（iouyap起動、SVI表示タイミング、expect並列実行時の競合）は[構築ログ](../04_構築/構築ログ_2026-07-05.md)参照。

Illumio / Cisco TrustSec が担う「ネットワーク中心 vs ホスト中心のμセグ」の両方の適用点を、IOL ACL（ネットワーク層）と nftables（ホスト層）で1つのラボ内に同居させたことで、両者の適用範囲の違い（VLAN間/VLAN内）を手で確認できた。
