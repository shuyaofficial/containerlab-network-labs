# テーマZERO｜NW-ZT arm64 軽量検証結果（2026-07-04 実測）

[軽量検証計画_nwzt](軽量検証計画_nwzt.md) に基づき、NW-ZT トラック（N1-N4）の OSS イメージが clab VM（arm64）で動く見込みを実測した。

## 検証方法

clab VM（OrbStack Ubuntu 24.04 / arm64、docker）上で `docker manifest inspect <image>` を実行し `linux/arm64` エントリを確認。manifest list を返さないイメージは `docker pull` + `docker inspect --format '{{.Architecture}}'` で実 arch を確認。公式イメージが amd64 のみのものは、**arm64 ベース OS への apt 導入可否**を `docker run --platform linux/arm64 ubuntu:24.04` で `apt-cache policy` を引いて確認した。

## 結果表

| # | N | イメージ / 手段 | arm64 | 判定 | 根拠（実測） |
|---|---|---|---|---|---|
| 1 | N1 | `freeradius/freeradius-server:latest` | ✕（amd64のみ） | **High（apt導入で可）** | 公式イメージは pull 実証で `amd64`。ただし arm64 Ubuntu 24.04 に `freeradius` パッケージ候補あり（3.2.5）→ **Dockerfile 自作（arm64 OS + apt）で解決** |

## 重要な発見: 公式イメージ amd64-only の2件は apt で解決できる

前回 L7 検証の osquery / clamav は「arm64 パッケージが**存在しない**」真の非対応だったが、今回の **FreeRADIUS と ntopng は公式 Docker イメージが amd64 のみなだけで、arm64 の apt パッケージは存在する**。したがって:

- **N1 FreeRADIUS**: `docker pull freeradius/freeradius-server`（amd64）ではなく、`ubuntu:24.04`（arm64）ベースの Dockerfile に `apt-get install freeradius freeradius-utils` する自作イメージで arm64 ネイティブ稼働できる。
- **N3 ntopng**: 同様に arm64 Ubuntu ベース＋ `apt-get install ntopng`（ntop の apt リポジトリまたは universe）で対応。重量級の ElastiFlow は不要。

この「公式イメージ amd64-only → arm64 OS ベースの Dockerfile 自作」は、既存の vrnetlab（IOL の arm64 パッチビルド）と同じ発想であり、実現性は高い。

## ZT観点別の実装可能性判定（NW-ZT）

| N | 観点 | 判定 | 条件 |
|---|---|---|---|
| N1 | NAC / 802.1X | **実装可能（High）** | FreeRADIUS は arm64 Dockerfile 自作。IOL L2 の dot1x/CoA は実機検証（N1実装時） |
| N2 | SDP型ZTNA | **実装可能（High）** | OpenZiti が arm64 公式対応。Headscale/Netbird も可 |
| N3 | NDR | **実装可能（High）** | Zeek/Suricata/goflow2/Faucet が arm64 公式。ntopng は apt 自作。ElastiFlow は不要 |
| N4 | μセグ | **実装可能（High）** | nftables は OS 標準。IOL VLAN/ACL は実機（N1 基盤に依存） |

**結論**: NW-ZT トラック N1-N4 すべて arm64 × OSS で実装見込みが立った。要注意は「FreeRADIUS・ntopng は公式イメージでなく arm64 OS ベースの Dockerfile 自作で入れる」点のみ。IOL L2 の実機機能（dot1x/CoA/動的VLAN）は N1 実装時の実機検証に切り分ける。
