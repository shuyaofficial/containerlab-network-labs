## 1. 最初に判明した制約（重要な前提）

- 配布イメージは GNS3/EVE-NG 用の **VMディスク or バイナリ**で、コンテナではない。
- この Mac は **arm64 (Apple M4)**、Cisco イメージは **x86 系** → エミュレーションが必要で遅い。
- 公式 [containerlab.dev/macos] の方針:
  - arm64ネイティブ（FRR / SR Linux 等）を最優先で使え。
  - x86 は Rosetta で動かせるが「遅い、または起動しないこともある」。
  - VMベースはネステッド仮想化が必要（M3+ / macOS15+）→ M4なので要件は満たす。
  - Cisco で名前が挙がる現実解は **IOL（IOS on Linux）**。

→ 結論: arm64ネイティブのFRRを「土台」、Ciscoは IOL を Rosetta で動かす方針に決定。

### (D) 本物 Cisco IOL を動かす（ここが本題・難所）
ユーザー提供の `.bin`（i86bi_linux 系 IOU/IOL イメージ）を vrnetlab の `cisco/iol` でコンテナ化。
途中で出た **4つの壁** をすべて解決した:

| # | 症状 | 原因 | 解決 |
|---|---|---|---|
| 1 | `make: command not found` | clabマシンに make 無し | `apt install make` |
| 2 | `qemu-i386: Could not open /lib/ld-linux.so.2` | イメージが **32bit i386**、Dockerfileは64bit Debian | Dockerfileに `dpkg --add-architecture i386` + `libc6:i386 libgcc-s1:i386` を追加 |
| 3 | `IOURC: Could not open iourc file` | 旧IOUは**ライセンス必須** | entrypointに CiscoIOUKeygen を組込み、コンテナのhostid/hostnameに一致する iourc を自動生成 |
| 4 | 起動時の「初期設定ダイアログ」でclab自動設定が衝突 | 旧イメージは対話ダイアログを出す＋エミュで起動が遅い | entrypointを **expect** でラップし、ダイアログに自動で `no` 応答→その後 `interact` でコンソール維持 |

- ビルドは必ず `DOCKER_DEFAULT_PLATFORM=linux/amd64 make docker-image`（amd64で固める→Rosetta実行）。
- 最初の検証で **Cisco IOS 15.6(3)M3a が `r1#` まで起動・CLI応答**を確認。

### (E) 3台（→現在4台）OSPFラボ
- `~/network-lab/ospf3/` に r1-r2-r3 線形 + OSPF を構築（後にr4追加）。
- 各ノードの startup-config（`r1.cfg` 等）に Loopback / インターフェース / OSPF を記述。
- 検証結果:
  - r2(hub) が両隣と `FULL`。
  - `r1# ping 3.3.3.3 source 1.1.1.1` → **100% 成功**（2ホップ先のr3ループバックまで疎通）。
  - OSPFで経路学習も確認（`O 3.3.3.3 [110/21]`）。

### (F) 全イメージのビルド（計11個）
- 提供された11個の `.bin` をすべて `vrnetlab/cisco_iol:<タグ>` としてビルド完了。
- L3×5、L2×6。一覧は `../環境/環境説明.md` の §4 参照。

### (G) 後片付け
- ビルド済みイメージはバイナリを内部に保持しているため、`~/Downloads` の元 `.bin` 11個を削除（容量解放）。

## 3. よくあるハマりどころ（FAQ）

**Q. 機器にSSHできない（`ssh admin@clab-ospf3-r1` → No route to host）**
A. この旧IOLは管理IF/SSHが未設定のため。機器へは **コンソール接続**で入る:
```bash
sudo docker attach clab-ospf3-r1   # → Enter → r1#
```
**抜けるのは `Ctrl-P` → `Ctrl-Q`**（`exit`やCtrl-Cで抜けるとノードが止まることがある）。
VSCodeなら CONTAINERLAB EXPLORER でノード右クリック → Attach（コンソール）。
※ 将来SSHしたい場合は、各ノードに `crypto key generate rsa` 等の設定投入が別途必要。

**Q. `docker ps --format ...` で変な出力になる**
A. `--format` は Goテンプレート（例 `'{{.Names}}'`）が必須。ただ一覧を見るだけなら
オプション無しの `sudo docker ps` でOK（NAMES列が機器名）。

**Q. Linux/Dockerで IOSの「?」みたいな補完は?**
A. `?` は無い。代わりに:
- ヘルプ: `docker ps --help`（`--help`は必ず末尾に単独で）
- 補完: **Tabキー**
- マニュアル: `man docker`

**Q. 起動が遅い**
A. arm64上でx86をエミュしているため（IOL約2分）。仕様。FRR/SR Linuxなら瞬時。

## 5. 他機種・他ベンダーへの拡張（2026-06-02）

IOL以外（Cisco vIOS + 他ベンダー）を追加。arm64でフルVM(x86 qcow2)を動かす壁を2改造で突破。

### (A) 突破した壁（再ビルド時も維持が必要・最重要）

| # | 症状 | 原因 | 解決（改造箇所） |
|---|---|---|---|
| 1 | `qemu-system-x86_64: CPU model 'host' requires KVM or HVF` でqemu即死 | /dev/kvm無し(arm64)なのに `-cpu host` 固定。KVM不在時に`-enable-kvm`は外すがcpuはhostのまま | `~/vrnetlab/common/vrnetlab.py`：`self._cpu=cpu` 直後に「kvm無 かつ cpu==host なら qemu64」フォールバック追加。**全VM機種に効く**（backup: `vrnetlab.py.bak`） |
| 2 | vIOSが設定ダイアログ後に無言→bootstrap永久待ち(unhealthy) | `con_expect` の `read()` が無出力時にブロックしnudgeに到達しない | `cisco/vios/docker/launch.py`：ダイアログ応答後にフラグを立て、各spin冒頭で `\r` を先行送信しプロンプトを出す（backup: `launch.py.bak`） |

> 両方とも hellt/vrnetlab 共有ファイルの改造。**イメージ再ビルドで反映**。戻すなら `.bak` を復元。
> 実機KVM環境（/dev/kvm有）では改造1は自動無効（host維持）。

### (E) ハマりどころ
- `pkill -f "make build"` が**自分のSSHシェルにマッチして自滅**(exit255)。`[m]ake build` のbracketトリック or PID指定で回避。
- `unzip` 未導入→`apt install unzip`。`gdown`/`qemu-utils`/`sshpass` も追加導入。
- Google Drive: ファイルによりquotaで `gdown` 失敗→別バージョン/別ミラーで回避。
- vIOSのmgmt(Gi0/0)はclabのtc-tapモードではadmin-down（直SSH不可）。CLIは `docker attach` かコンソール(telnet 5000)で。データプレーンは正常。

### (F) ユーザー提供イメージの取り込み結果（2026-06-02・追加）

ユーザーがログインDLした Fortinet / Palo Alto 等を取り込み試行。

| 提供物 | 結果 | 補足 |
|---|---|---|
| FortiGate `FGT-v7.2.0` / `v7.2.4` | ✅ **healthy検証済**（約2分）。`vrnetlab/vr-fortios:7.2.0` `:7.2.4` | tgz内 `virtioa.qcow2` を `fortios-vX.Y.Z.qcow2` にリネーム |
| FortiFirewall `FFW v7.4.2`(qcow2 2GB) | ◯ 同builderで可（FortiOS系）。未ビルド | `fortios-v7.4.2.qcow2` にして `make` すれば同様 |
| FortiFirewall `FFW v7.0.13` | ✗ raw 81MBでフルディスクでない（ファーム断片）→不可 | |
| `virtioa.qcow2`(単体) | = FortiOSディスク（stringsで確認）。FGTと重複 | |
| Palo Alto `8.0.0` | ✗ **DL破損**（先頭約6.8MBが全ゼロ＝gzip/tar/qcow2いずれでもない）＋PA-VMは4-6GBでarm64非現実的 | 再DLしても本機では実用困難 |
| FortiManager `FMG 7.2.2`/`6.2.3` | ✗ **Containerlab非対応**（vrnetlabビルダ無・clab kind無。管理アプライアンスで2ディスク構成） | どうしても要るなら素のqemu VMで（clab外） |
| Juniper vMX（参考リンクのみ・画像なし） | ✗ 6GB/dual-VM/ビルドにKVM要でarm64非現実的 | 代替: **cJunosEvolved**(コンテナ・`docker load`) や **cRPD** を推奨（Qiita記事も cJunosEvolved 利用） |

**FortiGate固有のハマり所**:
1. `fortinet/fortigate` は**旧式builder**（`vrnetlab-base`未使用・独自Dockerfile）。`make docker-image`(汎用)は `IMG_NAME` 未設定でタグ `vrnetlab/fortinet_:` が不正→失敗。**`make`（既定ゴール `docker-build-fortigate`）を使う**（→ `vrnetlab/vr-fortios:<ver>`）。
2. 独自Dockerfileが `FROM debian:bookworm-slim` + `qemu-kvm` のみ。**arm64では `qemu-kvm` がaarch64版qemuを入れ `qemu-system-x86_64` が無い**→起動時 `command not found`。Dockerfileに **`qemu-system-x86`** を追加して解決（→ arm64ネイティブqemuがx86をTCGエミュ＝他より1層分速い）。
3. `common/*` をbuild時にコピーするので、改造1（cpu host→qemu64）は効く。
