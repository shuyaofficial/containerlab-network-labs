1. 物理設計#

項目番号設計項目決定方針対応要件ID
1.1物理設計方針老朽LAN（old-rt/old-core/old-sw1/old-sw2）には一切手を加えず、監視専用ノード（mon-core・cap-sw・nms・cap）を新設し、cap-swをold-core⇔mon-core間にインラインで挿入する。REQ-C03 REQ-C04
1.2老朽ポイント（温存対象）単一コアold-core（全VLANのSVIとstatic route集約=SPOF）、VLAN20のold-sw1/old-sw2両方への延伸、static一辺倒の経路制御、旧IOS（old-rt=15.6.3M3a等）の4点は意図的に温存し改修しない。REQ-L01 REQ-C03
1.3ノード構成IOL6台（old-rt/old-core/old-sw1/old-sw2/mon-core/cap-sw）、Linux5台（nms/cap/pc-a/pc-b/srv-file）、Zabbix3台（zbx-db/zbx-srv/zbx-web）の計14ノード・リンク11本で構成する。REQ-M01
1.4インライン挿入点cap-swはold-core（E1/0）⇔mon-core（E0/1）間の監視系transit（VLAN900）にのみインラインで挿入し、老朽LAN内部の他区間には新規機器・リンクを追加しない。REQ-C04

ノード配置とリンク結線の詳細はノード配置・リンク収容表を参照。

2. 論理設計#

項目番号設計項目決定方針対応要件ID
2.1論理設計方針監視対象IPはOOB（172.28.28.0/24）ではなく、機器管理VLAN90または監視系transit VLAN900のin-bandデータプレーンIPを使用する。REQ-M01
2.2監視セグメント（VLAN100）10.28.100.0/24、デフォルトGWはmon-coreのSVI（10.28.100.1）。nms（.10）・zbx-srv（.20）を収容する。REQ-M01
2.3監視系transit（VLAN900）10.28.0.0/29（old-core=.1／mon-core=.2／cap-sw=.3）。/30ではなく/29とした設計根拠は下表参照。REQ-M07
2.4WAN transit（VLAN901・老朽LAN側/変更なし）10.28.1.0/30（old-rt=.1／old-core VLAN901 SVI=.2）。老朽LAN側の既存構成のため変更しない。REQ-C03
比較観点案A: 監視系transitを/29とする採用案B: /30（P2Pのみ）
ホスト収容数利用可能ホスト6、old-core（.1）／mon-core（.2）／cap-sw（.3）の3台を収容できる利用可能ホスト2のみで、インラインのcap-swにIPを割り当てる余地がない
cap-swの管理性cap-swがVLAN900上でin-band管理IP（.3）を持てるため、SSH等での個別アクセスが可能cap-swにIPを割り当てられず、transit上での直接管理ができない
アドレス効率/29はアドレス効率をやや犠牲にするが、実利用は6中3/30はアドレス効率が高いが、インライン挿入前提の要件を満たせない
採用理由インライン挿入されるcap-sw自身に管理IPを持たせる必要があるため、利用可能ホスト数6の/29を採用した（詳細はIPアドレス管理表.md参照）。

3. 監視設計#

項目番号設計項目決定方針対応要件ID
3.1監視設計方針ポーリング対象は老朽LAN側4台＋監視系2台の計6 IOL機器のin-band IPとし、system/interfaces MIBを中心に段階的にポーリング範囲を広げる。REQ-M01
3.2SPAN設計cap-swでmonitor session 1 source interface Ethernet0/1 both＋destination interface Ethernet0/3を設定する。source指定の設計根拠は下表参照。REQ-M07 REQ-M08
比較観点案A: source=E0/1のみ採用案B: source=E0/1＋E0/2両方
ミラー対象フレーム監視系transitを通過する全フレームはE0/1を必ず1回通過するため、これのみで捕捉できる同一フレームが往路E0/1・復路E0/2の双方でミラーされ、2重に記録される
キャプチャ解析への影響Wireshark上で1フレーム=1パケットとして正しく解析できる同一パケットが2つ記録され、トラフィック量やシーケンス解析が誤る
設定の複雑さsource指定が1ポートのみでシンプルsource指定は2ポートで、別途二重ミラー対策が必要になる
採用理由cap-swはold-core（E0/1）⇔mon-core（E0/2）間にインライン挿入されており、通過フレームは必ずE0/1を経由するため、E0/1のみをsourceにすることで二重ミラーを防止した。
3.3主要OIDsysName（.1.3.6.1.2.1.1.5）／ifDescr（.1.3.6.1.2.1.2.2.1.2）／ifOperStatus（.1.3.6.1.2.1.2.2.1.8）／ifInOctets（32bit, .1.3.6.1.2.1.2.2.1.10）／ifHCInOctets（64bit, .1.3.6.1.2.1.31.1.1.1.6）を主要ポーリング対象とする。詳細は詳細設計書・監視設計書.mdを参照。REQ-M02 REQ-M03 REQ-M04

4. セキュリティ設計（v2c→v3）#

項目番号設計項目決定方針対応要件ID
4.1セキュリティ設計方針SNMPはv2c（community public）で導入し、SPANキャプチャで平文の危険性を確認したうえでv3（authPriv）へ段階移行し、移行完了後にv2cを閉塞する。REQ-S01 REQ-S02
4.2v2c→v3移行方式移行方式は下表の比較のとおり、v2c→観測→v3の段階移行を採用する。REQ-L05
比較観点案A: v3一斉導入案B: v2c→観測→v3段階移行採用
学習効果v2cの平文リスクを体験できないまま、暗号化前提の設定に入ってしまうcommunity平文の危険性をSPANキャプチャで実際に確認したうえで、移行理由を体感できる
導入の速さ初期投入時から認証・暗号化設定が必要で手順が多いまずcommunity ROの一発疎通で確認でき、MIB探索やIF-MIBカウンタ監視を先に学べる
実務との整合性新規構築では妥当だが、既存v2c環境からの移行学習にはならない実務でも多いv2c→v3の段階移行・混在運用（トラップはv2c継続）を体験できる
採用理由Mission3〜4でv2cによるMIB探索／IF-MIBカウンタ監視を先に習得し、Mission5でキャプチャによる危険性の実演を経てv3へ移行する、教育効果を重視した段階移行を採用した。
項目番号設計項目決定方針対応要件ID
4.3SNMPv3パラメータgroup NMS-GROUP（v3 priv）／user nmsuser、auth SHA SnmpAuth28!、priv AES128 SnmpPriv28!（学習用値）。REQ-S01
4.4トラップ運用linkdown/linkup/coldstartを10.28.100.10（nms, snmptrapd）へversion 2c publicで送信する。ポーリングv3化後もトラップはv2c混在運用とする。REQ-M05 REQ-M06
