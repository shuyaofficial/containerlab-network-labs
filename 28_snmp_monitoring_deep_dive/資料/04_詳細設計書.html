nms・cap（監視サーバ／キャプチャホスト）#

両者ともwbitt/network-multitoolコンテナ。nmsはSNMPポーリング・トラップ受信を担い、capはcap-swのSPAN宛先（Ethernet0/3）を収容してpromiscuousモードでミラーフレームを受信する。

項目仕様
nms eth110.28.100.10/24（GW 10.28.100.1）
nms 追加経路ip route add 10.28.0.0/16 via 10.28.100.1 dev eth1
nms mgmt IP172.28.28.101
nms 備考snmptrapd利用時はapk add net-snmpの追加導入が必要（未同梱・ISS-04）
cap eth1IPなし・promiscuousモード
cap mgmt IP172.28.28.102
cap 備考SPAN destination受け（tcpdump/Wiresharkでミラーフレームを観測）
clab.yml exec抜粋（nms／cap）
# nms
ip link set eth1 up
ip link set eth1 mtu 1500
ip address add 10.28.100.10/24 dev eth1
ip route add 10.28.0.0/16 via 10.28.100.1 dev eth1

# cap（SPAN destination・promiscuous）
ip link set eth1 up
ip link set eth1 mtu 1500
ip link set eth1 promisc on

zabbixイメージは非rootユーザーで動作するため、containerlabのexecディレクティブ（コンテナ起動ユーザー権限で実行）ではzbx-srvのeth1へIP／経路を投入できない。そのためdeploy.shのzbx_net()がdocker exec -u 0（root指定）で後付け投入する（ISS-03）。

deploy.sh zbx_net()（抜粋）

docker exec -u 0 clab-snmp-monitoring-lab-zbx-srv ip link set eth1 up
docker exec -u 0 clab-snmp-monitoring-lab-zbx-srv ip link set eth1 mtu 1500
docker exec -u 0 clab-snmp-monitoring-lab-zbx-srv ip address replace 10.28.100.20/24 dev eth1
docker exec -u 0 clab-snmp-monitoring-lab-zbx-srv ip route replace 10.28.0.0/16 via 10.28.100.1 dev eth1

SNMP共通設計#

監視対象6台のIOL機器（old-rt／old-core／old-sw1／old-sw2／mon-core／cap-sw）に共通で投入するSNMP・トラップ・SPAN設計。詳細は監視設計書.mdを正本とする。

v2c→v3移行仕様

項目仕様
SNMPv2c communitypublic（RO）
SNMPv2c 用途Mission3〜4で使用
SNMPv3 groupNMS-GROUP（v3 priv）
SNMPv3 usernmsuser
SNMPv3 user 認証auth SHA SnmpAuth28! / priv AES128 SnmpPriv28!（学習用値）
移行完了後no snmp-server community public
移行完了後 効果v2cアクセスを閉塞
SNMP共通設定（全6台のIOLに共通投入）
! --- Mission3: SNMPv2c ---
snmp-server community public RO
snmp-server location "<設置場所>"
snmp-server contact noc@example.jp
!
! --- Mission5: SNMPv3(authPriv)へ移行 ---
snmp-server group NMS-GROUP v3 priv
snmp-server user nmsuser NMS-GROUP v3 auth sha SnmpAuth28! priv aes 128 SnmpPriv28!
! 移行完了後（v2cアクセスを閉塞）
no snmp-server community public

トラップ仕様

項目仕様
通知種別linkdown / linkup / coldstart
送信先10.28.100.10（nms）version 2c community public

SPAN仕様（E0/1のみsourceの根拠）

cap-swはold-core⇔mon-core間の監視系transit（VLAN900）にインラインで挿入されている（E0/1=old-core側、E0/2=mon-core側）ため、この区間を通過するフレームは必ずEthernet0/1を1回通過する。monitor session 1 source interface Ethernet0/1 bothによりE0/1のみをsourceに指定すれば、往路・復路とも重複なく全通過トラフィックを捕捉できる。E0/1・E0/2の両方をsourceに指定すると、同一フレームが2回（受信時＋送信時）ミラーされ、二重キャプチャとなってしまう。

SPAN設定（cap-sw・再掲）
monitor session 1 source interface Ethernet0/1 both
monitor session 1 destination interface Ethernet0/3

32bit/64bitカウンタ問題

ifInOctets / ifOutOctets は32bitカウンタであり、最大値2^32-1バイト（約4.3GB）でラップアラウンド（0に戻って再カウント）する。高速・高トラフィックな回線ではこの上限に短時間で到達し、単純な差分計算では実際のトラフィック量を過小評価してしまう。そのため64bitカウンタであるifHCInOctets / ifHCOutOctets（High Capacity、最大値約1.8×10^19バイト）の利用が推奨される（RFC2863）。Mission4ではiperf3でトラフィックを生成し、32bit版と64bit版の両方をポーリング・比較してこの問題を体感する。
