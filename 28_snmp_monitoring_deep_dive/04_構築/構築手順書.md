# Theme 28 構築手順書

対象トポロジ: `legacy-lan.clab.yml`（ラボ名 `snmp-monitoring-lab`、mgmt `172.28.28.0/24`、コンテナ名prefix `clab-snmp-monitoring-lab-`）

## 1. デプロイ

```bash
ssh clab@orb
cd /Users/shuya/Documents/claude/Mac仮想環境構築/28_snmp_monitoring_deep_dive/04_構築
./deploy.sh deploy
```

- containerlabがトポロジをデプロイし、IOL 6台（old-rt / old-core / old-sw1 / old-sw2 / mon-core / cap-sw）＋ Linux 5台（nms / cap / pc-a / pc-b / srv-file）＋ Zabbixスタック3台（zbx-db / zbx-srv / zbx-web）の計14ノードが起動します。
- `entrypoint_attach.sh`がIOLブート完了後（+約60秒）に`iouyap`を自動起動します。デプロイ直後は配線がまだ完成していないため、**60秒程度待ってから**次の確認に進んでください。
- 自動起動が働かない場合のみ、手動リカバリとして`./deploy.sh iouyap`を実行します。

### IOLライセンスの扱い（deploy.shが自動処理）

- 本テーマのL2系イメージ（`L2-15.2` / `L2-advipservices-2017`）と旧IOS（`15.6.3M3a`）は、ライセンスが`/iol/.iourc`ファイルではなく**イメージ側`/entrypoint.sh`内**に埋め込まれています。本ラボは`entrypoint_attach.sh`をbindで上書きするため、そのままではライセンスが渡らずIOLが`License Error`で起動しません（テーマ27 ISS-08の亜種）。
- 対策として`deploy.sh deploy`は、デプロイ前にVM内のIOLイメージからライセンス値を抽出し、VMローカルの`/opt/clab/.iourc`（mode 600）へ書き出します。`legacy-lan.clab.yml`が全IOLノードへ`/opt/clab/.iourc:/iol/.iourc:ro`をbindし、`entrypoint_attach.sh`の「有効値なら尊重」ロジックでそのまま使われます。
- **必ず`./deploy.sh deploy`経由でデプロイしてください**（`containerlab deploy`を直接実行すると、初回は`/opt/clab/.iourc`が無くbindが失敗します）。
- ライセンス値はVM内（Dockerイメージと`/opt/clab/.iourc`）にのみ存在します。**gitリポジトリ・Mac側へコピーしないこと**。

## 3. Linuxノードのeth1自動設定

`legacy-lan.clab.yml`の各Linuxノードは`exec`ブロックにより、コンテナ起動時に以下が自動投入済みです（手動設定不要）。

| ノード | eth1 IP | デフォルトゲートウェイ | 備考 |
|---|---|---|---|
| nms | 10.28.100.10/24 | なし（`10.28.0.0/16 via 10.28.100.1`の個別経路のみ） | 監視サーバ。defaultはmgmt(eth0)側に残し、`apk`等の外部到達を維持 |
| pc-a | 10.28.10.10/24 | 10.28.10.1（old-coreのVLAN10 SVI） | 営業部PC |
| pc-b | 10.28.20.10/24 | 10.28.20.1（old-coreのVLAN20 SVI） | 総務部PC |
| srv-file | 10.28.30.10/24 | 10.28.30.1（old-coreのVLAN30 SVI） | ファイルサーバ |
| cap | IPなし（promisc on） | ― | SPAN destination受け専用 |

- IPアドレス・デフォルトルートの投入はclab.ymlの`exec`で完了しているため、Linuxノードにログインしても追加のIP設定は不要です。
- IOS側（old-core・old-sw1・old-sw2・mon-core・cap-sw）のVLAN／SVI／staticルート／SNMP設定は、README_Lab_Challenge.mdのMissionとして受講者が手動投入します（clab.ymlには一切書き込みません）。

## 4. nmsへのnet-snmp導入

`wbitt/network-multitool`イメージには`snmpwalk`/`snmpget`/`snmptranslate`/`iperf3`が同梱済みです（2026-07-16のdeploy検証で確認）。**`snmptrapd`のみ未同梱**のため、Mission 6の前に以下でインストールします。

```bash
sudo docker exec -it clab-snmp-monitoring-lab-nms bash
apk add --no-cache net-snmp
```

- multitoolイメージはAlpineベースのため`apk`を使用します（`snmptrapd`は`net-snmp`パッケージに含まれ、`/usr/sbin/snmptrapd`に入ります）。
- nmsのdefaultルートはmgmt側（eth0）に残してあるため、ラボ稼働中でも`apk`のインターネット到達は生きています（clab.ymlの`exec`はラボ向けに`10.28.0.0/16 via 10.28.100.1`の個別経路のみ追加する設計。defaultをeth1へ向けるとDNS/apkが不通になるので変更しないこと）。
- インターネットに出られない環境の場合は、Mac側で取得済みのnet-snmp入りイメージ・パッケージを`docker cp`でコンテナに持ち込む等の代替手段を検討してください。

## 5. Zabbix監視スタック

`legacy-lan.clab.yml`には、SNMPv2c/v3に加えたもう1系統の監視基盤としてZabbixコンテナ3台が定義されています。`./deploy.sh deploy`実行時にIOL・Linux 11台と合わせて自動起動します（追加投入は不要）。

| ノード | イメージ | mgmt IP | データプレーンIP | 役割 |
|---|---|---|---|---|
| zbx-db | postgres:16-alpine | 172.28.28.111 | なし（mgmtのみ） | Zabbix用DB |
| zbx-srv | zabbix/zabbix-server-pgsql:alpine-7.0-latest（実バージョン7.0.28 LTS） | 172.28.28.112 | eth1=10.28.100.20/24（VLAN100、mon-core E0/3向け） | Zabbix Server（in-bandポーリング） |
| zbx-web | zabbix/zabbix-web-nginx-pgsql:alpine-7.0-latest | 172.28.28.113 | なし（mgmtのみ） | Zabbix Web UI |

- **Web UI**: MacからはVMポート8280経由で `http://clab.orb.local:8280` にアクセスします（確認済み）。初期ログインはZabbix既定値の **Admin / zabbix** です。初回ログイン後、必ずパスワードを変更してください。
- **DB接続情報**（学習用値）: user=`zabbix` / password=`ZbxLab28` / db=`zabbix`。実運用環境では絶対に使い回さないでください。
- **in-bandポーリングの前提条件**: zbx-srvのeth1はmon-core:eth3（Ethernet0/3）に接続されていますが、**IOS側（mon-core E0/3をVLAN100アクセスポート化）はMission 2で学習者が投入する設定です**。この設定が入るまでzbx-srvのin-band疎通（10.28.100.20経由のポーリング）は通りません。Web UI（mgmt経由の8280番ポート）自体は投入前でも利用できます。
- **zbx-srvのeth1 IP/経路について**: zabbixイメージが非rootで動作するため、`clab exec`でのIP投入ができません。そのため`deploy.sh`の`zbx_net()`関数がホスト側からeth1のIP（10.28.100.20/24）と個別経路（`10.28.0.0/16 via 10.28.100.1`）を投入します。zbx-srvコンテナを再起動した場合はIP/経路が失われるため、`./deploy.sh zbx-net`で再投入してください。
- Zabbixのポーリング経路はVLAN100→VLAN900のin-band経由です（cap-swのSPAN観測点をSNMPトラフィックが通過する設計意図を維持するため）。

## 6. snmptrapd起動手順（Mission 6）

nms上で以下を実行し、トラップ受信の受け皿を用意します。

```bash
cat > /tmp/snmptrapd.conf <<'EOF'
authCommunity log public
disableAuthorization yes
EOF

snmptrapd -f -Lo -c /tmp/snmptrapd.conf
```

- `-f`: フォアグラウンド起動（Ctrl-Cで停止するまで画面にログを流し続けます）。
- `-Lo`: 標準出力へログ出力。
- `authCommunity log public`: community `public`のトラップを受理してログに記録します。
- `disableAuthorization yes`: v1/v2cトラップの認可チェックを簡略化します（ラボ用途）。

old-sw1のE0/2を`shutdown`すると、linkDownトラップがこの画面に着信します。

## 7. Wiresharkキャプチャ（2系統）

SPAN先（cap）での観測は、用途に応じて以下の2系統があります。

### 7.1 capコンテナ内でtcpdump

```bash
sudo docker exec -it clab-snmp-monitoring-lab-cap sh
tcpdump -nni eth1 -w /tmp/span.pcap
```

v2cのcommunity文字列を平文確認する場合:
```bash
tcpdump -nni eth1 -A udp port 161
```

### 7.2 Mac側Wiresharkへ直結

```bash
ssh clab@orb "sudo nsenter -t \$(docker inspect -f '{{.State.Pid}}' clab-snmp-monitoring-lab-cap) -n tcpdump -U -nni eth1 -w -" | /Applications/Wireshark.app/Contents/MacOS/Wireshark -k -i -
```

- OrbStack VM内のcapコンテナのネットワーク名前空間へ`nsenter`で入り、tcpdumpの標準出力をパイプでMac側のWiresharkに渡します。
- キャプチャ対象はcap-swのSPAN destinationポート（cap:eth1）なので、old-core⇔mon-core間を通過する監視系トラフィック（SNMPを含む）がここに写ります。

## 8. destroy時の注意

```bash
./deploy.sh destroy
```

- 内部で`--cleanup`付きの`containerlab destroy`を実行するため、**nvram（保存済みIOSコンフィグ）ごと削除されます**。
- 投入したIOS設定を残したままラボを一旦壊したい場合は、`--cleanup`を付けずに直接実行してください。
  ```bash
  sudo containerlab destroy -t legacy-lan.clab.yml
  ```
