# 要件定義書: 監視基盤とSNMP深堀り

## 1. 背景と目的
本ラボの現行LAN（老朽社内LAN）は、監視の仕組みを一切持たない状態で稼働している。単一コアL3SW（old-core）に全VLANのSVIとstatic routeが集約されるSPOF構成、総務部VLAN20がold-sw1・old-sw2の2台にまたがってL2延伸されている構成、経路制御がstatic一辺倒、稼働機器のIOSが15.6と旧式という、複数の老朽ポイントを抱えている。監視が存在しないため、障害の予兆はおろか、発生した障害にすら気づけない状態にある。

本ラボでは、この老朽LANには手を加えず（改修は行わず）、監視L3SW（mon-core）・監視サーバ（nms）・Wireshark用SW（cap-sw、SPAN経由でcapへミラー）を後付けで増設し、SNMPによる監視を v2c → walk/OID/MIB → IF-MIB → v3(authPriv) → trap の順に段階的に深堀りする。

なお、本ラボで扱う老朽LANは、後続テーマ29「社内LAN更改・移行・移設」の移行元となる連作構成である。本ラボで明らかになる老朽ポイント（SPOF・VLAN延伸・static一辺倒・旧IOS）は、テーマ29における更改の動機として意図的に設計されている。

## 2. 監視要件

### 2.1 監視対象
- 老朽LAN側4機器（old-rt, old-core, old-sw1, old-sw2）と、増設した監視系2機器（mon-core, cap-sw）の計6 IOL機器をSNMP監視対象とする。
- 監視対象IPは、containerlabのOOB管理ネットワーク（172.28.28.0/24）ではなく、in-band側のデータプレーンIP（機器管理VLAN90 SVI、または監視系transit VLAN900 SVI）を使用する。実運用の監視システムが業務用IPを対象に監視するのと同様の構成である。

### 2.2 ポーリング要件
- SNMPv2c（community: `public`, RO）で、system/interfacesのMIB情報をポーリングできること。
- 名前OID・数値OIDの双方でGet/Walkが行えること、`snmptranslate`でMIBツリー構造を確認できること。
- IF-MIB（ifDescr, ifOperStatus, ifInOctets, ifOutOctets, ifHCInOctets, ifHCOutOctets）をポーリングし、トラフィックカウンタの増減を確認できること。
- 32bitカウンタ（ifInOctets等）のラップアラウンド問題を理解し、64bitカウンタ（ifHCInOctets等）の必要性を説明できること。

### 2.3 SNMPv3セキュリティ要件
- community文字列は平文でネットワーク上を流れるため、キャプチャすれば容易に窃取できる。この危険性をSPANキャプチャで実際に確認したうえで、認証・暗号化を備えるSNMPv3（authPriv）へ移行できること。
- SNMPv3移行後は`no snmp-server community public`によりv2cアクセスを閉塞し、以後v2cでのアクセスが失敗することを確認できること。

### 2.4 トラップ要件
- linkDown/linkUp/coldStartのイベントをトラップとしてnmsへ通知できること。
- old-sw1のインターフェース障害（shutdown）発生時に、nms側で実際にlinkDownトラップを受信できること。
- トラップはSNMPv3化した後もv2cのまま運用する（ポーリングはv3・トラップはv2cのままという混在運用は、実務でも監視ソフトウェアの対応状況次第で現実的な選択肢である）。

## 3. キャプチャ要件
- 老朽LANの監視系transitセグメント（old-core⇔mon-core間）にWireshark用SW（cap-sw）をインラインで挿入し、SPAN（ミラーリング）によりcapへ全通過トラフィックを転送できること。
- capはSPAN宛先としてpromiscuousモードで待受け、tcpdump/Wiresharkでキャプチャ・解析できること。
- SNMPv2c時のcommunity文字列平文、SNMPv3化後の暗号化状況を、キャプチャで比較確認できること。

## 4. 制約事項
- 既存の老朽LAN（old-rt/old-core/old-sw1/old-sw2、VLAN10/20/30/90の構成、static routing）には一切手を入れない。改修・是正はテーマ29のスコープであり、本ラボでは老朽ポイントをそのまま温存する。
- 監視基盤の追加によるインライン挿入は、cap-sw部（old-core⇔mon-core間）のみに限定し、老朽LAN内部の他区間には新規の機器・リンクを挿入しない。
- 全SNMP・SPAN設定はIOS CLIで完結させる。nms側はnet-snmp系ツール（snmpwalk/snmpget/snmptranslate/snmptrapd等）を用いる想定とし、商用NMSソフトウェアは用いない。
