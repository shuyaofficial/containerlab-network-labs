> MIBはツリー状の階層構造を持ち、OIDはそのツリー上のノードを数値で一意に識別する。SNMPは「GetRequest / GetNextRequest / SetRequest / GetResponse / Trap」の5種類のメッセージを組み合わせ、SNMP Get・SNMP Set・SNMP Trapという3種の動作パターンを実現する（GetはUDP/161のユニキャストでOIDを問い合わせて値を取得、TrapはUDP/162でエージェント発の障害通知を送る）。いずれもコミュニティ名が一致してはじめて通信が成立する。

## 3. ポーリング設計

### 3.1 SNMPv2c（Mission3〜4）
- 監視対象6機器全てに `snmp-server community public RO` を投入する（RO=読み取り専用）。
- `snmp-server location <設置場所>` / `snmp-server contact noc@example.jp` を各機器に設定する（設置場所はフロアA/フロアB/サーバ室/監視ラック等、パラメータシート参照）。
- nms側コマンド例:
  - `snmpwalk -v2c -c public 10.28.90.1 system`（systemサブツリーを丸ごとwalk）
  - `snmpget -v2c -c public 10.28.90.1 1.3.6.1.2.1.1.5.0`（sysNameを数値OIDでget）
  - `snmptranslate -Tp -IR system`（systemサブツリーの構造をローカルMIBから表示）

### 3.2 SNMPv3（Mission5）
- SPANキャプチャでv2cのcommunity平文の危険性を確認した後、以下を投入しauthPriv（認証SHA＋暗号化AES128）へ移行する。
  - `snmp-server group NMS-GROUP v3 priv`
  - `snmp-server user nmsuser NMS-GROUP v3 auth sha SnmpAuth28! priv aes 128 SnmpPriv28!`
  - ※`SnmpAuth28!` / `SnmpPriv28!` は本ラボ専用の学習用パスフレーズであり、実運用では使い回さないこと。
- 移行完了後、`no snmp-server community public` によりv2cアクセスを閉塞する。
- nms側コマンド例:
  - `snmpwalk -v3 -l authPriv -u nmsuser -a SHA -A 'SnmpAuth28!' -x AES -X 'SnmpPriv28!' 10.28.90.1 system`

## 4. トラップ設計
- 監視対象機器に以下を投入し、linkDown/linkUp/coldStartをnmsへ通知する。
  - `snmp-server enable traps snmp linkdown linkup coldstart`
  - `snmp-server host 10.28.100.10 version 2c public`（トラップはv2cのまま運用。ポーリングv3・トラップv2cの混在は実務でも現実的な構成）
- 受信側（nms）はnet-snmpの`snmptrapd`で待受ける。community `public` を許可する設定（`/tmp/snmptrapd.conf`に`authCommunity log public`と`disableAuthorization yes`）を行い、フォアグラウンドで起動して受信ログを確認する（作成手順は構築手順書 §5参照）。
  - `snmptrapd -f -Lo -c /tmp/snmptrapd.conf`
- 試験（old-sw1のEthernet0/2を`shutdown`）でlinkDownトラップの実受信を確認する（試験計画書 Mission6参照）。

## 5. SPAN設計根拠
- cap-swはold-core⇔mon-core間の監視系transit（VLAN900）にインラインで挿入されている（E0/1=old-core側、E0/2=mon-core側）ため、この区間を通過するフレームは必ずEthernet0/1を1回通過する。
- `monitor session 1 source interface Ethernet0/1 both` によりE0/1のみをsourceに指定すれば、往路・復路とも重複なく全通過トラフィックを捕捉できる。E0/1・E0/2の両方をsourceに指定すると、同一フレームが2回（受信時＋送信時）ミラーされ、二重キャプチャとなってしまう。
- `monitor session 1 destination interface Ethernet0/3` で、SPANの宛先をcap収容ポートとする。宛先ポートはVLANに依存せずミラーフレームを転送する。

## 6. 32bit/64bitカウンタ問題
- `ifInOctets` / `ifOutOctets` は32bitカウンタであり、最大値 2^32-1 バイト（約4.3GB）でラップアラウンド（0に戻って再カウント）する。高速・高トラフィックな回線ではこの上限に短時間で到達し、単純な差分計算では実際のトラフィック量を過小評価してしまう。
- そのため、64bitカウンタである `ifHCInOctets` / `ifHCOutOctets`（High Capacity、最大値 約1.8×10^19バイト）の利用が推奨される（RFC2863）。
- 本ラボのMission4では、iperf3でトラフィックを生成し、32bit版と64bit版の両方をポーリング・比較して、この問題を体感する。
