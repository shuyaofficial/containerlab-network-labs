# 基本設計書: 監視基盤とSNMP深堀り

## 1. 現行LAN構成（老朽ポイント3点）
本ラボの現行LANは以下3点の老朽ポイントを抱えたまま運用されている（是正はテーマ29のスコープであり、本ラボでは温存する）。

- **単一コアのSPOF**: old-core 1台に全VLAN（10/20/30/90/900/901）のSVIとstatic routeが集約されており、old-coreが停止すると全セグメントが通信不能になる。
- **VLAN20のL2延伸**: 総務部セグメント（VLAN20, 10.28.20.0/24）がold-sw1・old-sw2の2台にまたがってトランクで延伸されており、本来フロアAで閉じてよいブロードキャストドメインが不要に広がっている。
- **static一辺倒・旧IOS**: old-rt⇔old-core間、old-core⇔mon-core間ともにstatic routeのみで構成され、動的ルーティングプロトコルは使用していない。old-rtは旧IOS 15.6を使用している。

## 2. 監視セグメント設計
- **VLAN100（監視セグメント, 10.28.100.0/24）**: 監視サーバnmsを収容する。デフォルトGWはmon-coreのSVI（10.28.100.1）。
- **VLAN900（監視系transit, 10.28.0.0/29）**: old-core⇔cap-sw⇔mon-coreを結ぶ監視系のtransitセグメント。old-core=10.28.0.1、mon-core=10.28.0.2、cap-sw=10.28.0.3。
  - **/30ではなく/29とした設計根拠**: cap-swはold-core⇔mon-core間にインラインで挿入されるが、cap-sw自身もin-band管理IP（.3）を持つ必要がある。A-B間の2アドレスに加えてcap-swの1アドレスを収容できるよう、利用可能ホスト数6の/29を採用している。
- **VLAN901（WAN transit, 10.28.1.0/30）**: old-rt（.1）⇔old-core（.2）を結ぶ既存のWAN transitセグメント（老朽LAN側、変更なし）。

## 3. SPAN設計
- **実装箇所**: cap-sw
- **設計内容**: old-core⇔mon-core間の監視系transit（VLAN900）にcap-swをインラインで挿入し、cap-swの以下2ポートを通過する全フレームをSPANでミラーする。
  - `monitor session 1 source interface Ethernet0/1 both`
  - `monitor session 1 destination interface Ethernet0/3`（cap収容ポート）
- **sourceをEthernet0/1のみとした設計根拠**: cap-swはインライン構成（E0/1=old-core側、E0/2=mon-core側）であるため、監視系transitを通過するフレームは必ずE0/1を1回通過する。E0/1とE0/2の両方をsourceに指定すると、同一フレームが往路・復路で2重にミラーされてしまう。片側（E0/1）のみをsourceに指定することで、通過する全トラフィックを重複なく1回のミラーで捕捉できる。

## 4. SNMPバージョン移行方針
- **Mission3（v2c導入）**: 監視対象6 IOL機器全てに`snmp-server community public RO`を投入し、MIBツリー（system, interfaces）をwalk/get可能にする。
- **Mission4（IF-MIB活用）**: v2cのまま、IF-MIBのifIn/OutOctets（32bit）とifHCIn/OutOctets（64bit）をポーリングし、トラフィック監視の基礎を確立する。
- **Mission5（v3移行）**: SPANキャプチャでv2cのcommunity文字列が平文であることを確認したうえで、`snmp-server group NMS-GROUP v3 priv`と`snmp-server user nmsuser NMS-GROUP v3 auth sha ... priv aes 128 ...`によりauthPriv（認証＋暗号化）のSNMPv3へ移行する。移行完了後は`no snmp-server community public`でv2cアクセスを閉塞する。
- **Mission6（トラップ）**: トラップは`snmp-server host 10.28.100.10 version 2c public`のままv2cで運用する。ポーリングはv3、トラップはv2cという構成は、トラップ側の設定変更コストや監視ソフトウェアの対応状況を踏まえると、実務でも現実的な混在パターンである。
